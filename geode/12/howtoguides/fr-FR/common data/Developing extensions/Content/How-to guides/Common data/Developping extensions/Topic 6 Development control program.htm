Programme de contrôle des développements

Introduction

Cette technologie offre une grande liberté de modification et propose une solution adéquate pour leur protection, grâce à des codes d'activité spécifiques ou verticaux.

Toutefois, cette technologie est fréquemment décriée pour la lourdeur et la difficulté d’automatisation du processus d'application des correctifs, principalement en raison des conflits et erreurs liés à cette liberté qui nécessitent des rectifications manuelles lorsqu'ils se produisent.

Cela devient particulièrement crucial pour les mises en oeuvre du Cloud car nos clients attendent dans le même temps :

  • avoir la possibilité de faire des développements sur mesure ou verticaux dans ces environnements,
  • s'assurer que l'application des patchs standards peut se faire automatiquement et sans conflits.

C'est pourquoi il est important de limiter la possibilité pour les partenaires de créer des développements sur mesure et verticaux. Cela peut se faire en établissant des règles qui fixent des normes, mais qui limitent également les modifications de dictionnaire. Ces règles ne sont pas obligatoires que dans les mises en œuvre du Cloud. Elles sont également essentielles pour garantir à tous nos clients une mise à niveau fluide.

Sage a créé un outil de vérification qui fonctionne sur les environnements déjà existants. Cet outil a été amélioré dans l’update 10. Ce document présente les règles qui seront contrôlées par cette dernière version.

Cet outil ne contrôle que la cohérence des dictionnaires (méta-données). À l'avenir, Sage prévoit d'améliorer l'outil pour y ajouter la vérification des sources.

Tel que le dictionnaire est conçu aujourd'hui, il peut être difficile de suivre les règles dans certains cas, notamment dans le code classique. Le code Syracuse a été conçu pour rendre le respect de ces règles possible dans ces cas.

Lancer l'outil

L'outil est fourni comme une fonction classique appelée ACTLSPE. Il inclut une fenêtre dans laquelle vous pouvez entrer des paramètres supplémentaires. Il doit être exécuté dans le dossier où les développements verticaux et sur mesure ont été réalisés.

Les paramètres à renseigner sont :

  • les dossier à tester,
  • le dossier de référence (en général le dossier d'application),
  • des plages optionnelles pour les éléments spécifiques ou verticaux comme des noms, numéros de menus locaux ou tables diverses.

Le script s'exécute et renvoie un fichier journal contenant quatre niveaux d'information :

  • information : décrit les modifications sans risque qui peuvent avoir un impact si elles sont effectuées différemment par un autre fournisseur. Par exemple, l'augmentation de la longueur d'un type de données standard. Il s'agit juste d'un rappel des points à vérifier lorsque plusieurs développements verticaux ou sur mesure sont installés ensemble dans un environnement donné,
  • attention : décrit les modifications qui ne sont pas considérées comme des erreurs, mais qui peuvent le devenir. Par exemple, si les dictionnaires sont améliorés pour faciliter la mise à niveau en proposant une manière différente de développer certaines fonctionnalités,
  • erreur : décrit une erreur dans la conception qui empêche une mise à niveau automatique correcte. Cette erreur empêche l'installation sur les environnements Cloud mais peut être gérés pour les environnements on-premise. Cela rend le processus de mise à niveau plus difficile car il faudra vérifier ces éléments,
  • erreur fatale : décrit une erreur qui empêche une mise à niveau correcte, c'est le cas pour la modification des éléments du dictionnaire.

Détail des règles

Convention de nommage

Le premier point important est que des conventions de nommage sont prévues pour gérer les conflits potentiels entre les partenaires fournissant des développements verticaux ou des add-ons. Chaque partenaire souhaitant développer de telles fonctionnalités reçoit des plages de codification. La règle est la suivante :

  • les noms de plages d'add-ons commencent par X,
  • les noms de plages verticales commencent par Y,
  • les noms de plages sur mesure commencent par Z.

Cette règle s'applique uniquement au niveau d'imbrication le plus élevé associé à l'élément. Par exemple :

  • un écran vertical doit suivre la règle de nommage, mais les sous-éléments (champs) peuvent être nommés librement,
  • sur un écran standard, les champs verticaux ajoutés doivent suivre la règle de nommage,
  • la même règle s'applique aux colonnes et indices de tables, noms de collection ou méthodes sur les classes et représentations.

Les éléments contrôlés par la règle de nommage sont: les tables, les écrans, les fenêtres, les objets, les consultations, les actions, les types de données, les classes, les représentations, les codes d'activité, les définitions de paramètres, les scripts et les états. Les sous-éléments d'éléments standard sont vérifiés selon ce qui a été défini auparavant.

Les tables diverses et menus locaux ont des plages numériques.

Certains éléments de dictionnaires sont considérés comme étant non standard bien qu'ils ne commencent ni par X, Y ou Z :

  • les fonctions commençant par CONSX, CONSY, CONSZ (consultations),
  • les fenêtres commençant par DCNSX, DCNSY, DCNSZ, FCNSX, FCNSY, FCNSZ (associées à des consultations non standard),
  • les scripts commençant par SPE (les scripts commençant par SPV doivent être renommés à partir de l’update 8, voir ci-après).

Les éléments de dictionnaire et les scripts commençant par W sont des éléments générés. Par conséquent, il est interdit de créer directement ces éléments. Ils sont normalement générées à partir d'un modèle dont le nom ne commence pas par W.

Tout élément suivant ces règles de nommage est un élément non standard qui doit être protégé par un code d'activité commençant par X, Y ou Z. Tout élément qui ne respecte pas la convention de nommage et qui n'est pas présent dans le dossier de l'application est un élément non standard ajouté qui ne respecte pas la convention de nommage ( ce qui constitue une erreur).

Les éléments de dictionnaire commençant par A sont des éléments superviseurs. Ces éléments suivent des règles spécifiques. Les modifier entraîne généralement une erreur fatale.

L'impact ne peut être mesuré sur le code standard. Les variables internes peuvent être définies en fonction du type de données de la colonne standard. Si la taille d'une colonne doit être augmentée, le type de données générique doit être remplacé par un type de données non générique.

Ce type de requête doit être demandé comme une création de point d'entrée (avec la même priorité).

Définition de table diverse standard

Réduire ou modifier la longueur du code le cas échéant.

L'impact ne peut être mesuré sur le code standard.

Une case à cocher indique si la longueur peut être modifiée.

Règles supplémentaires ou commentaires

Les vérifications supplémentaires suivantes sont effectuées :

  • comme indiqué précédemment, toute modification sur les tables de superviseurs est considérée comme une erreur fatale,
  • l'outil de cohérence vérifie uniquement les menus locaux et non le chapitre consacré aux messages. Les manques sont indiqués comme étant des erreurs. Par exemple, si le choix N et le choix n+2 sont définies sans un choix N. le contrôle se fait uniquement sur les langues gérées dans le dossier,
  • les éléments non standard et qui ne sont pas protégés par un code d'activité non standard sont affichés en erreur.
  • les éléments non standard et qui ne sont pas nommés dans la plage de codification non standard sont affichés en erreur,

Les éléments de dictionnaire commençant par W sont exclus du journal. Le système considère que la cohérence a déjà été vérifié sur les éléments d'origine de la génération.

Ajouter des boutons aux fenêtres standard est parfois nécessaires, mais cela crée des problèmes que nous ne pouvons résoudre correctement :

  • seules quatre valeurs sont disponibles pour les extensions non-standard,
  • les conflits entre les extensions verticales ne peuvent être résolus et nécessitent un contrôle particulier lorsque plusieurs extensions verticales sont installées ensemble.

Pour les normalisations effectuées en version 7, il est possible de créer autant de liens que nécessaires, et leur nom est codifié.

Une recommandation particulière devrait être faite pour la création de pièces automatiques spécifiques. La meilleure solution consiste à les dupliquer et à les renommer pour faciliter leur mise à niveau à l'aide d'outils de comparaison. Les paramètres sont disponibles pour réaffecter le code de pièce automatique utilisé lorsqu'une opération de comptabilisation est effectuée.
