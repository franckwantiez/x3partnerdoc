Packaging

Cartographie de votre add-on

La plateforme technologique de développement est compatible avec :

  • les systèmes de bases de données Oracle et SQL-Server,
  • les systèmes d'exploitation Windows, AIX, Linux.

Même si la plate-forme vous donne la possibilité de prendre en charge ces systèmes et bases de données, elle doit être validée. Vous pouvez établir vos propres limites de prise en charge.

Tous les messages et textes sont stockés dans des fichiers dédiés.

Des processus et des outils sont disponibles pour traduire l'application.

Un add-on doit être traduit et livré au moins :

  • en anglais,

  • dans la langue source gérée par le partenaire.

La cartographie de votre add-on doit fournir :

  • une courte description,
  • sa disponibilité concernant :
    • les plateformes prises en charge,
    • les bases de données prises en charge,
    • les langues prises en charge,
  • sa compatibilité garantie avec une liste de versions, de mises à jour et de patchs.

Règles de qualité pour la documentation

En tant que partenaire estimé de Sage, il est de votre responsabilité de fournir la documentation liée à votre add-on.

Les documents suivants correspondent aux exigences minimales :

  • un guide de l'utilisateur avec une documentation fonctionnelle,
  • un guide d'installation et de configuration.

Vous pouvez également utiliser l'outil de documentation fourni avec l'application pour apporter une aide en ligne pour les champs et les fonctions.

Fournissez un fichier changelog avec chaque patch.

Veuillez vous reporter à la documention Comment écrire la documentation disponible sur le Centre d'aide en ligne pour en savoir plus sur l'outil de documentation livré avec la plateforme.

Patching

Lorsque vous créez des patchs pour des add-ons, vous devez utiliser un type de patch dédié.

Vous devez sélectionner cette option pour préserver toute action SPV potentiellement développée sur un asset dans la couche verticale. Ainsi les actions SPV seront conservées si une verticale se trouve dans le dossier client où est installé l'add-on. Les add-ons utilisent les actions de dictionnaire, ce qui ne devrait pas poser de problème.

Les fichiers de patch doivent suivre les règles de nommage pour profiter des contrôles de séquence automatiques fournis par la plateforme.

La règle de nommage des fichiers de patch est la suivante :

<AddOnId>_<AddOnPatchNumber>_<X3RequiredPatchLevel>_<X3Version>.dat

  • AddOnId : identificateur de l'add-on. Il ne devrait pas être P car il est réservé aux patchs standards Sage.
  • AddOnPatchNumber : numéro d'ordre séquentiel utilisé pour identifier les différents patchs add-on.
  • X3RequiredPatchLevel : numéro de liste des correctifs standard avec lesquels le package add-on est compatible.
  • X3Version : version du produit avec laquelle la liste de patchs d'add-ons est compatible.

De plus amples informations sont disponibles sur le centre d'aide en ligne.

Chaque version de package devrait contenir les éléments suivants :

  • un ensemble de fichiers de patch dans un répertoire FILPATCH,
  • une documentation en ligne fournie dans un répertoire séparé du fichier patch,
  • un répertoire contenant des exemples de code source de traitements d'aiguillages ( le répertoire ADDSRC est recommandé),
  • une documentation générale pour l'add-on,
  • éventuellement, un dossier de démonstration.
Les utilitaires patchés sous la forme d'un EXE doivent être ré-exécutables.
Le traitement d'aiguillage doit être livré dans un patch séparé.

Si vous sélectionnez Vertical, toutes les actions SPV sont supprimées et seules celles de votre patch sont retenues. Vous ne pouvez choisir ce type de patch que pour un développement spécifique pour un client donné. Il sera installé avec toutes les interactions à gérer.

Maintenance et support

Le développeur d'add-on doit gérer le processus de maintenance. Lorsqu'une liste de patchs standards est livrée, le développeur doit vérifier si le dernier package d'add-on est compatible avec la nouvelle liste de patchs. Si ce n'est pas le cas, le développeur doit fournir une nouvelle version de l'add-on dans le mois qui suit. Le partenaire doit communiquer les informations de compatibilité entre les packages d'add-on et les listes de patchs standards.

Le développeur de l'add-on doit également fournir le support pour ses propres add-ons. Le développeur fournira dans la version suivante du package, des corrections aux problèmes de l'add-on signalés par les clients. Des correctifs peuvent également être fournis en fonction de la gravité du problème :

  • pour un problème mineur : aucune action supplémentaire n'est requise,
  • pour un problème majeur : Il faut fournir un correctif au client concerné (si nécessaire),
  • pour un problème majeur ou bloquant ayant des répercussions sur plusieurs clients : Fournir un correctif aux clients concernés et publier une nouvelle version du package d'add-on.
