Règles de développement

Les add-ons doivent respecter un ensemble de règles pour éviter les conflits provoqués par la cohabitation d’extensions.

Dans ce chapitre :

  • nous détaillons les principales règles à appliquer,

  • nous mettons en évidence certaines limitations dont vous devez tenir compte,

  • nous vous fournissons des conseils pour concevoir et tester vos développements.

Les sujets abordés sont tous disponibles depuis la version 7.

Principales règles de développement à appliquer

Actions sur champ

La plateforme vous permet de définir des actions sur champs dans les écrans.

Une fonctionnalité vous permet d'utiliser une action verticale (SPV) sur chaque événement (initialisation, contrôle, etc.) de chaque champ.

Toutefois, il ne peut y avoir qu'une seule action de ce type pour chaque champ.

Dans le mode add-on, tous les partenaires partagent le même dictionnaire. Vous devez utiliser les actions du dictionnaire pour éviter tout conflit.

Le script vertical du dictionnaire écran est un programme unique et peut difficilement être partagé par différents partenaires.

Les noms d'action du dictionnaire doivent commencer par votre identifiant. Vous devez également les protéger avec un code activité.

Cette règle s'applique à toute action que vous ajouterez dans un champ des écrans standard.

Si vous êtes seul à travailler sur le script vertical, vous pouvez utiliser les actions SPV pour vos écrans qui :

  • sont protégés par votre code activité,

  • n'incluent pas de développement vertical.

Utilisez uniquement des actions du dictionnaire pour vos actions sur champs.

N'oubliez pas de définir un code activité sur l'action. Le nom du code activité doit commencer par votre identifiant unique.

Comment gérer la couche verticale avec modèle d'objet et points d'entrée

Un script distinct gère chacune de ces trois couches : spécifique, verticale, et standard.

Dans chaque script, les développeurs utilisent les points d'entrée fournis par les différents modèles dans lesquels ils travaillent.

Le code source du modèle ou objet (le plus utilisé) n'est pas fourni.

La documentation fournit la liste des points d'entrée disponibles ainsi qu'un ensemble de variables, que vous pouvez utiliser pour interagir avec le standard.

Une fonction ne peut faire l'objet que d'un seul script vertical.

C’est pourquoi différents acteurs pouvaient difficilement partager le script. Il fallait alors concevoir et utiliser un programme d’aiguillage géré manuellement pour exécuter de manière séquentielle les scripts des différents partenaires.

La marche à suivre :

  • avant l'update 8 : vous deviez gérer, de façon manuelle, le programme d’aiguillage, Cela a posé de nombreux problèmes par le passé.
  • depuis l'update 8 : le superviseur gère automatiquement le programme d'aiguillage des objets et points d'entrée. Cela représente une réelle amélioration.

Couche verticale avec des objets

Auparavant, le nom du script vertical était défini dans l'objet. Le nom du programme d’aiguillage était utilisé pour autoriser les appels multi-verticaux.

Désormais, une case à cocher figure dans la définition de l'objet. Si cette case est cochée, il existe au moins un traitement vertical sur l'objet.

Un bouton info vous fournit les noms de tous les traitements verticaux définis pour cet objet.

La définition des scripts verticaux à utiliser pour un objet est indiquée dans la table des points d'entrée.

L'ordre dans lequel les différents scripts devraient s'exécuter est également défini dans cette table.

Veuillez tenir compte des points suivants :

  • la table des points d'entrée est chargée comme un ensemble de variables globales lorsque vous créez une session classique. Si vous modifiez la liste des points d'entrée, vous devrez vous déconnecter et vous reconnecter pour que la modification soit prise en compte,
  • lorsque la table est chargée dans la mémoire, un contrôle vérifie que le script adx défini dans la liste des points d'entrée existe. Si ce n'est pas le cas, dans le dossier ou dans l’un des dossiers parents, le script n'est pas ajouté à la table mémoire et ne s'exécute pas, même si vous l'ajoutez après la création de session. Vous devez vous déconnecter et recréer la table mémoire.

Comprendre les ordres d'exécution des différentes couches

La même logique s'applique à tous les points d'entrée.

Le modèle appelle les éléments ci-dessous, s'ils existent, dans l'ordre suivant :

  • le script spécifique,
  • tous les scripts verticaux selon leur rang,
  • le script standard.

Deux variables globales sont utilisées pour contrôler l’exécution des différents scripts :

  • GPV : empêche l'exécution de la couche verticale lorsqu’elle est paramétrée à 1.
  • GPE : empêche l'exécution de la couche standard lorsqu’elle est paramétrée à 1.

Ces deux variables sont paramétrées à 0 par le standard avant l'exécution de la boucle globale.

Puis :

  • si la variable GPV est paramétrée à 1 dans le script spécifique, tous les scripts verticaux seront ignorés et ne seront pas exécutés. Ce comportement doit être validé au cas par cas sur un site du client,

  • si la variable GPE est paramétrée à 1 dans le script spécifique ou dans l'un des scripts verticaux, la couche standard ne sera pas exécutée. Ce comportement est exceptionnel. Il intervient généralement lorsque la couche standard doit être exécutée avant la couche verticale ou spécifique. Pensez à bien exécuter la couche standard.

Lorsque plusieurs scripts (par exemple, un script spécifique et plusieurs scripts verticaux) sont exécutés, la valeur de la variable GPE est la dernière à être définie. C'est pourquoi, dans les cas d'interaction complexes, le comportement doit être validé par le site client.

Cas complexes en exécution de points d'entrée

L'une des verticales requiert une exécution préalable de la couche standard.

Exemple

Trois scripts verticaux sont définis pour l'objet BPC.

Par exemple, le script XY23BPC2 requiert une exécution préalable de la couche standard dans le point d'entrée PICKE.

Dans ce cas, le script est : 

$ACTION
Case ACTION
When “PICKE” : Gosub ETIQP
………
Endcase
Return

$ETIQP
# Execute the standard code first
Gosub ACTION From SUBPTH

GPE=1   :# Avoid to reexecute the standard
#  Vertical code
……
……
Return

Dans ce cas, vous devez faire en sorte que le troisième script vertical XX34PH78 soit le prochain exécuté.

Sur un site client, l'ordre d'exécution des différentes verticales doit être défini en fonction des besoins spécifiques.

Si une verticale requiert que la couche standard soit exécutée en premier plutôt que les autres, cette verticale doit être exécutée en dernier, et l'ordre d'exécution doit être adapté.

Vérifiez également qu'un autre script vertical :

  • ne paramètre pas GPE à 0 après que votre script l'a paramétré à 1. Dans ce cas, le standard s'exécutera deux fois,
  • ne s'exécute pas avant le standard.
Il s'agit de cas exceptionnels. Elles doivent être vérifiées sur un site client car il est impossible de gérer et d'anticiper toutes les combinaisons de scripts verticaux.
Dans de rares cas, l'ordre doit être différent en fonction du point d'entrée. Sur un site client, l'une des verticales doit être scindée pour répondre à ce besoin spécifique car aucune modélisation ne gère ce type de requête.

Patching des points d'entrée à partir de l'update 8

  • Patching des points d'entrée :
    L'ordre d'exécution n'est jamais modifié via un patch. Toutefois, lorsqu'un nouveau point d'entrée est patché, l'ordre d'exécution est défini s'il a été établi.
  • En revalidation de dossier :
    le script vertical, s'il existe sur un objet, est automatiquement inséré dans la table des points d'entrée.

Gérer la couche verticale avec d'autres modèles : programme d'aiguillage

Avec les autres modèles pour lesquels le programme d'aiguillage n'est pas automatiquement créé, il vous faut gérer l'aiguillage manuellement.

Vous devez le faire pour les :

  • actions,
  • consultations,
  • modèles import/export,
  • états.

Les consultations, modèles d'importation/exportation et états sont généralement dupliqués lorsque vous souhaitez les personnaliser. Vous devez surtout vous focaliser sur les scripts associés aux actions.

Pour ces différents éléments, il n'existe pas de couche verticale.

Le seul script disponible est le Script spécifique qui sera utilisé comme un aiguillage polyvalent.

États de développement

Les états standards sont développés avec l'outil Crystal Report.

Voici quelques conseils sur la manière d'élaborer vos rapports :

  • utilisez la charte de développement des états Crystal Reports telle que définie dans le centre d'aide en ligne,
    développez vos états à l'aide d'un des modèles d'états suivants : ATEMPLATE_RPT1 ou ATEMPLATE_RPT2,

  • le nom de l'état doit commencer par votre identifiant et porter le même nom que le code d'état du dictionnaire,

  • tous les états doivent se baser sur une source ODBC,
  • vous devez fournir des états dans les différentes langues auxquelles vous avez accès en tant que partenaire. L'anglais est obligatoire,
  • les états seront conformes aux règles de développement de gestion des autorisations.

Reportez-vous au support de formation.

Limites principales

La plate-forme technologique connaît certaines limites. Sage cherche à corriger ces limitations.

Certains aspects des fonctions classiques doivent être pris en compte :

Le nombre de colonnes dans une table

est limité à 512. Lorsque vous travaillez avec une table standard proche de cette limite, créez et gérez votre table de complément pour éviter les problèmes.

La longueur d’une ligne dans une table

est limitée à 8 Kb.

Si la table standard est proche de cette limite, vous pouvez ajouter et gérer votre propre table.

Le nombre de sections dans un masque

est limité à 15.

Si vous ajoutez plusieurs sections et approchez de la limite, il vous faut ajouter un nouveau masque.

L'authoring vous permet de concevoir une mise en page adaptée à vos besoins.

Le nombre de champs dans un masque

est limité à 500 dans la définition.

limité à 300.

si vous ajoutez plusieurs champs et approchez de la limite, il vous faut ajouter un nouveau masque.

Interactions avec la couche standard

Évitez d'apporter des modifications à la fonctionnalité d'un élément core, notamment pour les localisations. Ces fonctionnalités sont destinées à coexister dans un même dossier client. Il est important de se rappeler qu'un add-on est un moyen de renforcer et d'étendre la fonctionnalité core et non de la contourner.

Points particuliers à se rappeler :

  • n'utilisez pas votre code d'activité pour modifier ou supprimer un élément de base,
  • ne désactivez pas de bouton, menu, champ, etc.,
  • ne modifiez pas le comportement d'un bouton ou d'une menu,
  • utilisez uniquement GPE=1 pour appeler directement le core mais pas pour l'éviter,
  • utilisez un menu plutôt qu'un bouton dans une fenêtre core,
  • utilisez votre propre écran dans une fenêtre core ou créer votre propre pop-up complémentaire,
  • évitez de partager un même espace disponible sur des écrans core entre différents add-ons installés au sein d'un même dossier client,
  • ce type de conflit doit être géré par l'équipe de mise en œuvre du projet.

Faites attention, lorsque vous complétez une fonction core. Si la couche verticale (spécifique dans certains dictionnaires) est déjà utilisée par un add-on ou asset vertical, ne forcez pas le traitement d'aiguillage existant de cet add-on ou asset vertical.

Dans cette situation et pour vos développements futurs, vous devez utiliser la stratégie du traitement d'aiguillage qui appelle votre propre traitement applicatif.

  • Pour utiliser un point d'entrée, utilisez la stratégie du traitement d'aiguillage pour aider la mise à jour des traitements d'aiguillage.
  • Créez un patch distinct pour ces derniers. Ensuite, ajoutez un exemple de ce à quoi devrait ressembler le traitement des sources dans un dossier dédié nommé ADDSRC que vous livrez dans votre patch.
  • Complétez le fichier Readme en ajoutant la référence des traitements d'aiguillage et les traitements core correspondants.
Ne modifiez jamais l'utilisation d'un élément core.

Test

Pour tester une fonction core :

  • vérifiez la fonctionnalité globale de la fonction core, une fois que les add-ons et les assets ont été ajoutés,
  • vérifiez le traitement d'aiguillage dans la couche verticale, dans la couche spécifique pour certains dictionnaires ainsi que dans les points d'entrée,
  • vérifiez que les actions de champ coexistantes fonctionnent ensemble dans un écran core,
  • vérifiez que les menus et boutons fonctionnent pour plusieurs add-ons dans une fenêtre core, évitez le même code menu pour deux add-ons de la même fenêtre core,
  • assurez-vous que les capacités de la plate-forme sont respectées (nombre de champs dans une table, nombre d'onglets dans une fenêtre, etc.).
