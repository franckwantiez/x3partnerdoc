Manage clustered indexes in particular cases

So far, the clustered indexes that have been delivered were indexes associated with supervisor tables. No issues occurred because:

  • No customization is done on these tables.

  • The way that these indexes are used is known.

For functional tables, it is more complicated:

  • A clustered index decreases the access to data on reading operations using the corresponding index.

  • If a lot of insertions are done with index values that are randomly placed on a clustered index, then adding indexes will constantly create additional index nodes and have a large overhead. You can reduce this by giving a fill factor to the index nodes.

    Fill factor example

    Select an X value under 100. When you rebuild the index, a node will be filled only at X% of its capacity.

Determine if a clustered index is relevant on a table

An ideal index to cluster is a surrogate key or an element that increases. Indeed, this will avoid inserting details in the middle of the table.

You need to reduce this phenomenon with the read/write ratio. If there aren’t many write operations in the table, then it is less problematic.

Stock journal (STOJOU), Texts to translate (ATEXTRA), or Products (ITMMASTER) tables are good examples. It can benefit 80% of customers, but be detrimental to the other 20% if left as is:

  • STOJOU: is relevant if not a manufacturing process.

  • ITMMASTER: depends on the counter the customer chose and if the configurator is used.

  • ATEXTRA: is not relevant if a lot of texts are created. For example, when creating products.

You can reduce the impact in the case of a medium read/write ratio by using the fill factor for the clustered index.

As some clustered indexes delivered might not be relevant to your case, you can degrade your global response time, which is not the target of the delivery that is not relevant in your case. The following table lists the different options that you have:

Read/write ratio Index option How to set up your clustered index
High (> 70%) A clustered index is relevant in your case if the read actions are done primarily with it. If Sage delivers the clustered index by default, keep it active. Otherwise, create a specific clustered index by checking the column.
Medium
(50 - 70%)		 A clustered index can be relevant, but you should consider adding a partial fill factor to keep space for insertion without reorganizing the index frequently. Keep the clustered index if it is delivered or add it as a specific one if not.
Add a line on the configuration file.
In the following example, INDEXCODE is the index code value in the index definition of the table:
$MSSQL_INDEXCODE
{"WITH (FILLFACTOR=80)"}
END
Low (< 50%) The overhead on the insert operation is too big. A clustered index is not the right solution. Deactivate the clustered index if activated in standard.

You can use an additional option if the many read actions of a few columns only are done on a given table. This is achieved in the X3 code:

  • By using the Columns instruction. Otherwise, a read is implemented with the select * from… syntax.

  • On a Fillbox instruction that fills a left list with only the declared columns.

If this specific case is detected, you can add the columns that are always read in the node index. Adding these columns avoids a second read operation to get the data after accessing the index. This is only relevant if you have a lot of these queries with a limited list of columns. You should only add the columns which are not already components of the key as these are already present.

You can implement this in the configuration file with the following syntax:

$MSSQL_INDEXCODE
{"INCLUDES (column_name)"}
END
Example

You have added a specific index (YPOS0) on the POSCOD table that only contains the POSCOD field. If you want to:

  • Decide that this specific index is the cluster one

  • Have access quickly to the country code (CRY) and the city code (POSCTY) at the same time

Then you can add the following lines in your configuration section:

$MSSQL_YPOS0
{"INCLUDES (CRY_0,POSCTY_0)"}
END
