Cartografía del add-on

La plataforma tecnológica de desarrollo es compatible con:

  • los sistemas de bases de datos de Oracle y SQL Server
  • los sistemas operativos Windows, AIX y Linux

A pesar de que la plataforma sea compatible con todos estos sistemas y bases de datos, es necesario validar. El entorno se puede limitar a un determinado público objetivo.

Todos los mensajes y textos se almacenan en ficheros dedicados.

Hay procesos y herramientas disponibles para traducir la aplicación.

El add-on se debe traducir y proporcionar, al menos, en:

  • inglés

  • el idioma del partner

La cartografía del add-on debe incluir lo siguiente:

  • una descripción corta
  • su disponibilidad en cuanto a compatibilidad con:
    • plataformas
    • bases de datos
    • idiomas
  • su compatibilidad certificada con versiones, actualizaciones y parches

Reglas de calidad para la documentación

Como partner de Sage, tienes la responsabilidad de proporcionar la documentación relacionada con el add-on.

Estos son los requisitos mínimos de documentación:

  • manual de usuario con documentación funcional
  • guía de instalación y configuración

También puedes utilizar la herramienta de documentación de la aplicación para ofrecer ayuda de campos y funciones en línea.

Entrega un fichero de registro de cambios (changelog) con cada parche.

Para más información sobre la herramienta de documentación que se entrega con la plataforma, consulta cómo redactar documentación en el centro de ayuda en línea.

Parches

Utiliza un tipo de parche exclusivo al crear los parches de los add-ons.

Selecciona esta opción para mantener todas las acciones SPV que se hayan desarrollado en un recurso de la capa vertical; estas acciones se conservan si la capa vertical está presente en el dossier cliente donde está instalado el add-on. Los add-ons utilizan acciones del diccionario, por lo que no debería suponer un problema.

Los ficheros de parches deben respetar las reglas de denominación para poder aprovechar los controles de secuencia automáticos de la plataforma.

Las reglas de las denominaciones de los ficheros de parches son las siguientes:

<AddOnId>_<AddOnPatchNumber>_<X3RequiredPatchLevel>_<X3Version>.dat

  • AddOnId es el identificador del add-on. No debe comenzar por P porque esta letra está reservada para los parches estándar de Sage.
  • AddOnPatchNumber es el número de secuencia con el que se identifican los diferentes ficheros de parches del add-on.
  • X3RequiredPatchLevel es el número de lista de parches estándar compatibles con el paquete del add-on.
  • X3Version es la versión del producto host compatible con la lista de parches del add-on.

Para más información, consulta el centro de ayuda en línea.

Cada versión del paquete debe incluir lo siguiente:

  • un conjunto de ficheros de parche en el directorio FILPATCH
  • documentación en línea en un directorio independiente de ficheros de parche
  • un directorio con ejemplos del código fuente de los procesamientos de conmutación (se recomienda el directorio ADDSRC)
  • documentación general para el add-on
  • un dossier para demostraciones (opcional)
Las herramientas a las que se les haya aplicado un parche en forma de ejecutable deben poder volver a ejecutarse.
El procesamiento de conmutación se debe entregar un parche independiente.

Si se selecciona la opción Vertical, se suprimen todas las acciones SPV y solo se aceptan las del parche. Solo puedes seleccionar este tipo de parche para un desarrollo específico y un cliente dado. Se instalará con todas las interacciones por gestionar.

Mantenimiento y soporte

El desarrollador del add-on es el responsable de su mantenimiento: cada vez que se saca una lista de parches estándar, debe verificar que el último paquete del add-on es compatible con esta. En el caso de que no lo sea, tiene un mes para proporcionar una nueva versión del add-on. El partner debe publicar la información relacionada con la compatibilidad entre los paquetes del add-on y las listas de parches estándar.

El desarrollador del add-on también debe proporcionar el soporte necesario: en la siguiente versión del paquete, debe corregir los problemas que los clientes hayan detectado. El tipo de corrección depende de la gravedad del problema:

  • Problemas menores: no es necesario tomar medidas adicionales.
  • Problemas importantes: hay que entregar un hotfix o revisión para los clientes afectados, si es necesario.
  • Problemas de bloqueo o problemas importantes que afectan a varios clientes: hay que entregar un hotfix o revisión para los clientes afectados y lanzar una nueva versión del paquete del add-on.
