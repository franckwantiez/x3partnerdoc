Reglas de denominación

Con las reglas de denominación se evitan conflictos entre las extensiones de desarrollo de diferentes partners, puesto que no permiten que el trabajo de otros partners sobrescriba la capa vertical.

Desde la primera versión, los rangos de las denominaciones son los siguientes:

  • capa estándar: A-V (A para el módulo Supervisor),
  • capa vertical: X-Y
  • capa específica: Z

Los elementos que comienzan por W corresponden a los elementos generados automáticamente, como las transacciones.

Antes había rangos para los menús locales y las tablas varias de la capa vertical.

Cuando solo un partner utiliza un dossier cliente, estas reglas se pueden implementar fácilmente.

Sin embargo, actualmente, los clientes esperan contar con un catálogo de soluciones que les permita seleccionar extensiones de distintos partners e instalarlas en su entorno con rapidez.

Sin reglas no es posible adaptarse a este nuevo requisito, que genera nuevas oportunidades comerciales.

Para poder ofrecer la instalación de extensiones de diferentes partners en un único dossier, hay que garantizar que no utilicen la misma denominación.

Por este motivo, Sage introdujo el concepto de identificador único.

Las extensiones de código que se entregan con la versión 7 permiten a cada partner utilizar a petición lo siguiente:

  • un identificador único, normalmente en forma de código alfanumérico de cuatro cifras
  • un rango de menús locales y de tablas varias

Este identificador se debe incluir en la cabecera de todos los nombres que se creen en el diccionario y en los programas. Por ejemplo, XX12.

Estas reglas de denominación se aplican a todos los elementos del diccionario que utilizan una codificación alfanumérica como las pantallas, las clases, las representaciones, los códigos de actividad, las acciones, el código fuente, etc.

Estas convenciones también se deben utilizar para las claves, las abreviaturas y los parámetros de los elementos.

Por tanto, los nuevos rangos de las denominaciones son estos:

  • capa estándar: A-V (A para el módulo Supervisor),
  • capa vertical:
    • add-ons: X*** (El identificador es único; si se respetan las reglas de denominación, no se genera ningún conflicto.)
    • recursos: Y (Puede haber conflictos si instalas recursos diferentes en un mismo dossier.)
  • capa específica: Z (Los elementos son específicos del cliente. No hay conflictos que gestionar.)

Los rangos 13000-13999 y 18000-29999 son exclusivos de los menús locales y las tablas varias, pero se pueden reservar otros a petición.

El equipo de Global Product Delivery de Sage se encarga de asignar códigos a partir de las solicitudes de los partners.

Los códigos se almacenan en una base de datos única para evitar que se dupliquen las asignaciones.

Los datos administrativos (personalizaciones, pases, elementos de menú, etc.) se almacenan en la base de datos MongoDB.

Cada componente de los datos administrativos y su identificador también deben ser únicos; debe comenzar por este o incluirlo en determinados casos, que se explican más adelante.

Desde la actualización 8, se recomienda encarecidamente a los partners que utilicen el identificador entregado para etiquetar sus elementos. Este identificador desempeña la misma función que el código de actividad para los componentes almacenados en la base de datos MongoDB (ver apartado Identificador entregado).

La codificación de los identificadores almacenados en la base de datos MongoDB no tiene limitaciones de tamaño.

Si has creado una extensión antes de la implementación de estas reglas, cámbiale el nombre antes de pasarla al programa de los add-ons, que ofrece nuevas oportunidades comerciales.

Al desarrollar una extensión nueva, aplica las reglas de denominación y de desarrollo de extensiones de inmediato.

Si instalas una capa vertical que no siga estas reglas, puede haber conflictos.

Trata de solucionar el problema directamente en el proyecto en el que se produjo; los conflictos no se pueden prever.

Sage promueve el uso de add-ons y el cambio de nombre de los códigos de los recursos.

Todos los elementos desarrollados deben comenzar por el identificador único correspondiente que Sage ha asignado o contenerlo.

Por ejemplo, el identificador único de Sage Portugal es XX04. El conjunto de todos sus desarrollos debe comenzar por XX04.

Sage proporciona, a petición y a cada partner, un rango de menús locales y de tablas varias adaptadas a sus necesidades. Por ejemplo, para Sage Portugal, este rango es 13220-13259.

Para la transcodificación de las tablas de importación y exportación, hay que definir una regla que evite conflictos. Sin embargo, como no se suelen producir con frecuencia, esta regla todavía no se ha implementado.

En los próximos párrafos, se explica cómo aplicar las reglas de denominación.

Elementos añadidos en una entidad estándar

Al añadir campos en una entidad estándar, como una pantalla:

  • el nombre de los campos debe empezar por tu identificador;

  • en cada campo debes definir un código de actividad que comience también por este identificador.

En la siguiente captura de pantalla, X1QT corresponde al identificador único de cuatro cifras del partner.

Para facilitar la identificación, utiliza la siguiente convención para los campos que has creado, que completan los elementos del diccionario estándar (tablas, pantallas, etc.) y que aún no existen en la aplicación. Empieza añadiendo el identificador seguido de una barra baja.

En este ejemplo, podría ser X1QT_MAX.

Nueva entidad definida en el diccionario

Para una nueva entidad, como una tabla o clase:

  • el nombre y la abreviatura debe comenzar por tu identificador;
  • debes definir un código de actividad para el elemento que comience también por este identificador.

Para un elemento secundario, como una columna:

  • puedes utilizar una codificación libre porque no entrará en conflicto con las extensiones de otros partners al pertenecer estas a tu entidad;
  • no es necesario añadir un código de actividad para protegerlo porque ya lo hace el código definido en el nivel primario.

La codificación libre en las columnas facilita el uso de la asignación de clases y, por tanto, simplifica la codificación.

Menús locales y tablas varias

Sage asigna a los partners un rango de menús locales y de tablas varias que usar para el desarrollo.

Utiliza los valores incluidos en este rango y define códigos de actividad que comiencen por tu identificador para cada nuevo elemento que crees.

Código fuente

El nombre de todos los scripts que crees deben comenzar por tu identificador.

Los partners comparten los programas de conmutación; si necesitas uno, deja el nombre estándar.
Estos programas no se deben parchear, sino modificar manualmente en el entorno del cliente.
Las entidades específicas de los clientes, como los scripts o el diccionario, deben comenzar por Z. Protege las entidades del diccionario con un código de actividad.
Hay una excepción con los scripts cuyo nombre se propone automáticamente para los desarrollos personalizados, pues suelen comenzar por SPE. Hay solo uno, por lo que no es necesario modificarlo.

Datos administrativos

En esta categoría, se incluyen todos los elementos almacenados en la base de datos MongoDB.

La mayor parte de estos elementos se entrega a los clientes como parte del add-on. Sage quiere evitar conflictos con las denominaciones de otros elementos procedentes de las extensiones de otros partners durante la instalación del paquete en el entorno del cliente.

La solución consiste en incluir tu identificador único como parte del nombre o al principio.

Desde la actualización 8, es posible utilizar un identificador entregado, que permite etiquetar los elementos y bloquear su modificación, a menos que hayas iniciado sesión con este mismo identificador.

Identificador entregado

Desde la actualización 8, se puede asignar un identificador entregado a todas las opciones de administración.

Para activar esta funcionalidad, modifica el fichero nodelocal.js y autoriza este identificador.

Esto tiene dos ventajas:

  • Los clientes no pueden modificar tus desarrollos; solo pueden duplicarlos y cambiarles el nombre de ser necesario.

  • Puedes extraer todos los elementos vinculados a un identificador entregado en un solo parche.

Parametrización del identificador entregado

Como partner, puedes definir tu identificador entregado en un perfil de seguridad si el fichero nodelocal.js autoriza esta opción.

Un usuario con este perfil de seguridad puede marcar los datos como entregados.

Tú también puedes hacerlo en las entidades titulares del módulo de administración y marcar las páginas personalizadas como entregadas.

Sage utiliza este método para proteger los datos entregados y evitar que el cliente implemente actualizaciones incorrectas. Puedes duplicar los datos con el botón Guardar como, pero no modificarlos ni suprimirlos.

Elementos de menú, módulos y submódulos de menú

Estos elementos se pueden crear de dos formas:

  • directamente con la entidad Syracuse
  • con la herramienta de importación

Las reglas se adaptan para que sean coherentes.

La herramienta de importación crea elementos de menú solamente para las funciones clásicas y los gadgets de la versión 6.

Desde la versión 9, las reglas de denominación para crear los elementos personalizados o verticales son las siguientes (FUNCTION representa el código de la función):

  • SPE_X3_ERP_FUNCTION para las funciones de Sage X3 personalizadas
  • SPE_X3_HRM_FUNCTION para las funciones de Sage X3 HR & Payroll personalizadas
  • VER_X3_ERP_FUNCTION para los add-ons o las funciones verticales de Sage X3
  • VER_X3_HRM_FUNCTION para los add-ons o las funciones verticales de Sage X3 HR & Payroll

No es necesario que añadas tu identificador único porque el nombre de la función ya debería incluirlo.

La diferencia entre la denominación vertical y la específica durante la importación está relacionada con el parámetro que se utiliza al lanzar la copia:

  • Si el rol está vinculado a un perfil de seguridad con un identificador entregado definido, marca la casilla Parametrizar elementos importados como Factory de la página principal.
    Si la casilla está marcada, se utiliza el prefijo VER_.
  • De lo contrario, se utiliza el prefijo SPE_.

Al utilizar esta herramienta, solo se aplica una denominación independientemente del nombre de las funciones. Por ejemplo:

  • Si la casilla está marcada, se podría crear VER_X3_ERP_ZFUNC en una importación, a pesar de que no se deba utilizar el prefijo Z en las funciones verticales.
  • Si la casilla no está marcada, se podría crear SPE_X3_ERP_XFUNC en una importación, a pesar de que no se deba utilizar el prefijo X en las funciones personalizadas.

Para garantizar la coherencia global, aplica las mismas reglas cada vez que crees un elemento de menú para una función clásica directamente en lugar de hacerlo con la herramienta de importación.

En las funciones de Syracuse que añadas, ninguna función de importación crea las entradas automáticamente, por lo que tienes que crear el elemento del menú, el submódulo o el módulo.

Todavía no hay reglas definidas para este caso, pero deberías proceder de esta manera:

  • Incluye un identificador único como parte del código.
  • Incluye el nombre del add-on en el título.
  • Utiliza el identificador entregado para etiquetar el elemento.

Personalización de páginas

Al personalizar las páginas:

  • Introduce un código que incluya tu identificador único.
  • Añade un título y una descripción. El título puede contener el nombre del add-on al ser la única información que el usuario final puede ver. El código no se muestra.
  • Parametriza la opción Guardar como. Entregado solo está disponible si:
    • la función EnablePartnerID se ha parametrizado;
    • el perfil de seguridad tiene asignado un identificador entregado.

    El identificador entregado vinculado a tu perfil de seguridad está definido para este elemento, por lo que no hace falta introducir un valor.

Otros datos administrativos

Puedes:

  • comenzar la codificación con un identificador único;
  • incluir tu identificador único en cualquier posición de la clave.
