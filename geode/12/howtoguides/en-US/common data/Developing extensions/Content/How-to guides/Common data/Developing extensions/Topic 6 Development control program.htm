Development control program

Introduction

A common observation about the technology is that the tool gives you the freedom to make many modifications and provides a reasonable way of protecting them using specific or vertical activity codes.

But a common complaint about the technology is that the patching process is cumbersome and hard to automate, mostly because conflicts and errors linked to the freedom given need manual rectifications when they happen.

It becomes especially critical for Cloud implementations because our customers expect at the same time:

  • To have the ability to bespoke or vertical developments in these environments.
  • To ensure that the application of standard patches can be done automatically without conflicts.

This is why it is necessary to restrict the way partners can create bespoke and vertical developments. This can be given by defining rules that give norms, but also restrain dictionary modifications. These rules are not only mandatory in Cloud implementations. They are also critical to ensure to all our customers a smooth upgrade path.

Sage created a checking tool that runs on existing environments. This tool was improved in Update 10. This document describes the rules that will be checked by this latest version.

This tool only checks the consistency of the dictionaries (meta-data). In the future, Sage expects to improve the tool to also check the sources.

As the dictionary is designed today, it might be hard to follow the rules in some cases, especially in classic code. The Syracuse code has been designed to make it possible to follow these rules in these contexts.

Launching the tool

The tool is supplied as a classic function called ACTLSPE. It includes a window where you can enter additional parameters. It must be executed on the folder where the vertical and bespoke development has been done.

The parameters to be entered are:

  • The folder to be tested
  • The reference folder (usually the application folder)
  • Optional ranges for specific or vertical elements such as names and numbers for local menus and miscellaneous tables

The script runs and returns a log file with four levels of information:

  • Information: Describes safe changes that might have an impact if done differently by another supplier. For instance, increasing the length of a standard data type. This is just a reminder about points to be checked when several vertical or bespoke developments are installed together in a given environment.
  • Warning: Describes changes that are not considered errors, but might become errors in the future. For instance, if the dictionaries are enhanced to ease the upgrade by supplying a different way to develop some features.
  • Error: Describes an error in the design that will prevent a proper automatic upgrade. This prevents an installation on Cloud environments but can be managed for on-premise environments. It makes the upgrade process more difficult because you will have to check these elements.
  • Fatal error: Describes an error that prevents a proper upgrade to be done; This is the case for modification of dictionary elements.

Detail of the rules

Naming conventions

The first important point is that naming conventions are provided to handle the potential conflicts between partners supplying vertical developments or add-ons. Every partner willing to develop such features receives codification ranges. The rule is the following:

  • Add-ons range names start with X
  • Vertical range names start with Y
  • Bespoke range names start with Z

This rule applies only to the highest level of nesting associated with the element. For instance:

  • A vertical screen must follow the naming rule, but the sub-elements (fields) can be named freely.
  • On a standard screen, a vertical added field must follow the naming rule.
  • The same rule applies for columns and indexes on tables, collection names, or methods on classes and representations.

The elements controlled based on the naming rule are: tables, screens, windows, objects, inquiries, actions, data types, classes, representations, activity codes, parameters definitions, scripts, and reports. The sub-elements of standard elements are checked as defined before.

The miscellaneous tables and local menus have numeric ranges.

Some elements of dictionaries are considered as not standard even though they do not start with X, Y, or Z:

  • Functions starting with CONSX, CONSY, CONSZ (inquiries)
  • Windows starting with DCNSX, DCNSY, DCNSZ, FCNSX, FCNSY, FCNSZ (associated to non-standard inquiries)
  • Scripts starting with SPE (scripts starting with SPV must be renamed from Update 8, see below)

Dictionary elements and scripts starting with W are generated elements. It is therefore forbidden to create directly such elements. They are normally generated from a template which name does not start with W.

Any element following these naming rules is a not-standard element that must be protected by an activity code starting with X, Y, or Z. Any element not following the naming convention and not present on the application folder is an added non-standard element that is not following the naming convention (this is an error).

Dictionary elements starting with A are supervisor elements. These elements follow specific rules. Modifying them usually returns a fatal error.

Restrictions on classic dictionaries

The following restrictions apply to classic dictionaries:

Dictionary element

Operation

Reason and comments

Standard table, mask, window, object, action definition (header)

Set a specific activity code.

Replace standard activity code by another if it exists.

Assigning a specific activity code will deactivate any standard update on these entities.

Standard columns, fields, properties in standard tables, masks, classes, representations

Modify their characteristics even if they are protected by an activity code.

This prevents standard evolutions in the future. This rule is more restrictive than the previous recommendations. Modifications can be done on standard data types, and changes in localization and titles are done differently (by authoring). More restrictive controls can be done in scripts.

Standard column in a table

Change its data type, the corresponding length for the generic data types (L, DCB, A…), or the local menu associated with it.

The impact cannot be measured on the standard code. Internal variables might be defined according to the standard column data type. If the size of a column needs to be enlarged, the generic data type must be replaced by a non-generic data type.

This kind of request has to be requested like an entry point creation (with the same priority).

Standard miscellaneous table definition

Decrease the length of the code or change the length if not allowed.

Impact cannot be measured on the standard code.

A check box defines if the length can be changed.

Additional rules or comments

The following additional checks are done:

  • As stated before, any modification on supervisor tables is considered a fatal error.
  • The consistency tool checks only the local menus and not the chapter dedicated to messages. Gaps are signaled as errors. For example, if the choice N and the choice N+2 are defined without a choice N. The check is done only on the languages managed in the folder.
  • Any non-standard element not protected by a non-standard activity code is displayed as an error.
  • Any non-standard element not named in the non-standard codification range is displayed as an error.

Dictionary elements starting with W are excluded from the log. The system considers that the consistency has already been checked on the source elements of the generation.

Adding buttons in standard windows is sometimes necessary, but this creates issues that we cannot solve correctly:

  • Only four values are available for non-standard extensions.
  • Conflicts between verticals cannot be solved and need a specific check when several verticals are installed together.

The normalization applied in Version 7 makes it possible to create as many links as necessary, with a codified name.

Specific recommendations should be provided to people creating specific automatic journals. The best practice is to duplicate and rename them to handle more easily their upgrade using comparison tools. Parameters are available to re-assign the automatic journal code used when a posting operation is done.
