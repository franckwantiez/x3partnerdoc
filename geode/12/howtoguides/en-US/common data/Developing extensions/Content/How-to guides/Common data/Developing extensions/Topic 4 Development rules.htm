Development rules

Add-ons must respect a set of rules to avoid technical conflicts due to extensions' cohabitation.

This chapter:

  • Focuses on the main rules you have to apply

  • Highlights limitations you have to take care of

  • Gives you advice on how to design and test your developments

All topics described are available since Version 7.

Main development rules to apply

Field actions

From the platform, you can define actions on screen fields.

A feature is available to use a vertical action (SPV) on each event (initialization, control, etc.) on each field.

But there can be only one action of this type in each field.

In the add-on mode, all partners share the dictionary. You need to use dictionary actions to avoid conflicts.

The screen dictionary vertical script is a unique program and cannot be shared easily by different partners.

Dictionary action names should start with your identifier. You also need to protect them with an activity code.

This rule applies to any action that you add in a field on a standard screen.

If you are alone to work on the vertical script, you can use SPV actions for your screens that:

  • Your activity code protects

  • Have no verticalization on them

Only use dictionary actions for your field actions.

Do not forget to set an activity code on the action. The activity code's name should begin with your unique identifier.

How to manage the vertical layer with an object model and entry points

A separate script manages each of the three layers: specific, vertical, and standard.

In each script, developers use the entry points provided by the different models on which they are working.

The source code of the model or object – which is the most used – is not provided.

The documentation provides the list of available entry points and a set of variables that you can use to interact with the standard.

There is only one vertical script for a function.

For this reason, different actors could not easily share the script. You had to create and use a manually managed switch program that sequentially called the scripts of the different partners.

The procedure is:

  • Before Update 8: You needed to manage the switch program manually. This caused a lot of problems in the past.
  • Since Update 8: The supervisor automatically manages the switch program for objects and entry points. This is a great improvement.

Vertical layer with objects

Before, the name of the vertical script was defined in the object. The switch program's name was used to authorize multi-vertical calls.

Today, there is a check box in the object definition. If this check box is selected, there is at least one vertical on the object.

An info button gives you the names of all the vertical processes defined for this object.

The definition of the vertical scripts to use on an object is set in the entry points table.

The order in which the different scripts should be executed is also defined in this table.

You should pay attention to the following:

  • The Entry points table is loaded as a set of global variables when you create a classic session. If you modify the entry points list, you should disconnect and reconnect for the modifications to be applied.
  • When the table is loaded in memory, there is a control that checks that the adx script defined in the entry points list exists. If not, in the folder or one of its mother folders, it is not added in the memory table and thus not executed even if you add it after creating your session. You need to disconnect and recreate the memory table.

Understanding the execution order of the different layers

The same logic applies to each entry point.

The model calls the following elements if they exist in this order:

  • The specific script
  • All the vertical scripts according to their rank order
  • The standard script

Two global variables are used to control the different scripts' execution:

  • GPV: Prevents the execution of the vertical layer if set to 1.
  • GPE: Prevents the execution of the standard layer if set to 1.

These two variables are set to 0 by the standard before the execution of the global loop.

Then:

  • If GPV is set to 1 in the specific script, all the vertical scripts will be skipped and not executed. This behavior should be validated on a case by case basis on a customer site.

  • If GPE is set to 1 either in the specific script or in one of the vertical scripts, the standard layer will not be executed. This behavior is exceptional. It is generally adopted when the standard layer must be executed before the vertical or specific layer. Do not skip the standard layer.

When several scripts (for example, a specific script and several vertical ones) are executed, the GPE variable's value is the last one to be set. In these complex interaction cases, the behavior must be validated on the customer’s site.

Complex cases in entry point execution

One of the verticals needs the standard layer to be executed first.

Example

Three vertical scripts are defined for the BPC object.

For example, the XY23BPC2 script needs the standard to be executed first in the PICKE entry point.

In this case, the script is:

$ACTION
Case ACTION
When “PICKE” : Gosub ETIQP
………
Endcase
Return

$ETIQP
# Execute the standard code first
Gosub ACTION From SUBPTH

GPE=1   :# Avoid to reexecute the standard
#  Vertical code
……
……
Return

In this case, you need the third vertical XX34PH78 script to be executed next.

On a customer site, the different verticals' execution order must be set according to specific requirements.

If a vertical requires the standard to be executed first and not the others, this vertical must be executed last, and the execution order needs to be adapted.

Also check that another vertical:

  • Is not setting GPE to 0 after your script has set it to 1. In this case, standard will be executed twice.
  • Is not executed before the standard.
These are exceptional cases. They need to be checked on a customer site as it is impossible to manage and anticipate all vertical combinations.
For rare cases, the order must be different according to the entry point. On the customer site, one of the verticals must be split to meet this specific need because no modeling manages this requirement.

Patching Entry points as of Update 8

  • Entry point patching:
    The execution order is never modified through a patch. But, when a new entry point is patched, the execution order is set if it has been defined.
  • At folder revalidation:
    The vertical script, if it exists on an object, is automatically inserted in the entry points table.

How to manage the vertical layer with other models: switch program

With other models for which the switch program is not created automatically, you need to manage the switch manually.

You need to do this for:

  • Actions
  • Inquiries
  • Import/export templates
  • Reports

Inquiries, import/export templates, and reports are generally duplicated when you want to customize them. You mainly need to focus on scripts associated with actions.

For these different items, there is no vertical layer.

The only available script is the Specific script that will be used as a switch in all cases.

Developing reports

Standard reports are developed with the Crystal Reports tool.

Here is some advice on how to develop your reports:

  • Use the Crystal Reports reports development charter defined in the online help center.
    Develop your reports using one of the following report templates: ATEMPLATE_RPT1 or ATEMPLATE_RPT2.

  • The report's name should start with your identifier and have the same name as the dictionary report code.

  • All reports must be based on an ODBC source.
  • You must provide reports in the languages that you manage as a partner. English is mandatory.
  • Reports will comply with the management of authorizations development rules.

Refer to the training support material.

Important limitations

The technological platform has some known limits. Sage is trying to fix these limitations.

Pay attention to these points with classic functions:

Number of columns in a table

Is limited to 512. When working with a standard table close to the limit, create and manage your complement table to avoid problems.

Length of a line in a table

Is limited to 8 Kb.

If the standard table is close to the limit, you can add and manage your own table.

Number of blocks in a mask

Is limited to 15.

You should add a new mask if you add several blocks and approach the limit.

The authoring enables you to draw a layout that fits your needs.

Number of fields in a mask

Is limited to 500 in the definition.

Is limited to 300 in validation.

You should add a new mask if you add several fields and approach the limit.

Interactions with the standard

Avoid making modifications to the functionality of a core element, especially for localizations. These functionalities are intended to co-exist in the same customer folder. It is important to remember that an add-on is a solution to enhance and extend the core functionality and not to bypass it.

Key points to remember:

  • Do not use your activity code to change or delete a core element.
  • Do not disable a button, menu, field, etc.
  • Do not change the behavior of a button or menu.
  • Use only GPE= 1 to call the core directly but not to avoid it.
  • Use a menu rather than a button in a core window.
  • Use your own screen in a core window or create your own additional pop-up.
  • Avoid sharing the same available space on core screens with different add-ons installed within a single customer folder.
  • This kind of conflict must be managed by the project implementation team.

Be careful, when you complete a core function. If the vertical or specific layer is already being used by an add-on or a vertical asset, do not override the existing switch processing belonging to this add-on or vertical asset.

In this situation and for future developments, use the strategy of the switch processing that calls your applicative processing.

  • To use an entry point, use the strategy of the switch processing to help update switch processings.
  • Create a separate patch for them. Then put an example of what the source processing should be in a dedicated directory named ADDSRC that you deliver in your patch.
  • Complete the readme file adding the reference of switch processings and the corresponding core processing.
Never modify the usage of a core element.

Testing

To test a core function:

  • Verify the global functionality of the core function after add-ons or assets have been added.
  • Verify the switch processing in the vertical layer, in a specific layer for some dictionaries, and in entry points.
  • Check that co-existing field actions work together in a core screen.
  • Check that menus and buttons work for several add-ons in a core window. Avoid having the same menu code for two add-ons in the same core window.
  • Check that the platform capacities are not exceeded (number of fields in a table, number of tabs in a window, etc.).
