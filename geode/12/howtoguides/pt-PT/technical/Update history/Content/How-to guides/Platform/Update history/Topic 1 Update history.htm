Update history

Open: Development > Utilities > Patches > Update

The cumulative patching function introduced in version 2019 R4 of Sage X3 has been constantly improved since. This function performs updates to the latest release regardless of the original release. It also fills a table that gives the complete history of the updates done on all the solution's folders. You can track precisely how the folders were updated.

The history stored here is for all folders. You can see the various steps the folders went through to reach the level of release they are in today.

The left list of this function displays the following information:

  • The sequence number. It starts from 1 and increases every time an operation updates a folder's global structure. Usually, it is an upgrade, but it can also be a folder import and a folder validation. In the example below, you can see the upgrades done on all the folders of a solution. There are four folders: X3, REFMAD, SEED, and TSTMAD.

  • The type of update. For now, it is Standard in all cases.

  • The internal version. This internal version number is associated with the release just before its delivery. The appendix at the end of the document gives the correspondence.

In the example above, lines 3 to 9 are linked to the same internal version, sometimes on the same folder. These lines concern operations such as folder validation or import. Lines 10 to 13 are related to upgrades. They bring up all four folders to version R090.023.00 (2020 R3). Then there are again four lines to upgrade the four folders to R090.24.054 (2020 R4). Finally, there is another upgrade to R090.025.061 (2021 R1). This is typically an environment in which the upgrades have been applied gradually.

In the example below, there are several update steps from R090.020.00 (2019 R5) to R090.21.000 (2020 R1), and then a direct jump to R090.025.061 (2021 R1).

If you select a record in the left list and look on the right side, you can see all the details about the operation performed in the corresponding folder.

The important information on this screen is:

  • The Process type, which can have the following values:

    • Integration: An upgrade has been performed.

    • Validation: A folder validation has been performed.

    • Import: A folder has been imported with the standard import feature.

    • Restoration: A folder has been imported with the bulk database import.

  • The Update type, which can have the following values:

    • Standard: A standard upgrade has been done (or any other operation).
    • Hot-fix: The integration of a hot-fix has been done.

The first table lists the Catalogs that have been installed. It refers to an internal entity generated by the Sage upgrade extraction procedure. A catalog refers to an upgrade that can be done without interruption of the upgrading process. Usually, there is a unique X3_resultcatalog line that corresponds to the main update; but, another catalog (X3_resultPATCHScriptcatalog) can be present in the first line. This second catalog corresponds to a patch done on the patching system. This patch is usually done first to allow the patcher to benefit from the latest performance improvements and run faster.

A right-click allows access to the log files of the integration. Two log files are generated:

  • The first one contains the log of the preliminary control steps.

  • The second one is the one generated by the integration by itself.

The second Detail table lists all the elements that have been patched. For a standard upgrade, this grid can contain thousands of lines. Note that you will only find here elements that needed to be patched (it depends on the version you started from) and not all the elements present in the upgrade file.

It is helpful to know which elements were patched during a particular upgrade. The Detail filter button opens a window where you can enter filters on all the grid's columns. For example, if you want to know which windows have been upgraded, you only need to select the object type AWI.
