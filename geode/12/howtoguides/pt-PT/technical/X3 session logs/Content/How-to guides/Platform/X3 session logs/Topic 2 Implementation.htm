Implementation

Open: Administration > Usage > Logs > X3 session logs

This function allows administrators to define a logging session along with the desired parameters to:

  • Identify logging sessions using tags and descriptions

  • Filter out undesired sessions and focus on desired users and functions

  • Set up the relevant log levels and log details as needed

  • Enable or disable logging for the current session

You can see a list of initially existing log sessions, including the following information. Select the New X3 session log action or the Edit button on the existing log records to access the details.

Files: Label, description, and maximum log time

The first elements you set up identify your log settings and define how the files will behave:

  • Label: This helps you identify your log settings and is included in the file name. You can easily match the file with the log settings.

  • Description: A user-friendly description for your log file sets.

  • Max log time: The maximum time in minutes after which logging is stopped and the log setting disabled. This allows you to avoid logging sessions indefinitely which can expose users to a drop in performance.

Target: Set up what you want to log

Define what you need to log in the Target section.

The Type allows you to focus on different areas of the solution:

Batch administration

This logs administration actions related to the Sage X3 batch server, such as starting or stopping the batch server. You will be prompted to select which batch server to monitor.

Batch query

This selection logs all operations performed by a batch job session.

Web service

This covers web service sessions and their execution.

Representation

This logs all operations performed by representation sessions (for example, sessions using hybrid class/representation objects).

Function (Classic)

This allows you to target classic (object) functions of Sage X3.

When this is specified, Sage X3 will create a log file each time a classic function is accessed if it meets the filtering criteria:

  • The selected endpoint (mandatory)

  • The specified function code (for example, GESITM, GESSOH, and so on). The function code can be located by:

    • Hovering over the function menu:

    • Looking up the function in the Function dictionary (Development > Script dictionary > Functions).

    If no function is specified, all classic functions will be logged.

You can also select the user to focus logs on a single Sage X3 user.

When a user accesses a classic function that is covered by a log setting, a new session is created even if existing classic runtime sessions are available for that user. A new log file is created each time the user accesses a classic function or navigates from one function to the other.

Log levels: Define the details to log

You can select log levels from a list. Log levels depend on the type of entity (Type) that you selected.

Here is a list of the most common log levels available:

Log level Details Logged data / Examples
LOG_TRT0 Function calls with timing Logs 4GL calls to functions or labels (Gosub/Call/func) and produces timing statements for performance analysis. It does not show the details of the lines that are run within the subprograms or labels.
LOG_TRT1 All 4GL statements Logs all 4GL statements run, with timing information attached whenever there is a call to a subprogram or label, similar to the previous level. Only the first instruction displays when the source code is not available.
LOG_SQL0 SQL Read and For Logs database SQL queries triggered by 4GL For and Read statements.
LOG_SQL1 sadora and sadoss Covers specific sadoss and sadora service operations.
LOG_JSON JSON exchanges Logs all JSON exchanges with the runtime.
LOG_CLNT Classic mode exchanges Logs protocol streams between the runtime and the client part of the application (front-end web page) for classic objects.
LOG_LDAP LDAP requests Covers all LDAP protocol exchanges (LDAP management service).
LOG_OPAD opadxd exchanges Traces explicit opadxd operations. For example, socket operations with other runtime instances.
LOG_STRT Runtime initialization

Logs data exchanges, variable and memory initializations, and so on.

Done by the runtime when a session is started. For example, when a user accesses a classic function.
LOG_DBUG Standard debug trace Replicates the standard debug log file generated when running 4GL code, which lists all 4GL instructions executed by the runtime.
LOG_SYSC Security checks Logs security transactions (Sandbox, runtime configuration security).
LOG_X3SM Session management Covers application and database sessions creation/deletion.
LOG_FILES File access Logs all file access operations done by the runtime.
LOG_MEMORY Memory usage Adds memory usage information to the log.

You can select as many log levels as needed. The selected log levels result in a unique compounded number displayed in the Levels digest field along with the list of selected log levels.
