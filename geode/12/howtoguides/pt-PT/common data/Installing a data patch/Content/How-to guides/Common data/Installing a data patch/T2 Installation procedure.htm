Installation procedure

Related functions
  • Patch integration (PATCH)
  • Processes (GESAPR)
  • Transaction generation (GENMSKTRT)

Standard procedure

The installation of a data patch that relates to data with the key element LEG (legislation), such as the visual processes, is carried out in several steps:

  • Step 1: Installing the data patch on the Application endpoint (the parent endpoint).

  • Step 2: Installing the data patch on an operating or test endpoint (multi-legislation endpoint).

  • Step 3: Installing the data patch on a single-legislation endpoint.

  • Step 4: Validating the visual process transactions on each endpoint using the Transaction generation function (GENMSKTRT).


The installation procedure is different according to the operating or test endpoint legislation mode (multi-legislation or single-legislation).

On a single-legislation endpoint (LEG activity code = No), the LEG field is not displayed, although it is populated with the LEG of the endpoint (GDOSLEG).

On a multi-legislation endpoint (LEG activity code = Yes), the LEG field is displayed. When creating a record in a multi-legislation table, the value of this field must be initialized with the LEGFIL value.

General rules

This procedure makes it possible to comply with the rules defined for data with the LEG key element on a multi-legislation and single-legislation endpoint:

  • On a single-legislation endpoint you can only manage data if the LEG field is populated with the endpoint legislation.

  • On a multi-legislation endpoint you can manage data with the LEG field empty (the record is valid for all legislations), or with the LEG field populated (the record is then valid for the populated legislation only).

Step 1: Installing the data patch on the Application endpoint

This step installs the data patch on a multi-legislation Application endpoint (the parent endpoint).

The installation uses the standard Patch integration function (PATCH). The process is incremental.

Sage X3 will only install the data without a legislation and the data related to the legislations installed on the endpoint.
The standard Patch integration function (PATCH) suggests the update for all endpoints, that is the parent endpoint, and the single and multi-legislation endpoints.
Do not forget to remove them from the list.
Example

A data patch contains the following pairs:

  • Visual1, empty (Visual process, Legislation)

  • Visual1, FRA (Visual process, Legislation)


Sage X3 creates one pair on a GER, BRI multi-legislation endpoint: Visual1, empty.

Sage X3 creates 2 pairs on a FRA, BRI multi-legislation endpoint: Visual1, empty and Visual1, FRA.

Step 2: Installing the data patch on a multi-legislation endpoint

As for step 1, the installation uses the standard Patch integration function (PATCH).

Sage X3 will only install the data without a legislation and the data related to the legislations installed on the endpoint.
The standard Patch integration function (PATCH) suggests the update for all endpoints, that is the parent endpoint, and the single and multi-legislation endpoints.
Do not forget to remove them from the list.
You can use the Copy process described in Step 3: Installing the data patch on a single-legislation endpoint. However, please note that each visual process must be copied individually. You cannot copy them in batch mode.
Example

A data patch contains the following pairs:

  • Visual1, empty (Visual process, Legislation)

  • Visual1, FRA (Visual process, Legislation)


Sage X3 creates one pair on a GER, BRI multi-legislation endpoint: Visual1, empty.

Sage X3 creates 2 pairs on a FRA, BRI multi-legislation endpoint: Visual1, empty and Visual1, FRA.

Step 3: Installing the data patch on a single-legislation endpoint

The data patch cannot be directly installed on a single-legislation endpoint.

You must copy the updated visual processes from the Application endpoint to the single-legislation endpoint.

Use the Processes function (GESAPR). Click the Copy action to manually copy each visual process to the single-legislation endpoint.

The Copy action applies the following rules:

  • If the data is linked to a specific legislation, only the process related to the legislation of the single-legislation endpoint is copied.

    If the same data already exists on the single-legislation endpoint, the process is updated.

  • If the data is not linked to a legislation, the process is copied to the single-legislation endpoint, now populated with the legislation of the endpoint.

    If the same data already exists on the single-legislation endpoint, the process is updated.

Only copy one process when data is both not linked to a legislation and linked to a specific legislation.

Copy the process linked to a legislation if it is the same legislation as your single-legislation endpoint. Alternatively, copy the process without a legislation.

Example

An Application endpoint contains the following pairs:

  • Visual1, empty (Visual process, Legislation)

  • Visual1, FRA (Visual process, Legislation)


The Copy action copies the pair Visual1, FRA to a single-legislation FRA endpoint.

The Copy action copies the pair Visual1, empty to a single-legislation BRI endpoint. This creates (or updates) the pair Visual1, BRI on a single-legislation BRI endpoint.

Step 4: Validating the visual process transactions

The visual processes are not fully operational on the server immediately after installation or update. At this stage the links defined on the processes will not work. They need to be recoded to be compliant.

To make the visual process transactions compliant with Sage X3 you must validate them using the Transaction generation function (GENMSKTRT) on each endpoint where the visual processes were installed or copied:

  1. Select the folder (endpoint).

  2. Select Yes in the Validation column (APR code).

  3. Click OK to launch the validation process on the selected endpoint.

Repeat these steps for each endpoint on which the visual processes were installed.
