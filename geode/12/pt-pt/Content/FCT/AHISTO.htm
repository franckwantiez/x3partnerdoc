Histórico/Expurgo

Esta função permite de lançar as operações de expurgo e de arquivo.

Os softwares em tecnologia SAFE X3 podem gerar, pelo meio da função de expurgo / arquivo, a transferência versus os arquivos ou o expurgo de movimntos que não evoluem mais, e que são considerados como suficientemente antigos para não mais serem consultados frequentemente.

Apenas são expurgáveis ou arquivados os dados considerados como saldados (ie. não susceptíveis de continuar a evoluir )

O arquivo se faz por transferência dos dados de uma tabela do dossier versus uma tabelas de mesma estrutura (a dois campos juntos, CREHISDAT e CREHISUSR, que permitem de conhecer as datas e operadores tendo sido arquivado o dado). Esta tabela está definida num dossier dedicado, que utiliza por outro lado o dossier corrente como dossier mãe, e então a criação está feita automaticamente no primerio arquivo.

O interesse desta maneira de fazer, reside no facto que as funções de consulta standard sobre os dados do histórico funcionam então por simples conexão sobre este dossier, da mensma maneira que elas funcionam sobre o dossier em curso.

Quando se está conectado sobre este ficheiro arquivo, não se poderá ver os dados arquivados que separadamente, e simultâneamente com os dados de mesmo tipo não arquivados. Por outro lado, se uma tabela arquivo faz referência a dados de base (não susceptíveis de serem arquivados oir definição) ou a dados para os quais nenhuma tabela arquivo foi definida, a consulta poderá se fazer por zoom sobre estes dados não arquivados.

A parametrização das regras de expurgo permite definir as durações de vida a partir das quais os movimentos podem estar expurgados ou arquivados, em seguindo as regras de coerência por grupo de tabelas definidas por outro lado.
Em função do módulo ofício afetado, o tratamento pode se apoiar igualemente sobre o valor de certos parâmetros. Sua lista está dada em um documento anexo.

Se deseja ver em específico dos movimemtos situados por certos no dossier de exploração e para certos no dossier arquivo, falta definir os pedidos SQL de tipo união sobre uma das duas tabelas afetadas, por exemplo por meio de pedidos SQL ou de vistas especificos.

Pré-requisitos

Falta ter previamente criado o dossier de histórico associado ao dossier corrente se deseja gerar um arquivo, e do mesmo ter definido os parâmetros de expurgo / histórico corrrespondentes.

Gestão do ecrã

  • [Tab] Ecrã de registo
  • Observações

[Tab] Ecrã de registo

Quando se entra na função, encontramos um único ecrã permitindo de definir as informações afectando o expurgo a lançar.

Estes parâmetros preenchidos, a validação do ecrã provoca o lançamento da função. Quando esta está terminada, um rasto será afixado para conhecer, da meneira mais ou menos detalhada, o resultado.

[Block number] 1

  • Cód. ([field] COD)

Define o código de hidstórico/expurgo a lançar. Isto permite restringir a operação a um único grupo de tabelas Se se deseja lançar uma função geral, é suficiente de deixar este código vazio.

[Block number] 2

  • Histórico ([field] HISTO)

Define se o tratamento de histórico deve ser lançado.
  • Depuração ([field] EPUR)

Define se o tratamento de expurgo deve ser lançado.

[Block number] 3

  • Todas socied. ([field] ALLCPY)

Se esta check box está assinalada, o tratamento é lançado para todas as sociedades (senão, especificar-se-ão as quais).

  • Socied. ([field] CPY)

Código da sociedade afetada pelo tratamento.

[Block number] 4

  • Rastro detalhado ([field] TRDET)

Se esta check box está assinalada, ela provoca o registo, no rasto da operação, da lista de cada uma das linhas expurgadas ou arquivadas (na ocorrência, as chaves principais das tabelas principais do grupo de expurgo; o detalhe das linhas expurgadas ou arquivadas nas tabelas ligadas não foi jamais dada, mas indicar-se-á neste número). Se esta check box não foi assinalada, ter-se-á, para cada tabela, uma mensagem dando o número total de linhas expurgadas e arquivadas para a tabela.

Esta check box igualmente activa em caso de simulação, para conhecer o detalhe de que será feito se a função está lançada em real.

Atenção, utilização desta check box pode provocar a criação de ficheiro de rasto muito volumosa.
  • Simulação ([field] SIM)

Se esta check box está assinalada, a função se conterá de criar o rasto (detalhado ou não) desde que o tratamento fará sem realizar realmente o expurgo e ou o arquivo.

Observações

Esta operação pode ser extremamente longa. É em geral interessante de lançar em batch, e de maneira regular (os parâmetros de frequência por grupo de tabelas permitindo de definir o que deve ser feito a cada lançamento).
