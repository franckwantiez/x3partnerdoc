Escolha personaliz.

As tabelas diversas são tabelas anexas utilizáveis em todo o contexto do software para aí guardar os códigos e um número restrito de informações anexadas. Por exemplo, guardar-se-á os regimes de taxa, os modos de pagamento, os motivos de incobráveis...

Estas tabelas banalizadas, identificadas por um número, permitindo de guardar os elementos (intitulados traduzíveis, informações complementares, tabelas ligadas), identificadas por um código alfanumérico. Certas informações ligadas às tabelas diversas são da parametrização (o que quer dizer que a modificação destas características está perene em caso de alteração de versão). Elas são em consequência modificáveis por esta função. As informações modificáveis por parametrização são os seguintes :

  • o comprimento máximo dos códigos de certas tabelas diversas, sobre certas tabelas (desde logo que este comprimento é modificável, o que está definido ao nível do desenvolvimento). Em todos os casos, um parâmetro, nomeado MAXADI, definido em gestão de dossier, dá o comprimento máximo admissível (em versão 130, este parãmetro não existe, o comprimento sendo sempre igual a 3).
  • o código de acesso, que permite fechar o acesso à tabela (se um código de acesso está presente, o utilizador deve ter os direitos em modificação sobre o código para aceder à tabela em modificação).
  • O número de uma outra tabela diversa, na coluna Dependência, que permite definir que o registo de um valor controlado por uma tabela diversa está ligado ao valor do código registado na tabela ligada (o que permite definir uma primeira tabela contendo os códigos de uma família, e uma segunda tabela contendo os códigos de uma sub-família, cada sub-família estando ligada a uma família). O campo do ecrã sobre o qual se deseja efetuar este tipo de controlo de dependência deverá ser modificado (Ver tabela dependente).
  • um código atividade, que permite tornar específicos (i.é. não reatualizados por um patch standard) todas as características da tabela.

As informações da tabela diversa não consideradas como a parametrização, mas um desenvolvimeto, seu indicador Comprimento modificável, Isto significa que toda a modificação deste tipo não pode estar feito que na função de desenvolvimento dedicado, que ela deve estar perenizada por um código atividade específico, e que pode ser transmitido por intermédio de um ficheiro de patch.

Gestão do ecrã

[Tab] Ecrã de registo

Vê-se aparecer, em quadro, as diferentes tabelas, e suas características modificáveis. Está a notar que o campo comprimento não está sempre modificável (segundo o que foi registado na função de definição das tabelas diversas), então que os outros campos o são sem restrições particulares.

Quando a validação está feita, se o comprimento dos códigos de acesso de uma das tabelas foi modificada, um ecrã está afixado, pedindo se é necessário de revalidar os ecrãs. Responder Sim a esta questão provoca a revalidação dos ecrãs utilizando esta tabela diversa (tipo de dados ADI com o número de tabela correspondente). Pode-se perfeitamente responder Não a esta questão, porque é possível que se deseja modificar outros elementos e que se faça uma validação mais global ulteriormente. É todavia aconselhado de a fazer se apenas esta modificação deve estar feita.

Se ela não está realizada em saída desta função, a validação poderá ser feita para cada tabela diversa modificável de duas maneiras seguintes:

  • pela função de Validação nos utilitários ligados ao dicionário, em indicando que se deseja validar os ecrãs utilizando o tipo de dados ADI com o bom número da tabela
  • por utilização do botão na definição das tabelas diversas do menu desenvolvimento, a tabela diversa em questão estando em linha.

[Grid]

  • Tabela ([field] NUMTAB)

Este número identifica a tabela diversa. Um certo número de números são pré-definidos no software, mas é possível de acrescentar, por um desenvolvimento específico, nas selecção de números reservados a este efeito.
  • Descr. ([field] LIBEL)

Título associado ao código precedente

  • Comprim. ([field] LNG)

Regista-se aqui o comprimento do código da tabela corrente. Este comprimento não pode exceder o comprimento máximo definido pelo tipo de dados ADI.
  • Código acesso ([field] ACS)

Este código de acesso permite de restringir o acesso a uma tabela diversa (em registo ou em modificação).

  • Dependência ([field] DEPNUM)

Indica-se aqui o número da tabela diversa que serve de referência para registar a coluna Dependência. No momento de utilização da tabela diversa, isto impõe uma dependência entre dois campos registados pelas tabelas diversas ligadas.
  • Cód. activ. ([field] CODACT)

Um código actividade permite :

  • de tornar opcional um elemento do dicionário se o valor associado ao código actividade é nulo.
  • de assinar os elementos específicos desde logo são marcados por um código começando por X,Y ou Z.
  • de dimensionar um número de linhas máximas quando o código de actividade maerca os elementos de um quadro.

Assim, se o código actividade está não activo, o elemento marcado não será utilizável, e o código associado (se existe um) não será gerado nem activável.

Mensagens de erro

  • [Generic messages]

[In addition to the generic error messages, the following messages can appear during the entry] :

Comprimento muito grande (N)

Esta mensagem significa que se ensaia de definir um comprimento superior ao parâmetro definindo o comprimento máximo de uma tabela; este parâmetro está definido na ficha dossier.

A tabela contém os códigos de comprimento M

Esta mensagem significa que se ensaia de diminuir o comprimento dos códigos de uma tabela diversa em baixo do comprimento máximo de um código já registado na tabela.
