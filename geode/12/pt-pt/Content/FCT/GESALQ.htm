Query SQL

Os softwares em tecnologia SAFE X3 integram um requerente permitindo criar os ecrãs de interrogação a partir dos períodos exprimidos sob a forma de uma lista de campos emitidos de tabelas de base, estas tabelas estando ligadas por defeito, seja por relações explicitas nomeadas. Esta aproximação permite a um utilizador criar rapidamente os pedidos sem utilizar a linguagem particular, em tendo simplesmente um conhecimento das tabelas de base, conhecimento que pode ser guiado pelas documentações tecnicas em linha (MCD nomeadamente). Obtém em outro uma interface conviva de interrogação nos ecrãs multi-colunas com as possibilidades de túnel.

Este requerente apresenta todavia alguns limites, na medida onde os pedidos complexos não podem ser forçosamente exprimidos sob a forma de uma lista de campos. Assim, para os utilizadores tendo um perfil mais técnico e conhecendo a linguagem SQL, este requerente não permite obter facilmente os resultados desejados.

É porque foi criado um utilitário complementar, o requerente SQL, que permitre de exprimir os pedidos em utilizando a linguagem SQL clássico. Este requerente SQL não se distingue do requerente precedente que pelo seu ecrã de parametrização. Com efeito, os ecrãs criados por este requerente são chamados pela mesma função deexecução de pedidos que aqueles utilizados pelo requerente habitual.

Isto significa que, como requerente, o requerente SQL funciona por extração temporária de dados numa tabela (a mesma que o requerente clássico), depois por consulta desta tabela. Por outro lado, a consulta não se pode fazer que sobre um nível, as ruturas e acumulados não sendo possíveis a este nível.

Assim que o requerente clássico, a estrutura desta tabela temporária faz que cada utilizador conserve os resultados do último pedido feito, e a utilização desta funcinalidade implica três etapas diferentes :

  • definição do pedido por esta função, depois Validação afim de criar o ecrã de consulta correspondente.
  • lançamento do pedido (seja pelo botão, seja por entrada na função de interrogação e pedido de refrescamento dos dados)
  • interrogação do requerente se já foi executada.

Cabeçalho

Em cabeçalho, encontramos as informações permitindo o pedido e alguns parâmetros de levar geral.

[Block number] 1

  • Pedido ([field] COD)

Este código permite de identificar um pedido.

  • Descr. ([field] INTIT)

Permite de definir um titulo associado a cada ficha.
  • Desc. curta ([field] INTITSHO)

Este intitulado permite dar uma designação à ficha.
é utilizado em certos ecrãs ou listas quando não se teve suficiente local para visualizar o intitulado longo.
  • Activo ([field] ENAFLG)

Esta check box a assinalar permite ativar ou de desativar a ficha corrente sem portanto perder o seu conteúdo.

Uma ficha desativada não pode ser utilizada (por chamada do seu código) em outras fichas (documentos, parametrizações...), ou no tratamento de massa.

As habilitações sobre uma função dada podem interdir a criação de uma ficha ativa. Neste caso, a check box está desativada por defeito, e está modificável unicamente por um utilizador autorizado, ou via um circuito de assinatura definido por Workflow.

Características

  • Tipo de pedido ([field] ALLUSR)

Permite gerar a afixação de um pedido em consulta, com os comportamentos seguintes :

  • Pedido normal : o resultado do pedido está afixado se o pedido foi já consultado pelo utilizador. Com efeito, cada utilizador lançando o pedido criado um jogo de dados que lhe é próprio, e que não pode ser visto por outros.
  • Pedido partilhado : o resultado da execução deste pedido por um utilizador pode ser consultado por outros utilizadores. O interesse de ter pedidos partilhados reside no facto que um pedido pesado, lançado a um momento dado por um único utilizador, permite em seguida a um conjunto de utilizadores dispor de uma informação comum. É recomendado, para este género de pedido, que um único utilizador possa executar o pedido, os outros tendo acesso em consulta. Por outro lado, é perfeitamente desocupado de calcular este tipo de pedido em batch se ela está particularmente pesado a constituir ; o código da tarefa batch correspondente se chama em efeito. CALALH.
  • Pedido recalculado : o resultado do pedido não se afixa jamais automaticamente. Falta apoiar o botão refrescar ou registar os critérios. O interesse deste tipo de pedido é de evitar de recarregar os dados se se sabe que elas se estragam rapidamente.
  • Representação ([field] REPGRA)

Este indicador define a maneira cuja representação dos dados vai ser feito no ecrã. As possibilidades oferecidas são as seguintes :

  • Carater :  os dados são unicamente representados sob a forma de um quadro.
  • Car. ou Graph. : Os dados são apresentados sob um dos dois modos. O modo por defeito está determinado pelo campo Afixação por defeito definido nos parâmetros gráficos da ficha.
  • Car. ou Graph. : os dados são apresentados sob os dois modos simultaneamente. A posição do gráfico por relação ao quadro está determinado pelo campo Posição do separador Gráfico.
  • Gráfico :  os dados são unicamente apresentados sob a forma de um gráfico.
  • [field] OPTGRA

Permite de organizar a cohabitação entre carater e gráfico, quando o modo de apresentação dos dados permite os dois Segundo o caso :

  • se a representação está Car ou Graph, escolhe-se o valor por defeito da representação.
  • se a representação está Car e Graph, dá-se a posição do gráfico por relação ao quadro dos dados

[Tab] Parametrização

O primeiro separador permite então de registar o pedido e alguns parâmetros gerais.

Vai-se assim poder dar, no quadro titulo Colunas, das informações permitindo dispor de uma apresentação mais precisa. Com efeito, na execução, o pedido reenvia uma lista de valores que vão preencher um quadro. Na ausência de informações mais precisas :

  • as colunas do quadro não terão de título.
  • elas serão tipificadas alfanumericamente, numérico, ou datas, sem mais precisão, em função dos tipos de dados reenviados.

De mais, um pedido SQL não fixa forçosamente muito interesse. falta poder introduzir os parâmetros que serão registados no momento do seu lançamento. É o que permite o quadro com o título Parâmetros.

[Block number] 1

  • Lista ([field] RPT)

Define o código de uma lista que será associada à função Ficheiro / Lista a partir da consulta dos pedidos. Se não está preenchido, utiliza-se a lista ALISTE, que proporá uma atualização em página por defeito. É possível duplicar esta lista genérica (não será em princípio necessário que de modificar a colocação em página) para criar as listas perfeitaemnte adaptadas ao pedido assim definido.

O código lista registado aqui pode ser assoicado a um conjunto de códigos de impressão via a função dedicada.
  • Função ([field] FCTLNK)

Permite ligar as autorizações de acesso ao pedido às autorizações dadas sobre a função.

O princípio é então o seguinte verifica-se, na entrada no pedido, que a função ligada está autorizada ao utilizador (para pelo menos um estabelecimento, se a habilitação sobre esta função se faz por estabelecimento). Mas esta verificação se fazunicamente no caso onde o pedido não foi partilhado.

Com efeito, se o pedido foi partilhado, parte-se do princípio que as habilitações da pessoa tendo calculado o pedido se aplicarão e se propagam a todos aqueles que têm o direito de visualizar. Convém então de definir de maneira muito cuidada as habilitações aos pedidos partilhados, quando afectam os dados sensíveis. Isto é feito via o código de acesso situado neste separador.
  • Código acesso ([field] ACS)

Este código de acesso permite restringir o acesso à ficha corrente a certos utilizadores.
Se a zona está registada, apenas os utilizadores dispondo deste código de acesso no seu perfil podem visualizar e modificar esta ficha.

O direito de execução controla o facto em que se possa executar o pedido em questão (em refrescando o pedido ou em modificando os parâmetros e em a re-executando).
  • Nr.de linhas ([field] NBRLIG)

Permite definir um número de linhas por página lógica no ecrã de consulta gerada pelo requerente. Se este número de linhas excede o número de linhas físicas afixáveis, um ascensor aaprece à direita do quadro. Botões situados ao alto do ecrá permitem navegar entre páginas lógicas (visualizar a seguinte, precedente, ultima,primeira).
  • Máximo linhas ([field] MAXLIG)

O número de linhas máximo corresponde a um parâmetro do pedido (maxrows) que pára a pesquisa quando um número de linhas satisfazendo às condições foi encontrada na base (estas linhas serão em seguida ordenadas, mas a base não garante que n linhas encontradas são as N primeiras na ordem de ordenação). Trata-se de uma optimização permitindo limitar a carga para a base de dados.
  • Grupo ([field] GRP)

Permite dispor de um critério de classificação dos pedidos.
  • Nr.Colunas fixas ([field] NBRCOL)

Permite congelar as x primeiras colunas de um quadro em caso de paginação horizontal.
  • Tempos máx. ([field] MAXTIM)

Permite chegar a pesquisa ao termo deste tempo (exprimido em segundos). Isto permite evitar de esperar muito tempo neste caso nomeadamente de parametrização de um novo pedido.

Nota : Contrariaemmtne ao nº máximo de linhas, esta informação não é transmitida à base de dados. O tempo máximo pode ser então ultrapassado se a duração do pedido é muito importante.

Querie SQL

  • [field] TEXTE

Permite definir o pedido ele mesmo sob uma forma SQL clássico compreensível pela base de dados.

Quaisquer observações ao sujeito deste pedido :

  • Todo o pedido Select compreendido complexo pode ser escrito aqui, prevê que reenvia um número de campo inferior ao máximo admissível
  • falta se lembrar que o campo NOMCHAMP(índice), gera na tabela NOMTABLE, é definido, na base de dados, como o campo NOMCHAMP_indice na tabela NOMTABLE, índice valendo 0, 1, 2... N-1, N sendo a dimensão do campo

Assim, por exemplo, poder-se-á escrever :

Select LOGIN_0, CHEF_2 From AUTILIS
Where USR_0 like %1%
Order by CREDAT_0

  • Num pedido deste tipo, é possível de definir os parâmetros pela sintaxe %N%, que se refere ao parâmetro número N definido no quadro dos parâmetros (%1% no exemplo acima). Os parâmetros serão registados pelo meio do botão No momento da execução do pedido.

[Grid] Colunas

  • Descr. ([field] COLTIT)
  • Tipo ([field] COLTYP)

Este Tipo de dados permite indicar como deve se fazer a apresentação do dado. No caso de um menu local, indicar-se-á o número do menu local utilizado. Se o tipo de dados utilizado está ligado a um objecto , será possível de passar por túnel em gestão do objecto no momento da consulta.
  • Menu ([field] COLNUM)

Permite definir um número de menu local a utilizar para apresentar o valor numérico da coluna corrrespondente reenviada pelo pedido.
  • Tipo de gráfico ([field] GRA)

Este campo não pode ser registado que se o modo de representação escolhido no cabeçalho não é Caracter. Permite indicar a maneira cuja representação gráfica vai ser colocada em obra.

Pode tomar os valores seguintes :

  • Nenhum ou Literal se o campo da linha é de tipo alfanumérico. O campo utilizando o valor Título vai ser utilizado como título do gráfico. Atenção, apenas uma linha de quadro pode ter este valor, se está já presente sobre uma linha, falta então modificar o valor da linha levando o valor Título para lhe re-dar o valor Nenhum antes de poder afectar este valor a uma outra linha do quadro.
  • Nenhum, Valor ou Defeito se o campo da linha é de tipo numérico. A regra é a seguinte :
    • Nenhum significa que o campo não será nunca apresentado no gráfico.
    • Se o gráfico define no último separador permite fazer aparecer várias séries, todas as séries correspondente a campos marcados Defeito serão visualizados (eleas o serõa toda se ennhuma não está assim em avanço).
    • Se o gráfico define no último separador permite fazer aparecer várias séries, todas as séries correspondente a campos marcados Defeito serão visualizados (elas o serão toda se ennhuma está assim em avanço).
  • Representação ([field] REP)

Este campo não pode ser qualificado manualmente.

  • se o modo de representação escolhida no cabeçalho não o está Caracter.
  • sobre os campos numéricos cujo se decidiu afixar o valor.

Permite gerar as combinações de gráficos se o tipo de gráfico definido nos parâmetros graficos permite fazer aparecer várias séries independentes de tipo diferente (está ignorado nos outros casos). Na prática, as combinações utilizáveis são as seguintes :

  • Tipo = multi gráfico :, Gráfico por defeito =Linhas, Representação =multi :
  • Tipo = multi gráfico :, Gráfico por defeito =Barras, Representação =multi :

A regra é a então a seguinte :

  • Defeito significa que a apresentação gráfica da série corresponde ao gráfico por defeito do último separador (Barras, ou Linhas no caso que nos interessam).
  • Barraou linha permite forçar o modo de apresentação correspondente para série em questão.

[Grid] Parâmetros

  • Descr. ([field] PARTIT)

Se este campo está preenchido, um parâmetro cujo título corresponde a este campo poderá ser registada antes a execução do pedido (via o botão ).

Este parâmetro poderá ser utilizado no corpo do pedido pelo meio da sintaxe %N%, onde N é o número de linha correspondente no quadro dos parâmetros.
  • Tipo ([field] PARTYP)

Este tipo de dados permite facilitar o registo do parâmetro correspondente. Com efeito, se um tipo está definido :

  • os controlos de formato são realizados
  • Se o tipo de dados utilizado éstá ligado a um objeto, é possível de utilizar a janela de seleção correspondente.
  • Menu ([field] PARNUM)

Permite definir um número de menu local a utilizar para registar o valor do parâmetro.

  • Valor por defeito ([field] VALDEB)

[Tab] Gráfico

Este separador não pode ser preenchido que se uma apresentação gráfica é possível para o pedido. Define-se então a maneira cujos dados são apresentados sobre o gráfico.

Gráfico

  • Tipo ([field] TYPGRA)

Este campo é útil desde logo que temos mais de um valor numérico no quadro susceptivel de estar apresentado sob forma gráfica. Pode tomar dois valores :

  • Gráfico simples : neste caso, não se vê que uma coluna de valores à vez (um seletor permite escolher a qual)
  • Multi-gráfico : neste caso, vê-se todas as colunas afixáveis, combinadas segundo o parãmetro definido pelo parâmetro Representação.
  • Representação ([field] FSHGRA)

Quando se apresenta várias séries no gráfico (sob reserva que a apresentação de todas as séries seja idêntico e não seja de tipo sector), é possível de indicar como se combina as séries de valores na afixação, em registando o valor seguinte :

  • Multi permite apresentar os valores de cada série lado a lado (uma cor por série).
  • Acumulativo permite apresentar os valores em acumulado vertical.
  • Comparativo permite de os apresrntar em comparação relativa (a altura do gráfico está constante).
  • Gráfico p/defeito ([field] DEFGRA)

Define a representação gráfica utilizada para os dados a apresentar (por entre as escolhas Barras, Linhas, Áreas, Sectores). Uma imagem afixada em função da escolha do facto ilustra o tipo de apresentação obtido.

Se não se apresenta que só uma série de valores, a representação gráfica por defeito está afixado por este parâmetro.

Se se apresenta várias séries de valores, é possível de misturar os modos de apresentação (uma série sob forma de barras, uma outra sob a forma de linha, por exemplo). Neste caso, cada série de valores é representada em função do campo Representação situada no quadro dos campos :

  • Se vale Barra ou Linha, este modo de representação é utilizado para a série de valores afectados se é compatível com a representação gráfica por defeito (não é possível de misturar os sectores com barras ou das linhas).
  • Se vale Defeito, utiliza-se o modo de representação gráfica por defeito para a série.

Na utilização, se a representação das séries é homogénea (portanto se não existe que uma série no gráfico), dispõe-se de um selector de gráficos para passar num outro modo rede de apresentação.

Observação importante :

De referir que nenhuma filtragem ligada às habilitações foi realizada (o que seja as habilitações por papel, por estabelecimento, ou por código de acesso) com requerente SQL. É então aconselhado de proteger os pedidos deste tipo por um código acesso para limitar os direitos de visualização e ainda mais de execução destes pedidos. Assim, o acesso a esta função deve normalmemnte ser limitado a uma lista restringe os utilizadores.

Botões específicos

Validar

gera o ecrã de consulta associado, e verifica a exatitude do pedido em o submetido à base de dados para verificação sintaxica e semantica.
Executar

Permite o lançamento do pedido corrente. A execução se desencadeia em atualizando uma tabela temporária, e cadeia sobre a função de visualização dos pedidos, que permite visulizar o resultado do pedido.

Mensagens de erro

  • [Generic messages]

Pedido incorrecto

[ descrição de erro reenviado pela base de dados]

O pedido não está correcto, do ponto de vista da base de dados. Seja não está correcto do ponto de vista sintaxico, seja um elemento não existe. A descrição complementar dá o título de erro reenviado pela base de dados.

Acesso impossível a esta ficha

Tenta-se aceder a um pedido do requerente clássico com o requerente SQL (os códigos de um pedido de requerente SQL não podem ser utilizados para nomear um pedido clássico, e vice versa).

Excesso de zonas

O requerente lançado reenvia um grande número de colunas. De referir que o número de colunas susceptível de serem lidos pelo requerente SQL está limitado a 120.
