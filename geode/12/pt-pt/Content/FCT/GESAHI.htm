X3 Warehousing

Arquivo / Expurgo

Esta função permite definir as modalidades de histórico e de expurgo de uma tabela e eventualmente das tabelas detalhe associadas, assim que todas as tabelas que lhe estão ligados no dicionário e cujo código anulação está posicionado a supressão. Tomemos por exemplo, a fórmula de histórico SOH da tabela das tabelas das encomendas cliente SORDER, as tabelas SORDERP, SORDERQ etc históricos igualmente

Estas fórmulas de histórico são utilizadas no momento da função de histórico / expurgo.

É um tratamento supervisor, que a partir das fórmulas de histórico e da parametrrização, vai efetuar este histórico e expurgo É no entanto possivel de acrescentar as particularidades num tratamento cujo nome ser´identificado na fórmula de expurgo.

 O detalhe das modalidades de histórico/expurgo das tabelas de certos módulos está assim dada num documento anexo.

Observação importante :

É importante notar que o histórico que está feito aqui nada a ver com qualquer arquivo, nomeadamente fiscal. Trata-se com efeito de um histórico técnico destinado a responder a problemas de performances ligadas a volumetrias importantes sobre as tabelas ativas.

Assim, por exemplo, o histórico das escritas não se faz forçosaemnte para um exercicio completo, porque em caso de conciliação em cima sobre dois exercícios, o documento de exercício anterior não está histotizado, mesmo se pediu a historizar este exercício.

Esta historização não desalfandega então em NADA de realizar o arquivo fiscal a cada paragem. Afim de responder às exigências da administração fiscal , nós preconizamos de realizar, a cada paragem, a edição em pdf das listas legais (Jornais, Razão, Balancetes) e de exportar sob excel (ou outro formato neutro) as escritas contabilísticas do período, assim que o conjunto das parametrizações / dados tendo concorrido à sua constituição.

Se se utiliza por outro lado o conector GED, asssociado a um software de gestão eletrónica de documento, pode-se então por utro lado assegurar uma armazenagem seguro e intangivel dos dados desde a sua extração

Gestão do ecrã

[Tab] Ecrã de registo

O registo das fórmulas de expurgo se faz sobre um separador.

[Block number] 1

  • Cód. ([field] COD)

O código identifica a fórmula de expurgo.

O código de expurgo AUDITASD (tabela Audit SData) permite expurgar os registos da tabela de auditoria tendo sido tratados pelo tratamento de sincronização utilizada para a interface CRM. Os registos expurgados são aqueles cujo número de sequência está estritamente inferior ao número da última atualização da tabela de sincronização.

 A tabela de de sincronização AJSSYNC não está expurgada nem colocada a zero. Para suprimir os registos desta tabela, ela deve estar colocada a zero.
  • Descr. ([field] ZDES)

Destinado nomeadamente a figurar sobre as listas e os ecrãs nos quais o código da ficha pode ser registado ou selecionado, este texto permite de dar uma descrição em claro da ficha afetada.

Um intitulado curto pode também ser utilizado neste uso quando não há suficiente lugar.

Características

  • Desc. curta ([field] ZDESSHO)

Destinado nomeadamente a figurar sobre as listas e os ecrãs nos quais o código da ficha pode ser registado ou selecionado, este texto permite de dar uma descrição em claro da ficha afetada.

Um intitulado curto pode também ser utilizado neste uso quando não há suficiente lugar.
  • Cód. activ. ([field] CODACT)

Um código actividade permite :

  • de tornar opcional um elemento do dicionário se o valor associado ao código actividade é nulo.
  • de assinar os elementos específicos desde logo são marcados por um código começando por X,Y ou Z.
  • de dimensionar um número de linhas máximas quando o código de actividade maerca os elementos de um quadro.

Assim, se o código actividade está não activo, o elemento marcado não será utilizável, e o código associado (se existe um) não será gerado nem activável.
  • Módulo ([field] MODULE)

Módulo de pertença da parametrização. Este campo permite registar se o ecrã deve estar criado na base de dados do dossier. É o módulo ao qual ecrã está anexada está ativa para o dossier.

Tratamentos

  • Activo ([field] ENAFLG)

Esta check box a assinalar permite ativar ou de desativar a ficha corrente sem portanto perder o seu conteúdo.

Uma ficha desativada não pode ser utilizada (por chamada do seu código) em outras fichas (documentos, parametrizações...), ou no tratamento de massa.

As habilitações sobre uma função dada podem interdir a criação de uma ficha ativa. Neste caso, a check box está desativada por defeito, e está modificável unicamente por um utilizador autorizado, ou via um circuito de assinatura definido por Workflow.
  • Tratamento standard ([field] CTLTRT)

Tratamentos facultativos podendo compreender diversos sub-programas que serão chamados no mometo de execução de historização. Ver as características destes sub-programas. Os desenvolvimentos verticais deverão començar por X; os desenvolvimentos específicos deverão começar por Y ou Z. A atualização do tratamento específica não necessita de proteção por código de atividade.  
  • Tratamento específico ([field] SPETRT)

[Grid] Tabelas

  • Tabela ([field] TBL)

Tabelas a historizar e expurgar.

  • Tabela ligada ([field] LNKTBL)

É possível de preencher uma tabela principal assim que as tabelas de detalhe. Por consequente,para uma tabela detalhe, indicar-se-á, neste campo, a tabela principal na qual ela está associada. Para uma tabela detalhe, nenhum outro parâmetro não será registável. Uma tabela pode ser considerada como tabela de detalhe, se a chave primária está constituida pela chave primária da tabela principal + um identificador Pode-se registar até 20 tabelas por tabela principal

  • Ligação ([field] GRUTBL)

Sobre o produto Warehousing, é uma possibilidade suplementar às tabelas ligadas para constituir um grupo de historização coerente. Para uma tabela, com tabela de ligação, pode exprimir uma condição de expurgo ou de historização.
  • Acção ([field] PROLBL)

Sim: Sobre Warehousing, gera a chamada às ações "POINTE", "EPURE" e "HISTO". Estas ações são a escrever no tratamento standard ou específicos associados. Nestas ações, a variável [L]TABLE está preenchida. Se afeta OK=0, o grupo não está historizado/expurgado. Ver metodologia.
  • Socied. ([field] CPYFLD)

Os campos "Sociedade", "Estabelecimento" e "Data" permitindo de identificar os campos da tabela principal contendo respetivamente, a sociedade, o estabelecimento, a data. São os critérios standards para filtrar os registos da tabela proncipal a historizar.

  • Estab. ([field] FCYFLD)
  • Data ([field] DATFLD)
  • Formula p/histórico ([field] FRM)

Este campo permite definir filtros suplementares por uma fórmula para o histórico dos registos. O expurgo, utiliza esta fórmula, quando o histórico não foi activado por parametrização.
  • Coeficiente ([field] COE)

É um coeficiente multiplicador entre o tamanho da tabela do dossier de exploração e o tamanho da mesma tabela no dossier histórico. Serve em criação na tabela no dossier histórico, para indicar um dimensionamento requerido no "tablespace".

Utilizado no produto Warehousing unicamente. Para os outros produtos, o parâmetro HISCOE contém um coeficiente único para todas as tabelas históricas.

É importante notar que os tratamentos standards são entregues quando a condição de histórico ou de expurgo não é suficiente, e que estes tratamentos standards não devem ser alterados A sua modificação inconsiderada pode provocar os históricos e expurgos colocando em perigo a perenidade dos dados da base. É recomendado, se deseja modificar as regras de histórico / expurgo, de completar a fórmula de histórico em definindo por exemplo critérios mais restritivos e de escrever se necesário o apagamento dos dados complementares criados nas tabelas específicas no tratamento específico. Opções / Parâmetros expurgos

Documentação / Parágrafos

Esta função permite aceder à gestão da documentação, sobre o primeiro parágrafo da documentação (se ela existe) associada à ficha corrente.

Documentação / Ligações

Esta função permite aceder à gestão das ligações. Ela permite definir as relações entre a ficha corrente e de outras fichas (por exemplo das ligações entre funções e parâmetros). Estas ligações, puramente documentários, permitindo alimentar o mecanismo de geração dos esqueletos de documentação.

Documentação / Geração

Este menu permite lançar uma geração de documentação. A geração pode-se lançar igualmente a partir do botão [Génération] em baixo da janela.

Três tipo de geração podem ser lançadas, separadamte ou simultaneamente :

  • a geração do esqueleto de documentação a partir do dicionário (tabelas ADOCUMENT, ADOCBLB, ADOCCLB).
  • a geração da documentação a partir das tabels precedentes.
  • a geração da documentação sobre campo.

Os limites propostos por defeito têm em conta da ficha em curso, mas elas podem ser modificados ao lançamento.

Selecções

  • Todas as línguas existentes ([field] ALLLAN)

Se esta check box está assinalada, todas as línguas são tomadas em conta pela operação.

Se a check box não foi assinalada, tomar-se-á segundo o caso :

  • A língua por defeito do utilizador, se nenhum código língua não foi registado.
  • A língua registada por outro lado senão.
  • Língua ([field] LAN)

Permite de definir a língua de geração (quando a check box permitindo de gerar em todas as línguas não foi assinalada).
  • Todos os tipos ([field] ALLTYP)

Se esta check box está assinalada, todos os tipos de documentação são tomados em conta para a geração. Deve-se referir que, neste caso, limita-se a geração às documentações standard (apenas as documentações de código inferior a X são tomados em conta).
  • Tipo ([field] TYPEDOC)

Permite definir o tipo de documentação a tomar em conta no tratamento (quando não se assinalou a check box para os tomar todos em conta).
  • Desde códigos documentos ([field] FUNDEB)

Permite de dar um limite sobre os códigos a ter em conta para a geração de documentação.
  • Cód. documentação fim ([field] FUNFIN)

Geração ADOCUMENT

  • [field] GENDOC

Este botão redondo permite criar/atualizar a parametrização da documentação (hierarquia dos parágrafos) a partir dos elementos técnicos encontrados no dicionário dos dados. Pode-se escolher de retomar unicamente as documentações existentes ou criar novas documentações se necessário

As relações de documentação são igualmente tomados em conta. Os parágrafos susceptíveis de serem criados são definidos no anexo técnico correspondante.
  • Para todos os grupos ([field] ALLCREGRP)

Geração documento final

  • [field] GENFIN

Este botão redondo permite gerar a documentação final a partir do conteúdo dos parágrafos de documentação. Esta documentação foi criada em diferentes sub-directórios do directório definido pelo parâmetro utilizador DIRDOC.

A geração "Completa" trata o conjunto das documentações não tendo código de atividade mas também aquelas tendo os códigos atividades standards (código atividade < "X")

Falta utilizar a geração "segundo o código atividade" para obter a geração das documentações das funções tendo os códigos atividades ativos (quer sejam standards ou específicos).
  • Somente as documentações validadas ([field] HLPVLD)

Permite gerar o documento final unicamente para as documentações válidas.
  • Ajudas sobre campos ligados às documentações ([field] HLPLNKFLD)

Se esta check box está assinalada, trata-se especificamente as ajudas sobre campos relativos às documentações sobre função ou objecto tratados por outro lado. Para saber quais são as ajudas sobre campo a tratar, explora-se os ecrãs associados aos parágrafos de documentação correspondente.
  • Outras ajudas sobre campos ([field] HLPFLD)

Se esta check box assinalada, cria-se a documentação final a partir do conteúdo dos parágrafos de documentação. Esta documentação foi criada em diferentes sub-directórios FLD do directório definido pelo parâmetro utilizador DIRDOC.
  • Códigos ajuda campos inicio ([field] HLPDEB)

Permite de dar um limite sobre os códigos a ter em conta para a geração de documentação.
  • Cód. ajuda campos fim ([field] HLPFIN)
  • Directorio de geração ([field] REPERT)

[In addition to the generic error messages, the following messages can appear during the entry] :

Tabela não definida

A tabela registada no campo "tabela ligada" deve ter registado em tanto que a tabela a tratar no campo "tabela".

XXXXXX : Excesso de ligações sobre esta tabela

Tenta-se de registar mais de 20 tabelas detalhe para uma tabela principal

XXXXXX : Campo inexistente sobre a tabela YYYYYYY

Ligado a tabela YYYYYYY a uma tabela principal. Pelo menos um campo da chave primária da tabela principal não está encontrada nesta tabela detalhe YYYYYYY.

XXXXXX : Tipo de dados incorrecto

Ligado a tabela YYYYYYY a uma tabela principal. O campo XXXXX da chave primária da tabela principal está bem presente na tabela detalhe mas de um tipo de dados diferentes

Campo inexistente sobre a tabela XXXXXX 

Os campos de identificão de zona (sociedade, estabelecimento, data) devem referenciar uma zona que existe na tabela principal.

Tipo de dados incorrecto

O campo Zona sociedade deve referenciar uma zona de tipo CPY ou FCY.
O campo Zona estabelecimento deve referenciar uma zona de tipo CPY ou FCY.
O campo Zona data deve referenciar uma zona de tipo data.
