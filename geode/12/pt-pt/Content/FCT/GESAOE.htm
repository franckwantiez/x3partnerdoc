Modelos import/export

Esta função permite definir os formatos de ficheiros utilizados pelas funções de importação e de exportação de objeto afim de integrar os dados na base do software, ou ao contrário de os extrair.

Um ficheiro utilizável pelo módulo importação e gerado pelo módulo export está na base um ficheiro com uma estrutura de um dos tipos seguintes:

  • ficheiro liso com campos em comprimento fixo.
  • ficheiro liso com separador entre os campos (e registos).
  • ficheiro XML.

O módulo de importação / exportação utiliza a noção de objeto para permitir a atualização de dados na base. Um objeto sendo definido para um conjunto de tabelas e de ecrãs, assim que as ações standard associadas seja aos campos e à atualização, garante-se assim que o conjunto dos controlos e das ações necessárias no momento da atualização da base são efetuadas, poisque uma única descrição do objeto permite à vez gerar o código relativo a uma atualização em conversacional e a atualização por importação.

Quando um objeto não atualiza que uma só tabela, o modelo de importação descrito na lista dos campos da tabela a integrar, sabendo que um bloco de dados do ficheiro a importar ou exportar contém os dados de um registo. Quando várias tabelas são atualizadas por um objeto (por exemplo o cabeçalho e as linhas) encontrar-se-ão vários blocos de dados para uma instância de objeto a integrar (um bloco definindo o cabeçalho e N blocos definindo cada uma linha).

É de notar que é igualmente possível de importar uma tabela sem associar a uma noção de objeto. Deixa-se para tal o campo objeto vazio, e se contenta de dar no quadro do segundo separador tabela e os campos afetados. A importação se faz então sem nenhum outro controlo que aqueles induzidos para os formatos associados aos tipos de dados dos campos da tabela.

Cabeçalho

Um modelo de importação se identifica por um código alfanumérico Outro um intitulado, define-se, sobre dois separadores, as características técnicas do modelo.

  • Modelo ([field] EXT)

Código que identifica o modelo de import-export.
  • Descr. ([field] INTIT)

Permite de definir um titulo associado a cada ficha.
  • Activo ([field] ENAFLG)

Esta check box a assinalar permite ativar ou de desativar a ficha corrente sem portanto perder o seu conteúdo.

Uma ficha desativada não pode ser utilizada (por chamada do seu código) em outras fichas (documentos, parametrizações...), ou no tratamento de massa.

As habilitações sobre uma função dada podem interdir a criação de uma ficha ativa. Neste caso, a check box está desativada por defeito, e está modificável unicamente por um utilizador autorizado, ou via um circuito de assinatura definido por Workflow.
  • Bloquio de pré parameterização ([field] AFACTORYOW)

[Tab] Cabeçalho

Este separador define as características gerais do modelo, quer dizer :

  • os dados que devem estar exportados
  • a estrutura geral do ficheiro (formato, codificação, definição de grupos de dados)
  • dois parâmetros complementares

Geral

  • Assunto ([field] OBJ)

Define o código de objeto a importar ou exportar. No caso de exportação, esta zona está facultativa; o nome da tabela principal a exportar está indicada no bloco dos identificadores.
  • Função ([field] FONCTION)

Permite inicializar o contexto (nomeadamente se um mesmo objeto está utilizado por várias funções) e de verificar os direitos de acesso. Com efeito, os utilizadores devem ter os direitos de acesso idóneos à função para poder se servir do modelo.

Este campo é obrigatório.
  • Módulo ([field] MODULE)

Módulo de pertença da parametrização. Este campo permite registar se o ecrã deve estar criado na base de dados do dossier. É o módulo ao qual ecrã está anexada está ativa para o dossier.
  • Cód. activ. ([field] CODACT)

Um código actividade permite :

  • de tornar opcional um elemento do dicionário se o valor associado ao código actividade é nulo.
  • de assinar os elementos específicos desde logo são marcados por um código começando por X,Y ou Z.
  • de dimensionar um número de linhas máximas quando o código de actividade maerca os elementos de um quadro.

Assim, se o código actividade está não activo, o elemento marcado não será utilizável, e o código associado (se existe um) não será gerado nem activável.
  • Código acesso ([field] ACS)

Este código de acesso permite restringir o acesso à ficha corrente a certos utilizadores.
Se a zona está registada, apenas os utilizadores dispondo deste código de acesso no seu perfil podem visualizar e modificar esta ficha.

O direito de execução anexada a um código utilizador está tratado de maneira particular nos casos dos modelos de import-export : se um utilizador não tem o direito de execução, não poderá utilizar os modelos para importar ou exportar os dados.
  • Tratamento standard ([field] TRTIMP)

Define o tratamento standard na qual se encontram as etiquetas das ações que são chamadas nos tratamentos de importação-exportação.

Estes tratamentos permitem fazer as inicializações, controlos complemetares, e atualizações se necessário. Encontrar-se-á em anexo técnico a estrutura de um tal programa. De referir que os tratamentos standards nomeadas em geral IMPXXX, XXX sendo o código de importação, são fornecidas para um certo número de importações.

Para mais de informação sobre estas ações, a documentação anexa correspondente.

  • Tratamento especifico ([field] SPEIMP)

Define o tratamento específico, que é chamado antes o tratamento standard, e que permite realizar as mesmas ações em desativando se necessário o que este facto o tratamento standard.

As ações possíveis são nomeadamente das inicializações, controlos complementares, e das atualizações se necessário.

Para mais de informação sobre estas ações, a documentação anexa correspondente.

Estrutura

  • Tipo de ficheiro ([field] TYPFIL)

Define a estrutura utilizada para gerir os dados no ficheiro a importar ou exportar. Para mais informações, o parágrafo correspondente.

  • Separador de campo ([field] SEPFLD)

Define o separador entre dois campos.

Para registar um caracter não imprimível, falta entrar uma !\" (barra de fração inversa) seguido de 3 algarismos representando o código ascii do caracter em base decimal.
  • Separador de registo ([field] SEPREC)

Define o separador entre dois registos (grupos de dados).

Para registar um caracter não imprimível, falta entrar uma !\" (barra de fração inversa) seguido de 3 algarismos representando o código ascii do caracter em base decimal.

Frequente, utiliza-se :

  • o caracter line feed (\010), que corresponde ao fim da linha nos ficheiros textos sob Unix
  • a combinação dos dois caracteres quebra de página, nova linha (\013\010)n que corresponde ao fim da linha dos ficheiros textos Windows.
  • Delimitador de campo ([field] FLDLIM)

O delimitador de campo (geralmente o caracter " ) está acrescentado em última posição dos campos de tipo alfanumérico. Os campos numéricos e datas não são afetados por este delimitador.

Trata-se frequentemente de um dos caracteres seguintes :

  • 'simples quota'
  • "duplas aspas"
  • Formato fich. ([field] CODDBA)

Define o formato dos caracteres utilizados no ficheiro :

  • ascii é o formato clássico onde um caracter equivale a um byte num ficheiro. Este tipo de formato permite tratar os caracteres ocidentais clássicos, com diferentes jogos de caracteres possíveis, definidos no campo correspondente.
  • utf8 corresponde a um formato UNICODE onde o número de caracteres são as variáveis (vai de 1 a 4, 1 correspondente ao jogo de caracter latino não acentuados Este formato permite gerir todos os tipos de caracteres, e nomeadamente os caracteres chineses.
  • ucs2 corresponde a um formato standard Microsoft, no qual os caracteres são sistemeticamente guardados sobre 2 bytes

Export.

  • Export. ([field] EXPORT)

Se esta zona está assinalada, será possível de utilizar este modelo em Exportação de dados.
  • Crono. export. ([field] CHRNUM)

Este campo, unicamente afixado, guarda o valor do número de tempo quando a última exportação teve lugar. Isto permite, quando se realiza exportações cronológicas, de não tratar que o que aqui foi modificado depois que a última exportação teve lugar.

Transcodific.

  • Jogo de caracteres ([field] OPTCHA)

Quando se utiliza o jogo de caracteres ascii, é possível de utilizar diferentes formatos estandardizados :

  • O código ISO 8859 (que é por outro lado o jogo interno de Sage quando se trabalhe em ascii).
  • O código IBM PC
  • O código ASCII 7 bits (sem acentos : as letras acentuadas são convertidas na minúscula correspondente).
  • Separador décimal ([field] SEPDEC)

Define o separador decimal utilizado pelas cifras. Se esta zona está vazia, vai-se considerar que o separador decimal é o ".". (ponto).
  • Formato das datas ([field] OPTDAT)

Define a maneira cujo são codificados os campos de tipo data (ordem e número de caracteres para o ano).

Apenas a ordem dos números e o número de caracteres de ano pode assim estar especificado. A importação de eventuais caracteres de separação entre os campos são filtrados; assim as datas sob a forma 29-05-59, ou 09/04/1991 são descodificadas corretamente

O sub-programa de descodificação tem em conta da variável adxdcs do motor, preenchido pelo parâmetro DCS se encontram nos parâmetros gerais para definir a maneira cujo está descodificado um ano sobre dois números. DCS represente o ano "pivot" que define a alteração do século.

Por exemplo, se DCS vale 1940, todo o ano sobre dois números inferiores ou igual a 40 será considerada como fazendo parte de 21º século, e todos os anos superiores será considerado como fazendo parte do século 20. Será assim capaz de exprimir os anos entre 1949 e 2039 sobre dois números.

O formato AAAA-MM-DD está reservado para as exportações de tipo XML.
  • [field] LIBDAT

Título associado ao código precedente

  • Formato menus locais ([field] OPTMNL)

Os campos de tipo "menu local" são guardados sob a forma de um número representando a sua fila na tabela.

Segundo o valor desta zona o modelo vai exportar (onde se espera a encontrar em importação) :

  • 0 : neste caso, escreve-se a escolha sob a forma de um número indicando a fila do menu na tabela : 1 para a primeira escolha, 2 para o segundo,etc. Isto corresponde por outro lado ao formato interno sob o qual o menu local está guardado na base.
  • 1 : escreve então a escolha em utilizando o código (sobre um caracter) associado a cada escolha de menu local. Este código não está visível em gestão dos menus locais. Pode ser definido nas funções de desenvolvimento, em gestão das mensagens : temos a possibilidade de registar este código interno (que não serve que a este fim, o que não é mais, como nas antigas versões caracteres, um acelerador de registo).
  • n : (n>1) os n primeiros caracteres do literal afixado em registo. Quando se utiliza esta opção, o algoritmo de reconhecimento pesquisa sobre o primeiro caracter, depois o segundo, e assim de seguida, até que o seu título corresponda. Assim, se pesquisa a reconhecer CHQ num menu local cujos intitulados são Espécies, Transferência, Cheque, Letra, Carta Azul, o algoritmo vai reconhecer o cheque (apenas o título cujas duas primeiras letras correspondem).

Cômputo do facto que os títulos do menu local não são os literais utilizadas em afixação, o valor guardado na base sendo a fila na tabela, é perfeitamente possível de alterar o intitulado dos menus locais em tempo de uma importação, para que o algoritmo de reconhecimento funciona corretamente Atenção todavia, o facto de alterar os títulos dos menus locais não pode ser feito que em modo mono-utilizador, também isto não está envisajável nas transferências regulares ou automatizados.

Título associado ao código precedente

Importação

  • Importação ([field] IMPORT)

Se esta zona está assinalada, será possível de utilizar este modelo em Importação de dados.
  • Actualiz. autorizada ([field] OPTUPD)

Permite autorizar a modificação de um registo existente durante a importação

  • Sas ([field] AOWSTA)

Quando esta check box está assinalada, a importação de dados alimenta a peneira de importação-exportação com os dados errados. O facto de alimentar a peneira não impede por outro de criar um ficheiro de erro.
  • Import. especial ([field] OPTSPE)

Assinala que a integração dos dados na base se faz em utilizando as ações específicas definidas no tratamento cujo nome está dado na rubrica Tratamento importação. Este tratamento específico possui um número restrito de pontos de entrada e necessita então de escrever um tratamento incluindo todos os controlos que se deseja fazer.

O seu interesse reside no facto que pode assim reagrupar os controlos afim de otimizar o programa de importação. A estrutura dos tratametos de importações específica está descrita em anexo. Deve nomeadamente encontrar as ações seguintes :

  • uma etiqueta $RAZCRE que está diretamente chamada pela importação.
  • uma etiqueta $SAIMSK que está diretamente chamada pela importação a cada registo lido e substitui o Call SAIMSK standard (afetação e controlo das zonas da máscara a partir da classe [F])
  • Criação autorizada ([field] OPTCRE)

Quando esta check box está assinalada, o modelo de importação-exportação pode ser utilizado para criar de novo registos na base.
  • Workflow ([field] ENAWRK)

Se esta check box não está assinalada, os eventos de workflow relativos às operações elementares (criação, modificação em gestão de objecto) não são mais chamados. Assim, se se lança as importações terminando a actualizações em massa, evita-se o desencadeamento de uma multitude de eventos deste tipo, o que pode ser prejudicial às performances de importação e provocar o envio de mensagens em massa.

Portanto, os eventos de workflow relativos ao desencadeamentp da importação não estão desactivados.

  • Ligação ([field] FLGLNK)

Define a expressão de ligação, quer dizer uma sequência de valores, separados por ponto-e-vírgula, que dá os valores de chave que ligam a tabela de detalhe ao registo de cabeçalho.

No caso de um tipo de ficheiro de comprimento fixo, falta indicar o número de caracteres de cada registo.

[Tab] Campos

Neste quadro, definem-se os diferentes campos a importar, organizar em grupos identificados pela coluna Código, na qual se encontram um dos códigos definidos no quadro dos indicadores do primeiro separador (o campo pode restar vazio se nenhuma tabela foi definida).

Este separador contém o quadro definindo a estrutura detalhada a grupos existentes no primerio separador. É de notar :

  • Que não é obrigatório de definir os campos em todos os grupos (com efeito, alguns podem não ser que grupos "técnicos" definindo as relações). Imaginemos por exemplo que se pretende exportar os campos de cabeçalho de encomenda e os campos emitidos da ficha do cliente pagador, e este num grupo de dados (não se deseja ter separador de grupo de dados). Será apropriado neste caso leva a definir dois grupos (o primeiro definindo o percurso sobre as encomendas, o segundo associado ao cliente com a ligação apropriada). Colocar-se-á, por outro lado, no separador campo, que as linhas associadas ao segundo grupo. Estas linhas poderão incluir à vez as informações extraidas do cliente de cabeçalho da encomenda (poisque os dois estão em linha).
  • Que é imperativo de definir para cada grupo a posição onde se encontra o separador de grupo desde logo que se trata de um modelo utilizável em importação e que tem vários grupos. É o pseudo campo / (barra) que permite definir este.
  • Que os blocos devem ser arrumados de maneira sequencial : quando um bloco de nível inferior existe, deve seguir o bloco ao qual está ligado.

[Grid] Campos

  • Nr. ([field] NUMLIG)

Linha corrente do quadro
  • Indicador ([field] TYP)

Esta zona, não está registada que se o quadro dos identificadores de grupos de separador precedente não está vazio. Ela permite anexar a informação a exportar ou importar a um grupo de dados.
  • Tabela ([field] FIL)

Define a tabela da base de dados onde está definido o dado a importar ou a exportar. Deve-se referir que :

  • Este campo está obrigatório mesmo se não serve forçosamente (se se está num modelo de exportação e que se define uma expressão calculada, esta pode referenciar os campos emitidos de várias tabelas).
  • A tabela em questão não é forçosamente a tabela principal associada ao grupo, mas pode ser uma tabela ligada a tabela principal deste grupo ou de um dos grupos precedentes de nível superior. Se nenhum local foi encontrado, uma mensagem de advertência está afixado : Ligação automática não gerada. Isso significa que é da competência do tratamento de import ou de export (registo no primeiro separador) de realizar esta ligação manualmente (em declarando por exemplo a tabela na ação IMP_OUVRE, e a leitura ou da tabela na ação IMP_LIENS).
  • Campo ([field] FLD)

Permite indicar o nome da zona da tabela a importar ou exportar. Diferentes sintaxes são aqui possíveis para definir as informações a extrair ou a integrar :

  • O caracter /  (slash) significa que um indicador de grupo está escrito na exportação ou pesquisado no momento de importação. Num modelo permitindo a importação, quando vários grupos existem, este separador está obrigatório para cada grupo. O separador de grupo está suposto ser um campo normal (enquadrado por seperadores de campos e dos delimitadores se um modelo de tipo Delimitado está utilizado).
  • A sintaxe da mais simples é CHAMP(indice), quando um campo emitido da tabela declarada na coluna precedente está importada ou exportada. Uma janela de seleção permite de fazer aparecer os campos possíveis.
  • Se se regista aqui uma constante cadeia de caracteres enquadrados entre as duplas aspas ( " ), o campo será escrito tal que o ficheiro em exportação, e ignorado em importação.
  • Se não se regista nada, isso significa que haverá um campo vazio no ficheiro exportado (enquadrado por delemitadores de campo se o modelo é de tipo Delimitado). Em importação, isso significa que o campo correspondente deve estar ignorado.
  • A sintaxe *N, onde N é um número compreendido entre 1 e 99, é igualmente possível. Isto permite afetar (em importação) ou de ler (en exportação) a variável GIMP(N). GIMP é uma variável global, de tipo cadeia de caracteres, de comprimento máximo igual a 100, que deverá ser afetado num tratamento específico associado à importação ou à exportação.
  • A última possibilidade, utilizável unicamente em exportação, consiste em definir uma expressão calculada qualquer, precedido do sinal igual ( = ). Esta expressão, que pode fazer chamada a constantes, funções, variéveis, operadores, e de campos emitidos de tabelas em linha, está verificada sintaxamente no registo (o controlo do contexto, em particular o facto que as variáveis existem, não podem claro ser verificado).
  • Descr. ([field] COM)

Indicar um comentário para facilitar a compreensão da parametrização.
  • Limite ([field] SEL)

Três escolhas são possíveis nesta zona :

  • Não registadasignifica que nenhum limite de ínicio-fim para este campo não será registado ao lançamento de exportação.
  • Registadasignifica que um limite de ínicio-fim para este campo poderá ser registado ao lançamento de exportação.
  • Não trasnferido significa que um limite de início-fim poderá ser registado para este campo ao lançamento de exportação, mas que o campo não será transferido no momento de uma importação ou de uma exportação.
  • Posição ([field] LOC)

Esta coluna não é útil que no caso de um formato em comprimento fixo; neste caso, a posição dá o deslocamento por relação ao início do bloco ou do registo (posições dadas em número de "bytes", 1 significando que se encontra no início do bloco ou do registo). As posições devem estar compatíveis com o tamanho de registo.

Esta zona determina o comprimento do campo sobre o ficheiro sequêncial.

  • Pattern ([field] PATTERN)

Este campo é utilizado quando se realiza as importações/exportações ao formato XML.

  • Baliza ([field] BAL)

Este campo é utilizado quando se realiza as importações/exportações ao formato XML.

Com efeito, quando um ficheiro XML foi criado, tem-se a necessidade de vantagem de informações, nomeadamente para poder criar um ficheiro XSD descrevendo a estrutura do ficheiro XML e assim controlar a sua validade com utilitários de verificação de sintaxe integrados aos diferentres softwares ETL.

Este campo define o código da baliza descrevendo o campo exportado no modelo, tal que ele aparece no ficheiro XML.
  • Obrigatório ([field] OBL)

Este campo é utilizado quando se realiza as importações/exportações ao formato XML.

Com efeito, quando um ficheiro XML foi criado, tem-se a necessidade de vantagem de informações, nomeadamente para poder criar um ficheiro XSD descrevendo a estrutura do ficheiro XML e assim controlar a sua validade com utilitários de verificação de sintaxe integrados aos diferentres softwares ETL.

Este campo definese o campo é obrigatório ou não. Se o valor deste campo é igual a Sim                 , o ficheiro XSL conterá uma especificação de tipo minOccurs="1"
  • Nr. da tabela ([field] NUMTAB)

Este número, se existe, faz referência a uma tabela de transcodificação utilizável para transcodificar o campo lido e o tornar conforme ao formato esperado.

Geração do ficheiro

  • [field] TYPEXP
  • Ficheiro de dados ([field] FILEXT)

Permite definir o caminho de um ficheiro de dados por defeito que será proposto no momento do lançamento de importação ou de exportação (e utilizado de maneira automática em caso de lançamento de um encadeamento de importação ou de exportação).  Se o caminho ficheiro é relativo,  o directório de base é suposto ser o directório de base de instalação do software.

O caminho pode ser integrar o caracter #. Se é o caso, haverá gestão dos números sequênciais :

  • Em importação, isso significa que se vai procurar todos os ficheiros cujo modelo corresponde ao caminho, com # representando 5 caracteres (integra-se estes ficheiros na ordem crescente dos números).
  • Em exportação, isso significa que se cria um ficheiro integrando o valor formatado do contador [C]EXPORT sobre 5 caracteres. Isto vai normalmente de par com o facto de assinalar a check box Gestão tempo no momento do lançamento da função de exportação.

Principios de base

Quando se parametriza um modelo de importação, falta ter no espírito de quaisquer principios :

  • A importação permite as ações de criação e de modificação.
  • Para determinar a ação leva sobre um registo importado, o sistema efetua um teste de existência de objeto a tratar Este teste se efetua em carregando a chave primária do objeto com as informações de registo importado.
  • Por consequente, se a chave principal não está parametrizada, todos os registos passaram em modo criação (e não funciona que se a chave de objeto a criar pode ser afetada automaticamente, por exemplo pelo meio de um contador).
  • Uma importação corresponde a uma simulação de registo de todos os ecrãs ligados ao objeto importado.
  • Trata todas as zonas de ecrã em efetuando os mesmos controlos que um registo interativo.
  • Não toma em conta as zonas não registáveis no ecrã. (excepto casos particulares e referenciais). Por consequente, as zonas parametrizadas num modelo corespondente a uma zona não registável não são importadas.
  • A ordem na qual são parametrizadas as zonas no interior de um registo não têm importância. A importação carrega todas as zonas do ficheiro depois a importação na ordem dos campos nos ecrãs.

Modelos standard fornecidos

Tanto a exportação de dados é sempre possivel que seja o objeto, tanto a importação não o está sempre. Os mecanismos automáticos de descodificação dos fluxos de dados e chamada dos controlos ligados ao objeto automatiza grandemente a importação, mas não é suficiente para permitir uma importação automática sobre os objetos complexos. Assim, todo o objeto não é importável.

No dossier de referência, um modelo de importação (modificável) está fornecido para cada objeto cuja importação é possível. Pode todavia haver específicos ligados a esta importação : isto está definido na ajuda em linha associada ao modelo de importações para as quais as particularidades existem (esta ajuda está acessível por Alt + F1 quando o modelo está carregado).

Encontrar-se-á a lista das ajudas correspondentes (classificadas por módulo) a partir da ligação seguinte.

Quadro dos identificadores

A escolha da estrutura dos ficheiros a importar ou exportar depende das possibilidades de extração ou da integração que se encontra no software com o qual se pretende dialogar.

Em todos os casos, os dados devem estar organizados em grupos lógicos de linhas, que podem ser de tipos diferentes (em cabeçalho, detalhe, sub-detalhe, por exemplo), ou de um tipo único. A organização destes grupos está definida no quadro dos identificadores situado sobre o primeiro separador do modelo.

Cada grupo está associado a uma das tabelas da base de dados (a primeira está forçosamente a tabela principal do objeto, as seguintes são definidas por ligações a partir dos precedentes). Quando se utiliza o modelo para realizar uma exportação de objeto, é possível definir as ligações com não importa qual tabela de base sobre a qual uma ligação teórica existe, afim de extrair os dados ligados. Por outro lado, para um modelo de importação, apenas a atualização por objeto são efetivamente utilizáveis : é por exemplo inútil de esperar importar simultâneamente a encomenda e o cliente, o objeto encomenda não foi previsto para tal.

É importante notar que este quadro pode estar vazio se uma estrutura de dados a importar ou exportar está baseado sobre o percurso da única tabela principal (neste caso, a coluna Código da página seguinte restará a vazia). Não é mais necessário de fazer vários grupos de dados se várias tabelas ligadas devem estar exportadas simultâneamente. Com efeito, se faz aparecer, num mesmo grupo de dados, zonas extraidas de diferentes tabelas, o tratamento de exportação vai tentar de resolver as ligações entre tabelas em utilizando a estrutura das ligações descrita no dicionário. Isto supõe bem evidentemente que se não teve que uma ligação possível da tabela principal do grupo versus a tabela descrita (senão, a primeira ligação encontrada vai ser utilizada, e se pode que não seja o bom).

Existe um caso particular onde é necessário de criar ao menos um grupo : se o modelo está definido em comprimento fixo (neste caso, é com efeito necessário que se defina qualquer parte um comprimento de registo, e isto se faz na tabela dos grupos). Se não se deseja que o indicador de grupo seja obrigatoriamente presente na lista dos campos, é suficiente de definir este grupo com um código vazio : Apenas um grupo poderá então ser definido, e a coluna Código não será mais registado no separador seguinte.

O quadro dos identificadores não está acessível em registo que se o objeto é de tipo simples. Se define os identificadores de grupo, associa-se a cada identificador um nível, uma tabela, e as condições de ligações que permitem religar as linhas entre elas.

O nível vale 1 para a tabela principal a importar ou exportar (que não está em registo no quadro, mas deduzindo do objeto associado ao modelo).

Para toda a tabela ligada à tabela precedente, o nível é igual ao nível da tabela precedente se uma ligação biunivoca existe entre as duas tabelas, e vale este nível mais se vários registos são ligados a um registo da tabela precedente. A ligação está caracterizada pela chave destino a ler, e para a expressão dos segmentos das chaves cujo valor define as linhas ligadas.
