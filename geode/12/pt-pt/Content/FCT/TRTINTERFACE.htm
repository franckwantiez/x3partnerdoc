As interfaces

Objecto e Domínio

Objecto

SAGE Warehousing fornece numerodos ficheiros de interface standards para comunicar com as aplicações externas. Este documento apresenta os diferentes modelos de interface pré-definidos no "package" standard de Warehousing e sua colocação em obra.

Estes modelos são inteiramente parametrizáveis, tanto que as informações trnasmitidas que sobre as convenções de trocas. Nos clientes são assim inteiramente autónomos sobre a mestragem das trocas com o mundo exterior.

Domínio

O vocábulo N3 (Nível 3) designará a informática central (gestão das compras, gestão comercial, GPAO ...).

O vocábulo N2 será utilizado por SAGE Warehousing.

Princípio de actualização do stock N3

Em entrada:

O stock N3 está actualizado pelo meio dos interfaces:

  • Cômputo de recepção e / ou arrumação

Em saída :

O stock N3 está actualizado em emissão pelo meio dos interfaces:

  • Cômputo de preparação e / ou expedição

Outros eventos

O stock N3 pode ser colocado em fase (automaticamente ou manualmente) com o stock N2 pelo meio das interfaces :

  • Imagem do stock
  • Cômputo de ajustamentos
  • Cômputo de assemblagem

Princípio de troca de dados

Chamamos interface, a um protocolo de troca de informações entre duas aplicações distintas.

Distinguem-se as interfaces em importação e os interfaces em exportação. Os primeiros integram os dados de N3 em SAGE Warehousing, as segundas remontam as informações depois SAGE Warehousing versus o N3.

Existem 3 modos de funcionamento para os interfaces em importação:

  • criação, Acréscimo de dados em SAGE Warehousing
  • Modificação : Modificação de dados já existentes
  • supresssão : Supressão de dados já existentes

Princípio

As trocas entre o N3 e o N2 se fazem sob a forma de ficheiros emparelhados segundo o princípio seguinte :

Importação Warehousing trata os ficheiros que terão sido previamente registados por N3 nos dossiers (ou directórios) definidos para cada fluxo de interface.

  • Etapa 1 : Integração do ficheiro numa tabela de tratamento : controlo de coerência técnica.
  • Etapa 2 : Integração nas tabelas vivas de Warehousing : controlo funcional. Durante estes duas etapas, os ficheiros são salvaguardados nos directórios parametrizáveis.

Exportação Warehousing coloca a disposição N3 os ficheiros nos dossiers (ou directórios) definidos para cada fluxo de interface; N3 tendo em carga a recuperação destes ficheiros.

  • Etapa 1 : A validação integral de uma entrada, para o cômputo de arrumação, em fim de preparação de uma OE, para o cômputo de preparação, na validação de uma expedição, pelo cômputo de expedição, ou ainda a validação dos movimentos de ajustamentos,  os cômputos são gerados automaticamente num directório definido (parametrização).
  • Etapa 2 : extracção da tabela e geração de um ficheiro à disposição do N3. No momento da etapa 2, os ficheiros emitidos são alvaguardados num directório a parametrizar.

Directório

Os directórios nos quais são armazenados os dados do interface são inteiramente parametrizáveis. Todavia, por defeito, são definidos da maneira seguinte :

No quadro dos ficheiros importados :

  • Directório de importação
    Directório onde devem ser arrancados os ficheiros para ser tratados por Warehousing.
    Um directório parametrizável ("IMP") está definido por defeito para o conjunto dos interfaces.
    Directórios dedicados podem ser associados a cada inteface.
  • Directório de tratamento das importações
    Directório onde são colocados temporáriamente os ficheiros de importações em curso de tratamento.
    Um único directório parametrizável ("IMPTRT" está definido pelo conjunto dos interfaces.
  • Directório de erros importação
    Directório dos ficheiros de importação tendo dado lugar a erros de estrutura.
    Um único directório parametrizável ("IMPERR") está definido para o conjunto dos interfaces.
  • Directório de destino importação
    Directório de salvaguarda os ficheiros de importação tendo sido tratado sem erros de estrutura.
    Um único directório parametrizável ("IMPBAK") está definido para o conjunto dos interfaces.

No quadro dos ficheiros exportados :

  • Directório de exportação
    Directório onde devem ser arrancados os ficheiros.
    Um directório parametrizável ("EXP") está definido por defeito para o conjunto dos interfaces.
    Directórios dedicados podem ser associados a cada inteface.
  • Directório de tratamento das exportações
    Directório onde são colocados temporáriamente os ficheiros de exportações em curso de constituição.
    Um único directório parametrizável ("EXPTRT") está definido pelo conjunto dos interfaces.
  • Directório de salvaguarda exportação
    Directório de salvaguarda dos ficheiros de exportação gerados. Um único directório parametrizável ("EXPBAK") está definido pelo conjunto dos interfaces.

Parametrização 

Parâmetros de activação dos interfaces em exportação.

Qualquer que seja o fluxo de interface, a exportação pode ser limitada a certos conjuntos (estabelecimento, recipiente).

  • Parametrização interface : activação interface e ficheiro; limitação estabelecimento, recipiente.
  • Parâmetro de activação ligado ao modelo de interface

Por outro lado, as interfaces standards relativas às EE e às OE são condicionadas por um parâmetro "Transmissão N3" do código movimento.
As condições de activação particulares podem ser associadas a certos fluxos; elas são então mencionadas para os fluxos em questão.

Diferença entre ITASLEEP e ITAWAIT

No tratamento INTERFACE, existe um meio que torna sobre as diferenças interfaces. Neste meio, trata-se prioritariamente dos lotes de interfaces já existentes no estado "1 - A tratar", seja para os integrar na BDD para uma interface em recepção, seja para criar os ficheiros emparelhados no directório EXPREP para um interface em emissão. Para um interface em recepção, se não existe nenhum lote a tratar, pesquisa-se então se novos ficheiros emparelhados chegaram no directório IMPREP, e se é o caso constitui-se de novos lotes de interface a partit deste.

Esta pesquisa de novos ficheiros emparelhados no directório IMPREP implica o lançamento de uma ordem sistema custoso em recurso. Se faz frequentemente, "levam-se" os recursos da máquina, para não reestabelecer em mádia que pouco ficheiros em cada vez. Se o fazemos menos frequentemente, colocam-se menos, e reestabelece-se em média em cada vez mais, mas está-se menos reactivo.
A 1ª vez que é suposto lançar este pesquisa para um meio de interfaces, observa-se se está despejado ITASLEEP segundos depois a última vez onde se está colocada a questão e que se responde Sim. Se se despejou ITASLEEP segundos, vai-se então lançar estes recursos neste meio de interface, senão não fará neste meio aqui mas num meio seguinte quando ITASLEEP segundos serão ocorridos.
Se se faz um meio completo a vazio (sem tratamento de lote de interface, nem detecção de novos ficheiros emparelhados na recepção), espera-se ITAWAIT segundos antes de relançar um novo meio.

Para cada modelo de interface a utilizar, convém definir precisamente a parametrização a aplicar. Para isso, é aconselhado duplicar o existente afim de conservar a parametrização de referência.

  • As interfaces no sentido Warehousing versus o nível 3 são pré-fixados de X (para exportação).
  • As interfaces no sentido nível 3 versus Warehousing não são pré-faxadas.

Listam-se todas as interfaces definidas por um estabelecimento, ou para um conjunto estabelecimento (a preencher no separador "ambiente"). Por defeito (valor "*" para estabelecimentos e recipiente) as interfaces serão gerados idênticos para todos os estaeblecimentos e recipientes dos estabelecimentos de um dossier.

As informações no ecrã permitem indicar :

  • Se o interface está activo
  • Qual é a fila das importações / exportações para cada um dos fluxos
  • Sobre qual limite horário o interface está activo.
  • O nome do modelo a utilizar para o interface.
  • Se os ficheiros gerados em exportação devem fazer objecto de um contador particular
  • O nome do ficheiro a gerar para cada uma das interfaces.
  • O caminho de acesso a utilizar para integrar e remeter os ficheiros.

É então possível de parametrizar para os recipientes e/ou dos estabelecimentos distintos dos nomes diferentes de interfaces, e logo diferentes nomes arrancados nos directórios distintos. Poder-se-á ter por exemplo XSHP01 para o recipiente 01, com o modelo XSHP01, e interface XSHP02 para o recipiente 02, com o modelo XSHP02.

EXPLORAÇÃO / INTERFACES / MODELOS

Para todas as interfaces via troca de ficheiros, o separador Geral permite de descrever no modelo do interface como Sage Warehousing lê a estrutura do ficheiro de interface :

  • a sua estrutura (delimitada ou comprimento fixo),
  • o jogo de caracteres,
  • o formato das datas,
  • o tipo de separador decimal,
  • os separadores de linhas e de registos,
  • o caracter a interpretar em caso de não modificação...

No segundo separador, indica-se a lista dos campos das tabelas SAGE Warehousing que se deseja importar / exportar.

Para cada tabela indica-se então um código acção "$" que valha Criar / Modificar / Suprimir (valores 1, 2 ou 3), depois um código de separação "/" que indica o início dos campos de uma outra tabela, por exemplo "E" (cabeçalho) ou "L" (linha associada a uma entidade), "P" para pacotes associados.

A estrutura do modelo é modificável : acrescentar ou suprimir um campo da mesma tabela do modelo está com efeito possível, falta no entanto pensar a recalcular as posições da estrutura em fazendo clique direito com o rato e "ajustamento das posições" (a fazer sistematicamente em caso de modiifcações do modelo).

É fortemente aconselhado transmitir os identificadores código acção, tipo de registo, estabelecimento, depositante de maneira sistemática.

Definição da Estrutura:
A estrutura das informações no seio de um registo se define para cada modelo de interface:

  • Campos com comprimento variável (ou delimitados)
    • É o tipo de estrutura utilizado por defeito.
    • Os formatos mencionados nas descrições dos registos de informação correspondente aos formatos máximos de cada campo ; os caracteres não significativos (brancos ou zeros) diante de serem suprimidos.
    • Correspondem ao formato dos dados na base Warehousing.
    • O fim de cada campo (entre o último registo) está identiifcado por um separador de campo (classicamente ";" )
    • Por outro lado, cada campo pode ser cercado de delimitadores de campo (classicamente " " "); estes delimitadores de campo não são gerados em exportação para os camops numéricos e os campos de tipo data.
  • Campos com comprimento fixo
    • A posição e o comprimento de cada campo são fixos.
    • Não existe separador nem delimitador do campo.
    • As regras de apresentação e de enquadramento das informações (em particular dos campos numéricos) são parametrizáveis.
      • Os dados alfanuméricos são enquadrados à esquerda e completar por brancos.
      • Os dados numéricos são enquadrados à direita e completados por zeros se necessários, o separador decimal não está transmitido.
  • Separador de registo
    • Para registar um caracter não imprimível, falta entrar uma "\" (barra de fracção invertida) seguida de 3 números representando o código ascii do caracter em base decimal.
      Frequentemente, utiliza-se:
      • o caracter "line feed" (\010), que corresponde ao fim de linha nos ficheiros textos sob Unix.
      • a combinação de dois caracteres "carriage return, line feed (\013\010)n" que corresponde ao fim de linha nos ficheiros textos Windows.
    • Define o separador entre dois registos (grupos de dados).
    • Para registar um caracter não imprimível, falta entrar uma "\" (barra de fracção invertida) seguida de 3 números representando o código ascii do caracter em base decimal.
      Frequentemente, utiliza-se:
      • o caracter "line feed" (\010), que corresponde ao fim de linha nos ficheiros textos sob Unix.
      • a combinação de dois caracteres "carriage return, line feed (\013\010)n" que corresponde ao fim de linha nos ficheiros textos Windows.

Identificadores
Permite definir o código permitindo a identificação do tipo de registo ao nível de interface. Estes códigos são determinados em standard segundo as necessidades úteis a SAGE Warehousing, eles não são modificáveis em standard.

 
Transcodificação

  • Jogo de caracteres
    • ASCII (ISO 8859-1) está classicamente utilizado.
    • UNICODE (UTF8) é igualmente possível.
    • UCS-2 ?
  • Formato de datas possíveis : DDMMAA, DDMMAAAA, AAMMDD, AAAAMMDD.
    Supõe-se pelo seguinte que o formato escolhido é DDMMAAAA.
  • Formato dos menus locais
    Um menu local é um tipo de dados fazendo referência a uma lista pré-definida de valores admitidos.
    Classicamente nas interfaces, identifica-se cada um dos valores pelo meio de um dado numérico correspondente à sequência do valor na lista.
    Assim, para um indicador binário (não/sim) :
    O valor "1" significa "Não".
    O valor "2" significa "Sim".
  • Separador decimal
    Quando este último está comunicado (o que não é forçosamente o caso para uma estrutura com comprimento de campo fixo).
    Por defeito utiliza-se o ponto.
  • Simbolo de campo vazio
    Este símbolo pode ser importante no caso de importações em modificação ; ele permite reparar os dados incluidos no interface cujo não se deseja modificar os valores.
    Utiliza-se classicamente o símbolo "$".
  • Símbolo CR + LF
    Este símbolo é a utilizar para reparar os saltos de linhas no seio de campos multi-linhas (por exemplo, os campos de comentários) distintamente da maneira cujo se repara os fins dos registos.
    Utiliza-se classicamente o simbolo " |~ "

Tratamento de interface

Nós vamos abordar os sujeitos seguintes :

  • Lançar as interfaces
  • Expurgar os directórios
  • Consultar o estado de avanço do conjunto das interfaces.
  • a tarefa batch interface

Exploração / Interfaces / Lançamento

Os tratamentos de interface são lançados manualmente ou pelo meio de uma "assinatura" (tarefa recorrente lançada ao período parametrizável), para um interface ou para um conjunto dos interfaces. Neste último caso, os interfaces são tratados por tipo de fluxo segundo uma ordem parametrizável e, por uma importação para um tipo de fluxo dado, na ordem crescente dos números do ficheiro.

IMPORT
O tratamento de importação se decompõe em 2 etapas:

  • Etapa 1 : Importação do ficheiro
    Controlo de coerência da estrutura por relação à descrição do modelo.
    Em caso de sucesso, o ficheiro recebido está salvaguardado no directório de "destino importação", os dados são copiados nas tabelas dedicadas por tipo de fluxo depois tratadas na etapa 2.
    Em caso de prejuizo, o ficheiro recebido está salvaguardado no direcrório de "erro de importação", o ou os lotes de interface dão são reputados em "erro importação"; eles não são recicláveis.
  • Etapa 2 : Integração dos dados
    Controlo de coerência funcional dos dados para cada lote de interface.
    Em caso de sucesso, as actualizações são realizadas ; o lote está reputado "tratado".
    Em caso de insucesso, o lote está reputado em "erro de integração" ; a lista dos erros detectados está registado, o lote poderá ser reciclado ou saldado. Um erro detectado sobre uma parte de um lote (cabeçalho ou linha) se traduz então pela rejeição do conjunto dos registos ligados.

EXPORTAÇÃO
O tratamento de exportação se decompõe igualmente em 2 etapas :

  • Etapa 1 : Alimentação das tabelas de interface
    Esta alimentação está realizada pelos diferentes eventos desencadeantes (fim de preparação de uma OE, validação de uma lista de arrumação, etc.)
    Os lotes de interface são reputados "a tratar"
  • Etapa 2 : Geração do ficheiro de interface
    O conjunto dos lotes "a tratar" para um dado fluxo dado lugar à geração de um ficheiro.
    Este ficheiro está à disposição no directório "export" ligado ao fluxo e cópia no directório de "salvaguarda export".
    Os lotes são reputados "tratados".

Exploração / Interfaces / Expurgo dos directórios

Esta função permite expurgar manualmente os ficheiros de interface guardados nos diferentes directórios, indicados em valor parâmetro (capítulo INT) em indicando uma duração de conservação exprimida em dias.

Exploração / Interfaces / Consulta

Uma função standard permite a consulta e a gestão dos lotes de interface. Ela apresenta uma visão sintética do estado de cada um dos interfaces segundo que eles são em emissão ou em recepção ou em global (emissão de recepção).
Os status seguintes são possíveis :

  • A tratar
    O lote será integrado ou exportado ao próximo tratamento de interface.
  • Tratado
    O lote foi integrado ou exportado.
  • Erro importação
    A estrutura de um ou vários conjuntos dos lotes de um ficheiro de importação incorrecta ; os dados do ficheiro não foram integrados.
    Uma vez o erro identificado e as acções paliativas comprometidas, o status pode ser alterado em "Erro tratado".
  • Erro de integração
    Um erro "funcionalidade" foi detectada sobre o lote importado.
    Após análise do erro o lote poderá ser reportado "A tratar" para ser de novo considerado no próximo tratamento de interface (o lote será reciclado) ou ser decretado em "Erro tratado".
  • Erro exportação
    É produzido um erro sobre a criação do ficheiro de interface (etapa 2). E não de erro sobre a geração do lote (etapa 1).
  • Bloqueado
    Decisão de bloquear manualmente uma interface em erro (qualquer que ela seja) afim de extrair da população dos interfaces em erro e de a tratar ulteriormente.

Princípio de utilização:
1. Escolher o interface na lista apresentada sobre o separador Interfaces.
Estabelecimento, código interface, nome da parametrização são afixadas. A lista de interface está igualmente afixado.
2. Depois visualizar o detalhe dos lotes deste interface no separador Lots.
A referência está identificado do lote; em regra geral se trata da referência entrada esperada, referência entrada directa, referência ordem de entrega, código artigo.
O número faz referência à identificação de cada registo. O ficheiro faz referência ao contador de ficheiro gerado ou integrado. Se não está preenchido, isso significa que o registo está no estado "A tratar".
A data e hora de criação são relativas à data de integração dos dados na tabela interface. A data e hora de modificação são relativas à integração dos dados nas tabelas vivas.
3. e finalmente consultar o detalhe das linhas do lote seleccionado precedenteemnte no separador Linhas.

Este ecrã permite igualmente de gerar estas interfaces. Assim os lotes "Em erro" podem ser manualmente basculados numa lista "Erros Tratados" ou "A tratar" por clique direito sobre o lote em questão.

Exploração / Servidor batch / Tarefa batch Interface

A tarefa batch INTERFACE permite gerar automaticamente os ficheiros de interfaces (escrutínio directórios de interface, importação e exportação dos ficheiros interfaces).

Esta tarefa batch pode ser parada/arrancada de maneira externa ao produto Warehousing graças à colocação de um sistema de ficheiros testementas cujas eis as regras:

1) No arranque da tarefa batch INTERFACE, acrescenta-se de um ficheiro testemunha sob o directório do dossier ("/FIL/INTERFACE.run")
2) Seguinte ao arranque do ficheiro testemunha seguinte : "FIL/INTERFACE.stop" sob o repertório dossier, a paragem automática da tarefa do dossier correspondente se faz que após ter tratado todo um ciclo completo das parametrizações interface activos em recepção / emissão.
3) Seguinte ao registo do ficheiro testemunha seguinte : "FIL/INTERFACE.kill" sob o directória dossier, a paragem automática da tarefa do dossier correspondente se faz desde logo que um interface terminou de ser tratado.

Assim, a paragem do servidor batch por registo do ficheiro testemunha seguinte : "/SERVGX/FIL/kill" pararia previamente a tarefa batch INTERFACE.

Complementos

Ficheiros de interface

Os nomes dos ficheiros de interface são definidos como segue:

  • Raíz (10 caracteres alfanuméricos máximo)
    Sinónimo do fluxo de interface ; ela está definida na tabela dos "parâmetros".
    Na medida onde certa raíz identifica o fluxo de interface na função de seguimento das interfaces e de gestão das anomalias ; é importante de adoptar uma codificação coerente e significatica ;
    por exemplo :
    • 1 caracter "X" para as interfaces (N2 => N3)
    • 2 caracteres para a identificação do sistema terceiros
    • 2 ou 3 caracteres para a natureza do fluxo ("ART" para os artigos …etc.)
  • Número de sequência
    No caso dos interfaces (N2 => N3), o formato do número de sequência está definido pelo meio de um parâmetro geral; por outro lado, pode-se utilizar um único e mesmo contador para o conjunto das interfaces, ou bem definir os contadores dedicados em cada interface conjunto interface.
  • Extensão
    A extensão "imp" está classicamente utilizado para os ficheiros de importação e da extensão "exp" para aquele de exportação.

Registo no seio de um ficheiro de interface

Um ficheiro de interface é multi-registos.
O separador de fim de registo é um parâmetro do modelo de interface.
No caso das interfaces a 2 níveis tais que o interface artigo (nível artigo e recipiente) ou a interface de ordens de entrega (cabeçalho pedido e linha de pedido) :

  • Cada nível é reparado por uma informação "tipo de registo" distinto no início de registo mencionado nas descrições das estruturas de dados.
  • A relação de dependência está significada por ordem relativa dos registos; os registos de nível "linha" diante de seguir ao registo de nível "cabeçalho" aos quais se reafixam.
    Diz-se de lote de interface para designar um conjunto de registos relativos a um mesmo registo de cabeçalho.

Estrutura dos ficheiros de transacção

Ela está parametrizável (ver o capítulo Parametrização).
A posição e a escolha dos campos sendo modificável, este documento é um exemplo de estrutura dos ficheiros. Por outro lado, as regras de gestão fumcionais associadas aos campos, para as importações, sao constantes. A estrutura descrita neste documento é aquele de estrutura delimitada (na diferença de estruturas a comprimento fixo). O separador de registos é a devolução do carro (não indicado em cada descrição da estrutura).

Código acção importação

  • "1" = Criação
  • "2" = modificação
  • "3" = supressão
  • "4" = leitura

Controlos são realizados por relação a este código, depois outros controlos funcionais segundo o contexto.
 
Regras de gestão
 
 Para suprimir uma entidade mãe (uma O.E, um artigo, uma EE, ou uma ED) é suficiente de transmitir o cabeçalho com o código acção = "3" = supressão (neste caso as entidades filhas serão suprimidas automaticamente).
 
 Para suprimir uma entidade filha (uma linha de OE, um recipiente artigo, uma linha EE, ou uma linha ED) é suficiente de transmitir o cabeçalho com o código acção = "4" = leitura e aquela da linha a suprimir = "3" = supressão. Para suprimir todas as entidades filha é útil de passar pela supressão da entidade mãe.

Para modificar, o mesmo princípio está aplicado; é todavia possível de modificar certas zonas sem afectar outras em transmitindo a entidade a modificar com um código acção igual a "2" = modificação, e em colocando nas zonas a não modificar o código definido no modelo.

Para a supressão de dados, em importação, é impossível de suprimir uma entidade mãe (artigo por exemplo) se ela está utilizada por uma entidade associada (objecto de stock, movimento de stock, etc.)

Para a criação ou a modificação, é possível de colocar os valores por defeito para um campo de uma interface não trasnmitida.
Desde logo que uma parte de um lote foi rejeitado, o lote inteiro não foi integrado Exemplo : se uma linha de OE está rejeitada, a integralidade da encomenda será rejeitado.

Para a modificação de dados (cabeçalhos e/ou linhas), em importação, se as informações são transmitidas, o conteúdo do campo coloca em dia aquele de Warehousing.

Para evitar de actualizar os dados modificados em Warehousing, é possível transmitir um caracter no campo (a parametrizar) de maneira a indicar que o campo não deve ser modificado.

Para gerar correctamente as modificações sem alterar os modelos, trnasmitidos na mesam estrutura que em criação em indicando os dados modificados pelo N3, os outros dados sendo trnasmitidos com o valor "campos não modificáveis"
