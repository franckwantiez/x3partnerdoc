Monitor Workflow

O monitor Workflow permite de apresentar os eventos em espera de assinatura por um utilizador dado. Esta apresentação, parametrizável por transação, pode-se fazer sobre um ou vários separadores. A partir de cada evento, pode-se visualizar as informações relativas ao contexto de desencadeamento, às mensagens enviada, ao histórico das assinaturas. Pode-se igualmente assinar ou visar o evento segundo os casos.

Gestão do ecrã

  • Cabeçalho
  • [Tab] Plano de trabalho

O registo no plano de trabalho faz-se sobre um número variável de separadores, segundo a transação escolhida. Sobre cada um dos separadores, encontramos informações que podem ser diferentes, e parâmetros caso a caso. Assim, cada um dos separadores faz aparecer todo ou parte dos eventos a tratar, a repartição se fazendo segundo as regras parametrizáveis. Enfim, uma colocação em valor (por cor, pelo tamanho da fonte de caracteres) e uma ordenação segundo diferentes critérios realizado ao caso a caso.

Cabeçalho

Regista-se em cabeçalho de critérios de filtragem globais sobre os eventos a tratar, sabendo que filtros mais finos podem existir sobre cada separador da função.

  • Delegado excepcional ([field] DELEXP)

Quando esta check box está assinalada, o utilizador vê aparecer as notificações enviadas aos utilizadores para os quais possui uma delegação excecional.

Esta check box não pode ser assinalada que se o utilizador possui ao menos uma delegação deste tipo.
  • Destinatário ([field] USER)

Define o código utilizador destinatário dos eventos de Workflow em espera de assinatura que se deseja ver.

Não se pode registar que nestes dois casos :

  • Se a opção Todos os utilizadores está acordado ao utilizador para a função.
  • Se a check box Delegado excepcionalestá assinalada (neste caso, pode-se escolher o utilizador para o qual se tem uma delegação excepcional cujo se deseja ver as notificações).

Se este campo não pode ser registado, é por defeito igual ao valor corrente do utilizador.

Se o campo pode ser registado, mas que nenhum valor lhe está dado, as notificações seguintes podem ser vistos :

  • aquelas dos utilizadores para os quais se tem uma delegação excepcional se a check box Delegado excepcionalestá assinalada.
  • aquelas endereçadas à totalidade dos utilizadores senão.
  • Data início ([field] DATDEB)

Se esta data está preenchida, apenas os dados cuja data de última modificação ou da data de criação é superior ou igual são tomados em conta no tratamento.

De referir que, por defeito, a data proposta é a data do dia, diminuida de um número de dias igual ao valor do parãmetro WRKDAY.
  • Data fim ([field] DATFIN)

Se esta data está preenchida, apenas os dados cuja data de última modificação ou da data de criação é inferior ou igual são tomadas em conta no tratamento.

Por defeito esta data é igual à data do dia.

[Tab] Plano de trabalho

Encontramos em cada separador deste tipo (podem existir até 8), as notificações de Workflow para os quais uma regra de signatário ou de visto foi parametrizado.

A ordem de ordenação por defeito das informações, e a colocação em evidência (por um estilo) das linhas devem ser colocadas em exergo são parametrizáveis pelo separador.

Botões de navegação permitindo percorrer o conjunto das linhas do plano de trabalho.
  • [field] PAGPRV
  • [field] PAGNXT
  • [field] PAGLST

  • [field] PAGCUR
  • [field] SAICRI

Botão que permite abrir um ecrã de registo de um critério de selecção das linhas sob a forma de uma expressão : lógica.

Este critério está aplicado em todos os separadores em mais dos critérios específicos em cada separador.

[Grid]

  • Crono ([field] CHRONO)

Número cronológico atribuido quando um workflow está desencadeado. No caso de um Workflow manual, um número é atribuido para cada registo desencadeador. Em todos os outros casos, um único número cronológico está atribuido por desencadeamento.

Assim, quando uma notificação está enviado a vários utilizadores a partir de um mesmo evento, o número de tempo é o mesmo.
  • Destinatário ([field] DEST)

Define o destinatário da notificação. Se um visto ou uma assinatura está esperado, é normalmente ele que deve o fazer.

Este destinatário não é forçosamente o destinatário que está definido na regra de origem. Com efeito, se a regra prevê uma delegação ao primeiro destinatário livre, e que se o destinatário de partida definiu um delegado com poder para as datas afetadas, é o delegado que é o destinatário da notificação (ou seu delegado se ele mesmo foi delegado...).
  • Email destinatário ([field] EMAIL)

Quando a mensagem está enviada por regra a um destinatário, o seu endereço de mensagens aparece neste campo.
  • Nr. de workflow ([field] NUMGRP)

Este número de tempo está atribuido para cada linha tratada (existe então várias quando se tem um worflow de tipo em cabeçalho/ linha com os reagrupamentos.
  • Socied. ([field] CPY)

Quando um modelo de dados está anexado à regra, e que este modelo define uma sociedade corrente em função do contexto (que seja diretamente ou pelo intermédio de um estabelecimento), guarda-se no evento a sociedade correspondente

Esta sociedade pode ser utilizada (mas não é obrigatório) para definir a regra de afetação que determinará os destinatários da notificação.
  • Delegado ([field] DELEGUE)

Este campo vale Não se o utilizador é o primeiro destinatário da notificação. Vale Sim se o utilizador recebe a notificação em tanto que delegado não substituto (opções Todos ou Cascata), e que por outro lado pode assinar (se não existe poder de assinar, a notificação não aparece no plano de trabalho.

Falta notar que se este campo vale Sim, pelo menos uma outra linha portando o mesmo número de tempo com o valor Não existe na tabela guardando os dados do plano de trabalho (pode ter outras linhas com o mesmo número de tempo e este indicador igual a Sim se vários delegados com poder de assinatura existente). A primeira assinatura sobre uma das linhas afetadas coloca automaticamente a dia o conjunto de linhas levando o mesmo número de tempo em os considerando como assinados.
  • Envio mail ([field] ENVOI)

Este campo vale Sim se uma mensagem foi enviada no momento de execução da regra.
  • Emissor ([field] EMETTEUR)

Definir o emissor da notificação, quer dizer utilizador cuja ação desencadeou o worflow. Num workflow de tipo Assinatura, o emissor é aquele que assinou a notificação precedente Num workflow outro, o emissor é o utilizador cuja ação desencadeia o envio da notificação.
  • Email emissor ([field] MAIENV)

Quando uma linha do plano de trabalho foi assinada pelo desencadeamento de uma relação http depois uma mensagem, este campo guardado de mensagens do utilizador assinado da espécie.
  • Data emissão ([field] DATENV)

Corresponde à data sistema (tal que ela está conhecida do servidor de tratamento) no momento de execução do evento de workflow.
  • Hora emissão ([field] TIMENV)

Corresponde à hora sistema (tal que ele está conhecido do servidor de tratamento) no momento de execução do evento de Workflow.
  • Natureza de workflow ([field] NATURE)

Corresponde ao valor do campo Natureza de Workflow definida sobre o evento desencadeador. Pode ser um critério cómodo para triar as linhas, para as filtrar, ou para os despachar sobre os diversos separadores do plano de trabalho.
  • Texto seguim. ([field] TEXSUI)

Contém o texto avaliado a partir da rubrica Texto seguido definição no separador Seguimento da regra; usualmente, parametriza-se um cometário explicando as circunstâncias do desencadeamento de Workflow.
  • Tipo de evento ([field] TYPEVT)

Define o tipo de evento da regra de Workflow na origem da linha de notificação no plano de trabalho.
  • Cód. de evento ([field] CODEVT)

Corresponde ao código evento na origem do workflow, quer dizer do código :

  • de objeto no caso de um workflow objeto.
  • de lista no caso de um workflow de tipo impressão.
  • do modelo no caso de uma importação de uma exportação.
  • da função no caso de uma entrada sobre a função.
  • da regra de workflow tendo desencadeado o pedido de assinatura no de uma assinatura.
  • do código identificante o workflow diverso neste caso.
  • Cód. workflow ([field] CODWRK)

Identifica a regra de Workflow cuja execução foi enviada a notificação.
  • Operação ([field] OPERATION)

Contém o código operação no contexto desencadeante :

  • para um evento objecto, pode ser C (criação), M (modificação... ou o código do botão.
  • para um evento assinatura, o código da regra na origem da assinatura
  • para uma importação/exportação, as combinações I/E e D/F (importação/exportação, início /fim), por exemplo ID.
  • para uma entrada sobre função, o campo está vazio.
  • para uma impressão, é igual a C.
  • para o fim de uma tarefa, vale "0" se a tarefa está terminada sem erro, senão, encontramos o número de status seguido do início da mensagem de erro.
  • Prazo assinatura ([field] DATMAXSIG)

Este campo contém a data na qual o workflow deverá ter sido assinado. É igual à data de desencadeamento se nenhum processo de assinatura não foi colocado em obra, ou se nenhuma data limite foi definida.

Com efeito, quando o evento de Workflow supõe uma assinatura :

  • pode-se gerir uma data limite de Workflow (via a rubrica que dá uma expressão calculada correspondente na parametrização do Workflow).
  • pode-se em seguida desencadear os reavisos baseados sobre esta data limite. Estes reavisos fazem por regras de Workflow batch nomeados WRKREM1, WRKREM2, e WRKREM2E.

Quando um reaviso deste tipo está feito :

  • incrememnta-se o campo Número de reavisos, o que permite então de saber quantas vezes a assinatura foi relançada.
  • actualiza-se o campo Última data de reaviso, o que permite de conhecer na data na qual a última assibnatura teve lugar.

Se estas regras batch não são utilizados, e se nenhuma outra regra foi definida para movimentar estes campos, o número de reaviso resta igual a 0 e a data de reaviso resta vazio, porque nenhum tratamento standard do Workflow não modifica este valor.
  • Flag assinatura ([field] FLGSIG)

Este campo define o nível de assinatura correspondente à notificação da linha. Os valores seguintes, definidos pelo menu local 2922. são possíveis :

  • A ler : um acusado de leitura foi pedido, mas a notificação não foi ainda lida.
  • A assinar : uma assinatura foi pedida, mas a notificação não foi ainda assinada.
  • Lido : um acusado de leitura pedida, foi registado.
  • Assinado : uma assinatura foi pedida, ela está acordada.
  • Anulado : uma assinatura foi acordada, depois ela fez objeto de uma anulação de assinatura.
  • Resposta ([field] ACTSIG)

Este campo está vazio tanto que o evento não foi assinado. Se foi, encontra-se o código resposta que foi registado, tal que foi definido na tabela diversa número 54.
  • Signatário ([field] USRSIG)

Quando uma linha do quadro das notificações foi assinada, vê-se aparecer aqui o código de utilizador que o assinou. Não se trata forçosamente do destinatário de origem, pois que um outro utilizador pode ter poder para assinar.
  • Emais signatário ([field] MAISIG)

Quando uma mensagem incluindo um pedido de seguimento por activação de uma ligação http foi enviada, este campo permite de conhecer o endereço de servidor de e-mail do destinatário.
  • Data assinatura ([field] DATSIG)

Se a notificação de workflow supõe um seguimento (simples visto ou assinatura), este campo permite de saber a que hora o seguimento é feito (resta vazio tanto que ele não foi realizado).
  • Hora assinatura ([field] TIMSIG)

Se a notificação de workflow supõe um seguimento (simples visto ou assinatura), este campo permite de saber a que hora o seguimento foi feito (resta vazio tanto que ele não foi realizado).
  • Motivo resposta ([field] REASON)

Contém o titulo associado ao código motivo registado na assinatura da regra, quando este registo é possível no momento da assinatura. Isto supõe que se define uma lista de motivos possíveis pelo intermédio de um número de tabela diversa sobre a linha de assinatura correspondente.
  • Código motivo ([field] REACOD)

Contém o código motivo registado na assinatura da regra, quando este registo é possível no momento da assinatura. Isto supõe que se define uma lista de motivos possíveis pelo intermédio de um número de tabela diversa sobre a linha de assinatura correspondente.
  • Nr. motivo ([field] REANUM)

Quando se regista o código motivo foi previsto na assinatura da regra, encontra-se neste campo o número da tabela diversa que define os motivos possíveis sobre a linha de assinatura correspondente.
  • Chave arranque ([field] CLEOBJ)

No caso de um Workflow objeto, encontra-se neste campo a chave do objeto (é o campo Chave desencadeante de origem No caso de um workflow de tipo assinatura, encontra-se neste campo o número tempo na origem da assinatura. Em todos os outros casos, esta zona não foi preenchida.
  • Tabela de arranque ([field] IDENTREF)

Este campo técnico contém o código da tabela desencadeante e o código da chave correspondente, exatamente da mesma maneira que para o campo Identificador linha. Por outro lado, em caso de evento de tipo assinatura, resta vazia, quer dizer que não é transmitida depois a notificação de origem.
  • Identific. retorno ([field] IDENTRET)

Quando um ícone de devolução está associado ao workflow, este campo contém o nome da função sobre a qual a devolução se faz. Se a função é de tipo objeto, encontra-se em seguida, separada por um "/" da tabela, a chave correspondente (os segmentos de chaves, para uma chave em várias partes, sendo separadas por "~").
  • Identificador linha ([field] IDENTGRP)

Este campo técnico contém, na ordem, o nome da tabela na qual se encontra o evento desencadeante de origem, seguido da chave corrente ligada ao grupo que provocou o desencadeamento. O separador entre a tabela e a chave é "/", o separador entre os segmentos da chave (quando ela está composta de vários campos) é "~".

O conteúdo está definido da maneira seguinte :

  • no caso de um evento de tipo Assinatura, retoma-se o conteúdo da notificação de origem.
  • no caso ou evento é de tipo Objeto, retoma-se a tabela principal do objeto se pode ser determinado (o que significa que nada está registado se é um workflow genérico com um código objeto vazio). A chave corresponde à chave corrente no momento do desencadeamento workflow
  • em todos os outros casos, a tabela está definida unicamente se um modelo de dados está anexado ao workflow. Neste caso, é a tabela principal que é reenviada. A chave corresponde ao registo corrente no momento do desencadeamento.
  • Crono. origem ([field] NUMORG)

Este número tempo está preenchido unicamente sobre as notificações ligadas a um evento de tipo Assinatura; permite conhecer o número cronológico de evento de origem, aquele cuja assinatura desencadeou a linha corrente.
  • Destinatário origem ([field] USRORG)

Este campo não é preenchido que sobre as linhas asssociadas a um evento de tipo Assinatura. Permite conhecer o destinatário do workflow de origem (aquele que efetivamente recebeu a notificação após aplicação eventual das regras de delegação : isto não é forçosamente aquele que assinou).
  • Data reaviso ([field] DATREL)

Este campo contém a data na qual o último reaviso de assinatura foi feito sobre a notificação. Não está registada se nenhum processo de assinatura foi colocado em obra.

Com efeito, quando o evento de Workflow supõe uma assinatura :

  • pode-se gerir uma data limite de Workflow (via a rubrica que dá uma expressão calculada correspondente na parametrização do Workflow).
  • pode-se em seguida desencadear os reavisos baseados sobre esta data limite. Estes reavisos fazem por regras de Workflow batch nomeados WRKREM1, WRKREM2, e WRKREM2E.

Quando um reaviso deste tipo está feito :

  • incrememnta-se o campo Número de reavisos, o que permite então de saber quantas vezes a assinatura foi relançada.
  • actualiza-se o campo Última data de reaviso, o que permite de conhecer na data na qual a última assibnatura teve lugar.

Se estas regras batch não são utilizados, e se nenhuma outra regra foi definida para movimentar estes campos, o número de reaviso resta igual a 0 e a data de reaviso resta vazio, porque nenhum tratamento standard do Workflow não modifica este valor.
  • No. reaviso ([field] NBRREL)

Este campo contém a conta do número de reactivações de assinaturas feitas sobre o evento Workflow.

Com efeito, quando o evento de Workflow supõe uma assinatura :

  • pode-se gerir uma data limite de Workflow (via a rubrica que dá uma expressão calculada correspondente na parametrização do Workflow).
  • pode-se em seguida desencadear os reavisos baseados sobre esta data limite. Estes reavisos fazem por regras de Workflow batch nomeados WRKREM1, WRKREM2, e WRKREM2E.

Quando um reaviso deste tipo está feito :

  • incrememnta-se o campo Número de reavisos, o que permite então de saber quantas vezes a assinatura foi relançada.
  • actualiza-se o campo Última data de reaviso, o que permite de conhecer na data na qual a última assibnatura teve lugar.

Se estas regras batch não são utilizados, e se nenhuma outra regra foi definida para movimentar estes campos, o número de reaviso resta igual a 0 e a data de reaviso resta vazio, porque nenhum tratamento standard do Workflow não modifica este valor.
  • Nível da assinatura ([field] LEVSIG)

Este campo corresponde ao nível de assinatura, quando os eventos de assinatura se encadeando :

  • vale 0 sobre o evento de origem.
  • vale 1 se o evento é desencadeado pela primeira assinatura.
  • em cada evento que se encadeia, é incrementada de 1.

A variável LEVSIG correspondente é frequentemente utilizada nas regras de workflow quando a regra de afetação do evento de origem dá o conjunto dos signatários (1 a 9 máximo). É então suficiente, no evento de origem como os seus eventos de assinatura que se encadeiam, de dar como fórmula de destinatário USER (LEVSIG+1) para ter o utilizador número N na lista. Isto permite nomeadamente de ter as regras de assinatura recursivas, executadas tanto que resta dos signatários.
  • Icône de retorno ([field] CONTXT)

Quando um workflow envia uma mensagem com um ícone de branqueamento, o contexto de conexão (dossier, servidor, serviço, função...) está definido em forma de um campo alfanumérico.

Este campo técnico contém este contexto. Os diferentes segmentos deste camop são separados pelo caracter "/".
  • Retorno desktop ([field] CONTXTSYRA)
  • Retorno telemóvel ([field] CONTXTMOBI)
  • Chv. desencad. de origem ([field] CLEDEC)

Identifica a chave de origem do circuito de workflow. Esta chave, quando está composto de várias partes, se apresenta como uma concatenação das partes de chaves separadas pelo caracter til (~).

Os valores de chave dependem do contexto :

  • sobre um workflow objeto, trata-se da chave da ficha na origem do desencadeamento.
  • sobre um workflow manual, trata-se da chave da tabela principal sobre a qual se apoia o modelo de dados do workflow.
  • sobre um workflow sobre importação/exportação, sobre uma entrada numa função, nada foi registado nesta zona.
  • sobre um workflow sobre edição, nada está registado se a edição está lançada depois uma função de impressão (impressão, ou impressão por grupo). Por outro lado, se a edição está lançada por Ficheiro/Impressão ou Ficheiro/Lista depois um objeto, a chave da ficha está registada nesta zona.
  • sobre um workflow de assinatura, herda-se da chave da ficha de origem, guardado na linha de seguimento na origem da assinatura.
  • Nr. de signatários ([field] NBRUSR)

Define o número de utilizadores reenviados pela regra de afetação, quando existe uma.

Se não há mais, mas que a regra é de tipo Assinatura, herda-se do número de signatarios definidos na regra de origem.

Se nenhuma regra de afectação existe, o valor retornado é 0.
  • Assunto ([field] ABROBJ)

Quando o primeiro evento na origem de um circuito de assinatura é um objecto, este campo identifica o código do objecto, Nos outros casos,ele está vazio.

É em seguida transmitidos aos diferentes eventos de assinatura que poderão se encadear ao primeiro.
  • Primeiro destinatário ([field] USRTOP)

Define o primeiro destinatário do evento de workflow. Com efeito, um evento pode enviar uma notificação com várias séries de destinatários, cada uma de entre elas sendo definidas por uma linha no quadro dos destinatários. Este campo representa o destinatário avaliado pela primeira linha se a opção de delegação é Não, Todos, ou Cascata. Se a opção é O primeiro livre , e se o utilizador não está efectivamente presente, e deu a delegação, encontramos no campo utilizador delegado.
  • Contexto 1 ([field] VALCTX1)

Este campo faz parte de um conjunto de quinze campos genéricos permitindo de guardar os valores contextuais avaliados no momento do desencadeamento do workflow a partir de fórmulas registadas no quadro nomeado Contexto do separador Seguimento do workflow.

Os valores guardados neste quadro podem ser utilizados :

  • para transmitir a um evento de tipo Assinatura das informações úteis ao tratamento do seguinte processo.
  • para fazer aparecer as informações úteis no monitor de workflow. Neste caso, em parametrização da transação, pode-se lhe dar um título particular mais falado.

  • Assinatura 1 ([field] USRSIG1)

Permite definir as assinaturas sucessivas, quando uma regra de afectação de destinatários renvia N signatários para a regra de workflow. Esta informação é nomeadamente utilizada para permitir de gerar correctament as anulações de assinatura.

Operação desencadeadora

Esta função permite de voltar em zoom sobre a função e o contexto na origem do Workflow. No caso de um Workflow sobre objecto, volta-se sobre a ficha correspondente.

Operação desencadeante de origem.

Esta função está acessível quando o evento na origem da notificação é de tipo Assinatura; neste caso, remonta-se ao evento de origem (se necessidade de maneira recursiva) para reencontrar a função e a ficha que desencadeia o início do processo.

Devolução

Esta função permite de voltar sobre a função e no contexto escrito descritos pelo ícone de atraso da parametrização de Workflow. Se nada foi definido de particular, é a operação de origem que é proposta.

[Block number] 5

Esta função permite fazer aparecer , numa janela de seleção, a lista das mensagens enviadas aos utilizadores sucessivos tendo assinado a notificação de origem e as notificações seguintes (quando o processo de assinatura se faz a vários níveis).

As linhas aparecem na ordem cronológica inversa (as assinaturas as mais recentes então)

Se faz duplo-clique sobre uma das linhas, uma janela abre-se para fazer aparecer o detalhe da mensagem enviada e a lista dos destinatários.

Assinatura

Esta função permite de desencadear o processo de assinatura da linha. Ela não está disponível que se o utilizador tem poder de assinar, e se a linha não está ainda assinada.

Uma janela abre-se então para propor uma escolha de respostas possiveis, em função da parametrização. Um motivo suplementar pode igualmente ser em registo segundo a parametrização, ver um texto livre (para motivar uma recusa por exemplo).

Visto

Esta função permite visar a linha. Esta função não está disponível que se a linha não foi visada. Nenhum outro registo foi pedido, a linha passando ao estado Lido a partir do momento onde esta operação foi feita.

Anular

Esta função permite anular uma assinatura acordada (em revindo ao estado anterior do pedido).

Ela não está disponível que se as condições seguintes são todas preenchidas:

  • a linha foi assinada
  • O utilizador é o signatário
  • o direito de anulação foi acordado ao utilizador (é uma das opções na função).
  • existe, no processo de assinatura, uma resposta associada ao código operação CAN. Este código motivo desencadeará a ação de anulação de assinatura em remetendo a linha na lista antes da assinatura Um outro Workflow de tipo assinatura, baseado sobre a regra de origem e o código motivo CAN, pode então ser desencadeado se necessário.

Esta operação está diferente de uma operação de rejeito do pedido ou de recuso, que será tratado de maneira todo em feito clássica por parametrização standard.
