Modelos importación/exportación

Esta función permite definir los formatos de los ficheros utilizados en las funciones de importación y exportación de objeto para integrar (o extraer) los datos en la base del sistema.

Un fichero que se puede utilizar en el módulo de importación y que se puede generar en el modulo de exportación es un fichero con una de las siguientes estructuras:

  • Fichero plano con campos de longitud fija.
  • Fichero plano con separadores entre los campos (y registros).
  • Fichero XML.

El módulo de importación/exportación utiliza el concepto de objeto para permitir la actualización de los datos en la base. Un objeto está formado por un conjunto de tablas y pantallas, así como por acciones estándar asociadas a los campos y a la actualización. De esta forma, se garantiza que se realizan todos los controles y acciones necesarios para la actualización de la base, ya que una sola descripción del objeto permite generar el código de una actualización en línea y de una actualización por importación.

Si un objeto solo actualiza una tabla, el modelo de importación define la lista de campos de la tabla que hay que integrar, teniendo en cuenta que un bloque de datos del fichero de importación o exportación contiene los datos de un registro. Si el objeto actualiza varias tablas (la cabecera y las líneas, por ejemplo), se encontrarán varios bloques de datos para una instancia del objeto que se va a integrar (un bloque que define la cabecera y N bloques que definen, cada uno, una línea).

También se puede importar una tabla sin tener que asociarla a una noción de objeto. Para ello, se deja en blanco el campo de objeto y se completan la tabla y campos correspondientes de la segunda pestaña del cuadro. La importación se lleva a cabo sin realizar ningún control, aparte de los que proceden de los formatos asociados a los tipos de datos de los campos de la tabla.

Cabecera

Un modelo de importación/exportación se identifica por un código alfanumérico. Además de una descripción, se definen las características técnicas del modelo en dos pestañas.

  • Modelo (campo EXT)

Código que identifica el modelo de import-export.
  • Descripción (campo INTIT)

Permite definir un título asociado a cada ficha.
  • Activo (campo ENAFLG)
  • Propietario (campo AFACTORYOW)

Pestaña Cabecera

Esta pestaña define las características generales del modelo, es decir:

  • Los datos que se van a exportar.
  • La estructura general del fichero (formato, código, definición de grupos de datos).
  • Parámetros complementarios.

General

  • Objeto (campo OBJ)

Define el código del objeto a importar o exportar. Cuando se trate de exportación, este campo es opcional; el nombre de la tabla principal a exportar se indicará en el bloque identificador.
  • Función (campo FONCTION)

Permite inicializar el contexto, especialmente si un mismo objeto se utiliza para varias funciones, y verificar los permisos de acceso. De hecho, los usuarios deben tener permiso de acceso adecuados para la función para poder utilizar el modelo.

Si no se ha indicado la función, se considera que se tiene permiso para todas las plantas y que se tienen derechos de creación, modificación y de supresión.

  • Módulo (campo MODULE)

Módulo al que pertenece la parametrización
  • Código actividad (campo CODACT)

Un código de actividad permite:

  • hacer opcional un elemento del diccionario si el valor asociado al código de actividad es nulo.
  • firmar los elementos específicos si están marcados por un código que empieza por X, Y o Z.
  • dimensionar un número máximo de líneas cuando el código de actividad marca los elementos de un cuadro.

De este modo, si el código de actividad no está activo, el elemento marcado no podrá utilizarse y el código asociado, si lo hay, no se generará ni podrá ser activado.
  • Cód. de acceso (campo ACS)

Este código de acceso permite restringir el acceso a la ficha actual a ciertos usuarios.
Si se indica el campo, sólo los usuarios que disponen de este código de acceso en su perfil podrán visualizar y modificar esta ficha.

El permiso de ejecución vinculado a un código de usuario se trata de forma especial en el caso de lo modelos de import-export: si un usuario no tiene permiso de ejecución, no podrá utilizar el modelo para importar o exportar datos.

  • Script estándar (campo TRTIMP)

Define este tratamiento en el que se encuentran las etiquetas de las acciones que se ejecutan en los tratamientos de importación-exportación.

Estos tratamientos permite realizar inicializaciones, controles complementarios y actualizaciones (si necesario). En el anexo técnico se encuentra la estructura de dicho programa. Se aportar para cierto número de importaciones, tratamientos estándar denominados generalmente IMPXXX, XXX, siendo éste el código de importación,

Para más información, ver la documentación anexa correspondiente.

  • Script específico (campo SPEIMP)

Define el tratamiento específico, que aparece antes del tratamiento estándar, y que permite realizar las mismas acciones desactivando si es necesario lo que hace el tratamiento estándar.

Este proceso permite realizar inicializaciones, controles complementarios y actualizaciones (si necesario).

Para más información, ver la documentación anexa correspondiente.

Estructura

  • Tipo de fichero (campo TYPFIL)

Define la estructura utilizada para generar los datos en el fichero a importar o a exportar. Para más información, ver el párrafo correspondiente.
  • Separador de campo (campo SEPFLD)

Define el separador entre dos campos.

Para introducir un caracter no imprimible, es necesario introducir '\' (barra de fracción invertida) seguido de 3 cifras que representan el código ascii del caracter en base decimal.
  • Separador registro (campo SEPREC)

Define el separador entre dos registros (grupos de datos).

Para introducir un caracter no imprimible, es necesario introducir '\' (barra de fracción invertida) seguido de 3 cifras que representan el código ascii del caracter en base decimal.

A menudo se utiliza:

  • el¡caractère line feed (\010), que corresponde al fin de línea en los ficheros de textos en Unix.
  • la combinación de dos caracteres carriage return, line feed (\013\010)n que corresponde al fin de línea en los ficheros de los textos Windows.
  • Delimitador de campo (campo FLDLIM)

El delimitador de campo (generalmente el carácter ") se adjunta en la primera y última posiciòn de los campos de tipo alfanumérico. Los campos numéricos y las fechas no utilizan este delimitador.

Normalmente se trata de uno de los caracteres siguientes:

  • 'comillas simples'
  • "comillas dobles"
  • Formato fichero (campo CODDBA)

Define el formato de los caracteres utilizados en el fichero:

  • ascii es el formato clásico en el que un caracter equivale a un octeto en el ficher. Este tipo de formato permite tratar caracteres occidentales clásicos, con diferentes juegos de caracteres posibles, definidos en el campo correspondiente.
  • utf8 corresponde a un formato UNICODE en el que el número de caracteres es variable (de 1 a 4; 1 es el juego de caracteres latinos no acentuádos). Este formato permite gestionar todos los tipos de caracteres, especialmente los chinos.
  • ucs2 corresponde a un formato estandar Microsoft, en el que los caracteres se almacenan de forma sistemática en 2 octetos.

Export.

  • Export. (campo EXPORT)

Si esta zona está marcada, será posible utilizar este modelo en Exportación de datos.
  • Contador export. (campo CHRNUM)

Este campo, sólo de lectura, muestra el valor del crono en la última exportación. Esto permite, si se realizan exportaciones cronológicas, tratar sólo lo que ha sido modificado desde que se realizó la última exportación.

Transcodificación

  • Juego de caracteres (campo OPTCHA)

Cuando se utiliza el juego de caracteres ascii, es posible emplear diferentes formatos estandarizados:

  • El código ISO 8859 (que es por otro lado, el juego interno de Adonix cuando se trabaja en ascii)
  • El código IBM PC
  • El código ASCII 7 bits (sin acentos: Las letras acentuadas se convierten en la minúscula correspondiente.)
  • Separador decimal (campo SEPDEC)

Define el separador decimal utilizado para las cifras. Si este campo está vacío, se considerará que el saparador decimal es el '.' (punto).
  • Formato de fechas (campo OPTDAT)

Define la manera de codificar los campos de tipo fecha (orde y número de caracteres por año).

De esta manera se puede definir el orden de las cifras y el número de caracteres del año. En importación se filtran caracteres de separación entre los campos. Así, las fechas expresadas como 29-5-59 ó 09/04/1991 están descodificadas correctamente.

El subprograma de decodifiación toma en cuenta la variable adxdcs del motor, indicada por el parámetro DCS que se encuentra en los parámetros generales para definir la manera en que se descodifica un año en dos cifras. DCS representa el año "base" que indica el cambio de siglo.

Por ejemplo, si DCS es 1940, cualquier año de dos cifras inferior o igual a 40 se considerará como perteneciente al siglo 21 y cualquier año superior se considerará como perteneciente al siglo 20. De este modo, los años comprendidos entre 1940 y 2039 estarán formados por dos cifras.
  • campo LIBDAT

Título asociado al código anterior.

  • Formato menús locales (campo OPTMNL)

Los campos de tipo "menú local" se almacenan con un número que representa su rango en la tabla.

Según el valor de esta zona, el modelo va a exportar (o importar):

  • 0 : en ese caso, se escribe la opción en forma de una cifra que indica el rango del menú en la tabla: 1 para la primera opción, 2 para la segunda, etc. Esto corresponde al formato interno donde se almacena el menú local en la base.
  • 1 : se escribe la elección utilizando el código (un carácter) asociado a cada elección del menú local. Este código no puede ser visualizado en gestión de menús locales. Puede ser definido en las funciones de desarrollo, en gestión de mensajes.  Existe la posibilidad de capturar este código interno (útil únicamente para este fin. Ya no funciona como un acelerador de captura, como sucedía en las antiguas versiones carácteres).
  • n : (n>1) los n primeros caracteres de la descripción que aparecen en la introducción. Cuando se utiliza esta opción, el algoritmo de reconocimiento busca en el primer carácter, después en el segundo y así sucesivamente hasta que se encuentre aquel al que corresponde el título. De esta forma si se quiere reconocer CHQ en un menú local cuyos campo son Especies, Transferencia, Cheque, Letra, Tarjeta, el algoritmo reconocerá Cheque (solamente cuando las dos primeras letras se corresponden con el campo).

Teniendo en cuenta que los campos del menú local son extensiones que se utilizan en visualización, el valor almacenado en la base es el rango de la tabla, será posible cambiar el campo de los menús locales durante una importación para que el algortitmo de reconocimiento funcione correctamente. Ahora bien, el hecho de cambiar los campos de los menús locales solamente sólo es posible en modo monousuario. No será posible en las transferencias normales o automatizadas.
  • campo LIBMNL

Título asociado al código anterior.

Import.

  • Import. (campo IMPORT)

Si esta zona está marcada, será posible utilizar este modelo en Importación de datos.
  • Actualización autorizada (campo OPTUPD)

Permite autorizar la modificación de un registro existente durante la importación

  • Sas (campo AOWSTA)

Cuando se marca esta casilla, la importación de datos alimenta el sas de import-export con los datos erróneos. Esto no quita que se cree además un fichero de errores.

  • Importación especial (campo OPTSPE)

Indica que la introducción de datos en la base se realiza mediante acciones específicas definidas en el tratamiento cuyo nombre aparece en la rúbrica Tratamiento importación. Este proceso específico posee un número limitado de puntos de entrada. Por lo tanto necesita escribir un proceso que incluya todos los controles que se desea realizar.

La importancia reside en el hecho de que se pueden agrupar los controles para optimizar el programa de importación. La estructura de los tratamientos de importacion específicos se describe en el anexo. Se deberán encontrar las acciones siguientes:

  • una etiqueta $RAZCRE que es directamente llamada por la importación.
  • una etiqueta $SAIMSK que es directamente llamada por la importación en cada registro leído y reemplaza el Call SAIMSK estándar (asignación y control de las zonas de la pantalla a partir de la clase [F])
  • Creación autorizada (campo OPTCRE)
  • Workflow (campo ENAWRK)

Tabla Identificadores

  • N.º (campo NUMFLG)

Linea actual del cuadro
  • Nivel (campo FLGLEV)

Define el nivel del imbricación del grupo. El nivel 1 es el nivel principal, un nivel N+1 define un sub-detalle del nivel N que le precede.

  • Indicador (campo FLGREC)

Identifica el grupo por un código de 5 caracteres máximo, código que será retomado en el cuadro de los campos de la pestaña siguiente, y en el mismo fichero, así como en la cabeza de grupo.

  • Tabla (campo FLGFIL)

Este cuadro de indicaciones define la estructura de los grupos de registros. Ver el párrafo correspondiente.
  • Clave (campo FLGKEY)

Define la clave de la tabla vinculada utilizada para acceder al detalle de los registros del grupo, a partir de valores de campos de las tablas de niveles superiores utilizadas en la expresión del vínculo.

  • Vínc. (campo FLGLNK)

Define la expresión de vínculo, es decir, una serie de valores, separados por un ";", que da los valores de clave que vinculan la tabla de detalle con el registro de cabecera.

  • Longitud (campo RECLEN)

En el caso de un tipo de fichero de tamaño fijo, es necesario indicar el número de caracteres de cada registro.

Pestaña Campos

En este cuadro se definen los distintos campos que se van a importar, organizados en grupos identificados por la columna Código, en la que se encuentra uno de los códigos definidos en la tabla de indicadores de la primera pestaña (el campo puede permanecer en blanco si no se ha definido ninguna tabla).

En pestaña contiene el cuadro que define la estructura detallada de los grupos existentes en la primera pestaña. Tenga en cuenta que:

  • No es obligatorio definir los campos en todos los grupos (algunos pueden ser grupos meramente "técnicos" que definen enlaces). Si, por ejemplo, se van a exportar campos de la cabecera del pedido y campos de la ficha cliente pagador en un único grupo de datos (sin separador de grupos de datos). En este caso, habrá que definir dos grupos (el primero definirá los pedidos y el segundo estará asociado al cliente mediante el enlace apropiado). Sin embargo, en la pestaña campo solo se introducirán las líneas asociadas al segundo grupo. Estas líneas podrán incluir datos extraídos del cliente y de la cabecera del pedido (ambos están en línea).
  • En cada grupo, hay que definir la posición del separador de grupo cuando se trata de un modelo que se puede utilizar en importación y contiene varios grupos. Se define con el seudocampo / (barra).
  • Los bloques deben estar ordenados de forma secuencial. Cuando hay un bloque de nivel inferior, debe seguir al bloque al que está vinculado.

Tabla Campos

  • N.º (campo NUMLIG)

Linea actual del cuadro
  • Indicador (campo TYP)

Este campo se introduce solamente cuando el cuadro de los identificadores de grupos de la pestaña precedente no está vacío. Permite asociar datos a exportar o importar a un grupo de datos.

  • Tabla (campo FIL)

Aquí se define la tabla de la base de datos en la que figura el dato a importar o exportar. Observar que:

  • Este campo es obligatorio (en un modelo de exportación donde se define expresión calculada. Esta puede referenciar campos de varias tablas).
  • Esta tabla no es necesariamente la tabla principal asociada al grupo, pero puede ser una tabla vinculada a la tabla principal de este grupo o a uno de los grupos precedentes de nivel superior. Si no aparece ningún vínculo será visualizado un mensaje de advertencia: Enlace automático no gestionado. Esto significa que el enlace manual es competencia del procedo de importación o de exportación (capturado en la primera pestaña. Por ejemplo, definiendo la tabla en la acción IMP_OUVRE y la lectura de la tabla en la acción IMP_LIENS).
  • Campo (campo FLD)

Permite indicar el nombre del campo de la tabla a importar o a exportar. Son aceptadas varias sintaxis para definir los datos a extraer o integrar:

  • El carácter / (barra) significa que se ha escrito un indicador de grupo en exportación o que se ha buscado en importación. Este separador es obligatorio para cada grupo de un modelo que permita la importación, siempre que existan varios. El separador de grupo está considerado como un campo normal (justificado con separadores de campos y con delimitadores si se utiliza un modelo de tipo Delimitado).
  • La sintaxis más sencilla es CHAMP (índice), cuando se importa o exporta un campo de la tabla registrada en la columna precedente. Un ventana de selección muestra los campos posibles.
  • Si se captura una constante cadena de carácteres entre comillas ( " ), el campo será escrito tal y como se ha hecho en exportación. En importación no se tendrá en cuenta.
  • Si no se introduce nada, existirá un campo vacío en el fichero importación (justificado por delimitadores de campo si el modelo es de tipo Delimitado). En importación esto significa que el campo correspondiente debe ser ignorado.
  • La sintaxis *N, o N es un número comprendido entre 1 y 99, también es posible Lo que permite asignar (en importación) o leer (en exportación) la variable GIMP(N). GIMP es una variable global, de tipo cadena de caracteres, de tamaño máximo 100. Deberá ser asignada a un proceso específico asociado a la importación o a la exportación.
  • La última posibilidad, aplicable únicamente en exportación, consiste en definir una expresión calculada cualquiera, precedida del signo igual (=). Esta expresión puede ejecutar constantes, funciones, variables, operadores y campos extraídos de las tablas en línea. Es examinada sintácticamente en captura (el control del contexto no puede ser verificado).
  • Descripción (campo COM)

Indicar un comentario para facilitar la comprensión de la parametrización.
  • Límite (campo SEL)

Hay tres opciones posibles en este campo:

  • No captura significa que no se introducirán rangos de principio-fin de lanzamiento de la exportación.
  • captura significa que se podrán introducirán rangos de principio-fin de lanzamiento de la exportación.
  • No transferida significa que podrá introducirse un rango de principio-fin para este campo en el momento del lanzamiento de la exportación, pero el campo no se transformará por ello.
  • Posición (campo LOC)

Esta colúmna sólo es útil para un formato de tamaño fijo; en ese caso, la posición ofrece la diferencia con respecto al principio del módulo o del registro (posiciones dadas en número de octetos, 1 significa que nos encontramos al principio del módulo o del registro). Las posiciones han de ser compatibles con el tamaño del registro.
  • Longitud (campo LNG)

Esta zona determina el tamaño del campo en el fichero secuencial.
  • Formato (campo FMT)

Esta columna solamente se introduce si el formato es de tamaño fijo. En los importes numéricos, el formato elegido es definido con la forma nnn o nnn.mmm. Estas cifras pueden llevar el indicativo < o > (justificación izquierda o derecha completada con ceros. Se utilizará la derecha, por defecto). También pueden llevar el carácter + antes o después de la cifra (esto es obligatorio) y precedido del carácter * (no separador decimal). Generación de fichero

  • campo TYPEXP
  • Fichero de datos (campo FILEXT)

Permite definir la ruta de un fichero de datos por defecto, que será propuesto en el lanzamiento de la importación o de la exportación (y utilizado de manera automática en caso de lanzamiento de un enlace de importación o exportación). Si el fichero es relativo, el directorio de base es el directorio de base de instalación del sistema.

El camino puede integrar el carácter #. Puede haber gestión de números secuenciales:

  • En importación esto significa que se van a a buscar todos los ficheros cuyo modelo corresponda con el camino, con # que representa cifras (se integran los ficheros en orden creciente)
  • En exportación esto significa que se va a crear un fichero integrando el valor formateado [C]EXPORT en 5 cifras. Esto va normalmente parejo con el hecho de marcar la casilla Gestioon chrono cuando se lanza la funciónexportación

Por ejemplo, si el crono exportación es igual a 156, /u/tmp/fic#, permite generar el fichero /u/tmp/fic00156.
  • Directorio final (campo REPFIN)

Permite imponer un repertorio finalen el que el fichero será transferido después de ser importado. A falta de un valor, se utiliza el repertorio dado en los parámetros generales de la import/export.

Tabla

  • N.º (campo NUMLIG)

Linea actual del cuadro
  • Campo (campo CODZONE)

En esta columna se define el nombre de la zona de la tabla tal como se definirá en el aplicativo (a un campo de nombre NOMCHAMP definido en una tabla con la abreviatura ABV, se podrá acceder mediante la sintaxis [F:ABV]NOMCHAMP).

En la base de datos, cada zona corresponde a uno o varios campos (dependiendo de si la zona está dimensionada o no: los campos correspondientes se llaman NOMCHAMP_0, NOMCHAMP_1, NOMCHAMP_2, etc.)

Para introducir y mostrar un campo correspondiente a una pantalla, se le dará el mismo nombre que en el diccionario de pantallas y se utilizará simultáneamente la pantalla y la tabla en la gestión del objeto.
  • Descripción (campo INTITCOURT)

Título asociado al código anterior.

  • Selección (campo SELECT)

Si el campo está en Sí,  la zona se insertará en el cuadro principal. Por defecto, los campos no presentes en el cuadro principal aparecen en Sí, y los que están sin el cuadro principal, en No, 

Permite insertar de forma global un conjunto de campos procedentes de una tabla de un modelo, a partir de la línea en curso del cuadro.

Recalculo posiciones indicador

Esta función solo está disponible en los modelos con un tipo de fichero de longitud fija. Permite recalcular la posición de cada uno de los campos del grupo de datos en curso (con el mismo indicador de la línea). Se recalcula a partir de la posición 1 en el primer campo del grupo y se va añadiendo la longitud de cada campo para obtener la posición del campo siguiente.

Permite definir una condición lógica mediante los campos de la tabla. Sólo podrán exportarse las líneas que respondan a esta condición.

Este botón da acceso a una pantalla en la que se pueden definir los valores por defecto de los criterios para filtrar los datos exportados. Cuando se lanza la exportación, los criterios se muestran y se pueden modificar. Cuando se lanza una cadena de exportaciones, los criterios se aplican automáticamente, sin entrada, en cada uno de los modelos para los que se han definido.

Principios básicos

Cuando se parametriza un modelo de importación, hay que tener en cuenta determinados principios:

  • La importación permite crear y modificar.
  • Para determinar la acción que quiere realizar en un registro importado, el sistema lleva a cabo una prueba de existencia del objeto que se va a procesar. La prueba se lleva a cabo cargando la clave principal del objeto con los datos del registro importado.
  • Por lo tanto, si la clave principal no está parametrizada, todos los registros pasarán a modo creación (y funcionarán únicamente si la clave del objeto que se va a crear se puede asignar automáticamente, por ejemplo, mediante un contador).
  • Una importación corresponde a una simulación de entrada de todas las pantallas vinculadas al objeto importado.
  • Procesa todos los campos de la pantalla realizando los mismos controles que una entrada interactiva.
  • No se tienen en cuenta los campos que no se pueden completar en la pantalla (salvo casos específicos y referenciados). Por lo tanto, los campos parametrizados en un modelo que corresponde a un campo que no se puede completar no se importan.
  • El orden en el que se parametrizan los campos en el registro es irrelevante. La importación carga todos los campos del fichero y, a continuación, los importa en el orden de los campos en las pantallas.

Modelos estándar aportados

La exportación de datos siempre es posible, independientemente del objeto, pero la importación no. Los mecanismos automáticos de descodificación de los flujos de datos y la ejecución de los controles vinculados al objeto automatiza la importación, pero no basta para permitir una importación automática de los objetos complejos. Por lo tanto, no se pueden importar todos los objetos.

En el dossier de referencia, se proporciona un modelo de importación (modificable) para cada objeto importable. No obstante, puede haber especificaciones vinculadas a esta importación. Estas se definen en la ayuda en línea asociada a los modelos de importación con especificaciones (la ayuda está disponible pulsando Alt + F1 cuando el modelo está cargado).

La lista de ayudas correspondientes (clasificadas por módulos) está disponible en el siguiente enlace.

Cuadro de identificadores

La elección de la estructura de los ficheros de importación/exportación depende de la posibilidad de extracción o integración del sistema con el que se va a trabajar.

En todos los casos, los datos deben estar organizados en grupos lógicos de líneas, que pueden ser de varios tipos (cabecera, detalle, subdetalle) o de uno solo. La organización de estos grupos se define en el cuadro de identificadores situado en la primera pestaña del modelo.

Cada grupo está asociado a una de las tablas de la base de datos (la primera siempre es la tabla principal del objeto y las siguientes se definen mediante enlaces a partir de las anteriores). Cuando se utiliza el modelo para exportar un objeto, se pueden definir enlaces con cualquier tabla de la base que cuente con un enlace teórico para poder extraer los datos asociados. Sin embargo, en un modelo de importación, solo se pueden utilizar las tablas actualizadas por el objeto. Por ejemplo, no se pueden actualizar simultáneamente el pedido y el cliente. El objeto del pedido no está diseñado para ello.

Es importante tener en cuenta que este cuadro puede estar en blanco cuando una estructura de datos de importación/exportación está basada en una única tabla principal (en este caso, la columna Código de la página siguiente se mostrará en blanco). Cuando se exportan simultáneamente varias tablas vinculadas, tampoco es necesario realizar varios grupos de datos. De hecho, si algunos campos extraídos de varias tablas se muestran en un mismo grupo de datos, el proceso de exportación va a intentar definir los enlaces entre tablas utilizando la estructura de enlaces descrita en el diccionario. Esto significa que solo hay un enlace posible de la tabla principal a la tabla descrita (de lo contrario, se utiliza el primer enlace y es posible que no sea el adecuado).

Hay un caso específico en el que hay que crear al menos un grupo, cuando el modelo está definido con una longitud fija (en este caso, hay que definir una longitud de registro en el cuadro de grupos). Si no quiere que el indicador de grupo esté obligatoriamente en la lista de campos, basta con dejar en blanco el código del grupo. De esta forma, solo se podrá definir un grupo y ya no se completará la columna Código en la pestaña siguiente.

El cuadro de identificadores solo se puede completar cuando el objeto es simple. Si se definen identificadores de grupo, se asociarán a cada uno de ellos un nivel, una tabla y condiciones de enlace que permiten crear enlaces entre ellos.

El nivel es 1 para la tabla principal de importación/exportación (que no se introduce en el cuadro, sino que se deduce del objeto asociado al modelo).

En todas las tablas vinculadas a una tabla anterior, el nivel es igual al nivel de la tabla anterior cuando hay un enlace biunívoco entre ambas tablas. Este nivel será superior si se vinculan varios registros a un registro de la tabla anterior. El enlace se caracteriza por la clave de la tabla de destino que se va a leer y por la expresión de los segmentos de claves cuyo valor define las líneas vinculadas.

Ejemplo de grupos

Para ilustrar esta parametrización, se puede tomar como ejemplo un modelo (solo de exportación) con sociedades y plantas:

  • La tabla principal, que se denomina COMPANY, esta en el nivel 1.
  • Si queremos integrar los datos vinculados a la divisa contable (campo ACCCUR) en el fichero exportado, no es necesario definir el nuevo grupo vinculado a la tabla de sociedad, basta con definir los campos de la tabla TABCUR en el grupo. El motor de exportación recorrerá el diccionario para buscar un enlace entre las tablas COMPANY y TABCUR.  Al haber varios, se utiliza el primero (en este caso, el adecuado).
  • Si hubiéramos querido exportar los datos vinculados a la divisa del capital social (campo RGCCUR), habría que haber creado un segundo grupo de datos, también de nivel 1, basado en la tabla TABCUR, con RGCCUR en la columna vínculo. Este también habría sido el caso si el enlace no fuera explícito. Por ejemplo, el campo CREUSR, que corresponde al código del usuario que ha creado la ficha, utiliza un tipo genérico (A) que no permite vincular automáticamente. Si hubiéramos querido mostrar los datos del usuario que ha creado la ficha, habría que haber definido el enlace mediante un grupo.
  • Supongamos que ahora queremos mostrar la lista de plantas asociadas a la sociedad. Se creará un grupo de nivel 2, mediante la tabla FACILITY, con un enlace basado en el índice FCY0 que tomará el valor del campo CPY de la tabla principal.
  • Si quiere extraer los valores de parámetro asociados directamente a cada sociedad, se creará un segundo grupo de datos de nivel 2, mediante la tabla ADOVAL, con un enlace basado en el índice ADW0 que tomará el valor del campo CPY de la tabla principal.

Los diferentes formatos del fichero

Los formatos de fichero se determinan por el tipo, que puede tomar uno de los siguientes valores:

formato ascii 1

Es un fichero de longitud variable cuyos campos están separados mediante un separador (el separador de campo SC).

Campo 1 registro 1

SC

Campo 2 registro 1

SC

...

Campo N registro 1

SC

Campo 1 registro 2

SC

Campo 2 registro 2

SC

...

Campo N registro 2

SC

formato ascii 2

Es un fichero de longitud variable cuyos campos están separados mediante un separador (el separador de campo). Una vez terminado un registro, otro separador (el separador de línea denominado SL) sustituye al separador del campo.

Campo 1 registro 1

SC

Campo 2 registro 1

SC

...

Campo N registro 1

SL

Campo 1 registro 2

SC

Campo 2 registro 2

SC

...

Campo N registro 2

SL

Formato Delimitado

Es un fichero de longitud variable del mismo tipo que el fichero "Separador de registro" (dos separadores distintos). Asimismo, los campos de tipo cadena de caracteres están delimitados mediante un delimitador de campo (denominado DC). En este ejemplo, el segundo campo es de tipo alfanumérico.

Campo 1 registro 1

SC

DH

Campo 2 registro 1

DH

SC

...

Campo N registro 1

SL

Campo 1 registro 2

SC

DH

Campo 2 registro 2

DH

SC

...

Campo N registro 2

SL

Formato de tamaño fijo

Es un fichero cuyos campos están definidos con una longitud fija, sin separador de campo. La longitud total del registro debe proporcionarse en parámetros. Puede haber un separador de línea. En este caso, su longitud no debe tenerse en cuenta en la longitud del registro.

También se define la longitud de cada grupo cuanto los grupos de datos están definidos en el cuadro de indicadores.

Campo 1 registro 2

< ---------Campo 2 registro 1------------ >

...

< --Campo N registro 1-- >

SL

Campo 2 registro 2

< ---------Campo 2 registro 2------------ >

...

< --Campo N registro 2-- >

SL

Formato XML

Es un formato en el que los datos se definen en etiquetas XML.

En la exportación, se recupera un gran volumen de datos vinculados al modelo y a la extracción simultáneamente. El menú Opciones/Export. esquema de un modelo también permite exportar un fichero XSD que describe la estructura del fichero creado por el modelo.
En la importación, los datos útiles no son tan elevados, de modo que se pueden limitar al fichero que se quiere importar.
